# How people actually use Claude Fable 5

> Multi-agent web research, adversarially verified. Claims that could not be independently supported are flagged. Feeds the model-routing section of the prompt-loop-engineer agent.

## Angle: What tasks and domains people actually use Claude Fable 5 for — separating documented, named-customer usage from ecosystem hype and aspirational "use-case listicles," with the science/cyber caveat that many headline examples are actually its unsafeguarded sibling Mythos 5.

Claude Fable 5 (Anthropic's "Mythos-class" frontier model, GA June 9 2026; $10/$50 per M tokens, 1M context, 128k output) is positioned and used almost entirely for ONE thing: ambitious, long-horizon, autonomous work where depth is worth the wait — not short chats. The strongest, best-sourced real usage is agentic coding at scale (Stripe's 50M-line Ruby migration in a day is the one hard named case study, repeated everywhere), multi-day autonomous agents (Claude Code / Managed Agents, Zapier AutomationBench), and senior-level finance/knowledge-work reasoning (Hebbia, IMC). Anthropic's most dramatic examples — 10x protein design, a week of autonomous genomics research beating a published model — are largely attributed to Mythos 5 (the same model with safety classifiers removed, limited to Project Glasswing), and Fable 5 itself routes many bio/chem/cyber queries to an Opus 4.8 fallback, so "science/cyber use" is real but mostly NOT on the publicly available Fable 5. Writing/business-deliverable usage is real but the evidence is mostly advice-column framing (Forbes) and generic listicles rather than documented deployments. The "hardest-problem escalation" role the user's own config assigns Fable matches Anthropic's own product positioning ("the hardest knowledge work and coding problems") and Cursor's "class of long-horizon problems that were out of reach." Note context: Fable 5 had a turbulent rollout (US export controls June 12, lifted ~June 30, access repeatedly extended into July), which drives a lot of "grab it while it's free" hype content that should not be read as usage evidence.

**Findings:**

- **[high]** Large-scale agentic coding / codebase migration is the flagship, best-documented real use — and Stripe's 50M-line Ruby migration is the one hard named case study cited across nearly every source.
  - Anthropic's launch and multiple write-ups report Stripe used Fable 5 to perform a codebase-wide migration in a ~50-million-line Ruby codebase in a single day — work Stripe said would have taken a full team 'more than two months by hand.' The model scanned files, classified by complexity, applied transformations, ran tests, and logged failures for human review. This is repeated by AWS, Caylent, MindStudio, and the product page (as an unnamed 'engineering customer'), but it is essentially the SAME single case study, not many independent ones.
  - sources: https://www.anthropic.com/news/claude-fable-5-mythos-5 ; https://www.anthropic.com/claude/fable ; https://www.mindstudio.ai/blog/claude-fable-5-agentic-coding-real-world-results
- **[high]** Multi-day, unattended autonomous agents are the core design target — planning across stages, delegating to sub-agents, and checking its own work.
  - Anthropic states Fable 5 'can work for days at a time' in an agent harness like Claude Code or Claude Managed Agents. Practitioner reviews describe it running 'agents for days unattended,' getting complex work right on the first shot, and being suited to large refactors, automated test-suite generation, dependency upgrades across monorepos, and full-feature implementations touching multiple services — 'tasks where a human developer might block off a full day.' Anthropic docs bill it as 'built for the most demanding reasoning and long-horizon agentic work,' with supported agent features: memory tool, code execution, programmatic tool calling, context editing, compaction.

Numbers trace to Anthropic's model materials but I saw them via secondary reviews, so treat exact figures as reported-not-independently-verified.

## Angle: Developer workflows, prompting habits, and model-selection heuristics that people report for Claude Fable 5 (Anthropic's June 2026 Mythos-class flagship, `claude-fable-5`, $10/$50 per M tokens, 1M context) — how they get the most out of it and when they reach for it vs a cheaper model.

Fable 5 is real and post-my-training-cutoff (launched June 9, 2026); I verified it by fetching Anthropic's own platform docs, which are highly detailed and internally consistent, and cross-checked against practitioner blogs and one named Anthropic developer. The dominant, well-sourced advice is a genre shift in how you prompt: (1) reserve Fable 5 for the hardest long-horizon/agentic/document-heavy work and let cheaper models (Sonnet 5, Opus 4.8, Haiku 4.5) do routine work — "short, well-scoped requests are just as good and far cheaper" elsewhere; (2) treat the `effort` parameter (low/medium/high/xhigh) as the primary intelligence/latency/cost dial, defaulting to high, because "lower effort on Fable 5 often exceeds xhigh on prior models"; (3) adopt long-run agent habits — audit progress against tool results before reporting, keep a markdown memory/notes file, delegate to parallel async subagents, use fresh-context verifier subagents over self-critique, and add a verbatim send-to-user tool; (4) strip old, over-prescriptive prompts/skills and do NOT tell the model to "show its reasoning" (it can trip the reasoning_extraction refusal classifier and force a costly fallback to Opus 4.8). The strongest source by far is Anthropic's official "Prompting Claude Fable 5" guide; community write-ups (The Decoder's coverage of Thariq Shihipar's "blindspot pass," MindStudio, Caylent, comparison blogs) echo and extend it. A few widely-repeated claims (a Claude Code "/effort ultracode" mode; specific SWE-Bench-Pro numbers) trace to single lower-tier blogs and are marked thin-evidence. Caveat: this is post-cutoff; I'm relying on live fetches, and the strongest evidence is Anthropic's own docs rather than large-scale independent developer field reports, which are still thin this early.

**Findings:**

- **[high]** Reserve Fable 5 for the hardest, longest, most ambiguous work; use cheaper models for everything routine — this is the single most repeated heuristic.
  - Anthropic's official prompting guide: 'The teams seeing the best outcomes apply Claude Fable 5 to their hardest unsolved problems; testing it only on simpler workloads tends to undersell its capability range.' It's 'particularly effective at end-to-end work that takes a person hours, days, or weeks.' Practitioners frame a model stack rather than a ladder: Sonnet 5 for execution/implementation and debugging, Opus 4.8 for judgment/review, and Fable 5 'only for complex migrations, hard investigations, or technical plans where the extra cost is justified.' Caylent is blunt: 'Short, well-scoped requests are just as good and far cheaper on Opus 4.8, Sonnet 5, or Haiku 4.5.'
  - sources: https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5 ; https://caylent.com/blog/claude-fable-5-anthropics-first-public-mythos-class-model ; https://codeculture.store/blogs/developer-culture/claude-sonnet-5-vs-opus-48-vs-fable-5
- **[high]** Treat the `effort` parameter (low/medium/high/xhigh) as the primary intelligence-vs-cost-vs-latency dial; default to high, not xhigh.
  - Official guide: 'Use high as the default for most tasks, with xhigh for the most capability-sensitive workloads and medium or low for routine work. Lower effort settings on Claude Fable 5 still perform well and often exceed xhigh performance on prior models.' Advice is to reduce effort if a task completes but takes longer than necessary or you want a more interactive style. Fable 5 has adaptive thinking always on (extended-thinking token budgets are gone; `thinking:{type:disabled}` is unsupported), so effort — not a thinking budget — is how you control reasoning depth. This is echoed by migration write-ups: 'lower effort levels on Fable 5 may still outperform higher effort levels on older models,' so not every migration needs max settings.
  - sources: https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5 ; https://platform.claude.com/docs/en/about-claude/models/introducing-claude-fable-5-and-claude-mythos-5 ; https://knightli.com/en/2026/06/10/claude-fable-5-prompting-guide/
- **[high]** Start tasks at the TOP of your difficulty range and let the model scope them — under-tasking Fable 5 is the common mistake.
  - Official 'Recommended scaffolding changes': 'Start at the top of your difficulty range. Pick a task harder than what you'd assign to prior models, and have Claude Fable 5 scope it, ask clarifying questions, and execute.' It reportedly does single-pass ('first-shot') implementations of systems that previously took days of iteration, and performs well when handed 'complex, multi-threaded requests and asked to determine next steps.'
  - sources: https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5
- **[high]** On long autonomous runs, force the model to ground progress claims in actual tool results before reporting — a specific anti-hallucination habit Anthropic ships as boilerplate.
  - Official guide gives a paste-in instruction: 'Before reporting progress, audit each claim against a tool result from this session. Only report work you can point to evidence for; if something is not yet verified, say so explicitly...' Anthropic says in testing this 'nearly eliminated fabricated status reports even on tasks designed to elicit them.' Migration blogs single this out as a top-5 practice.
  - sources: https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5 ; https://knightli.com/en/2026/06/10/claude-fable-5-prompting-guide/
- **[high]** Give Fable 5 a persistent memory: a plain markdown notes file where it records one lesson per file and references it across runs.
  - Official guide: 'Claude Fable 5 performs particularly well when it can record lessons from previous runs and reference them. Provide a place to write notes, as simple as a Markdown file.' Recommended rule: 'Store one lesson per file with a one-line summary at the top. Record corrections and confirmed approaches alike... delete notes that turn out to be wrong.' You can bootstrap it by asking the model to reflect on past sessions with subagents. The 'Introducing' page claims file-based memory improved Fable 5's goal-completion in complex game tasks '~3x more than for Opus 4.8.' The native memory tool is a supported feature at launch.
  - sources: https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5 ; https://platform.claude.com/docs/en/about-claude/models/introducing-claude-fable-5-and-claude-mythos-5
- **[high]** Lean on parallel subagents with ASYNCHRONOUS orchestration — Fable 5 dispatches and manages subagents far more reliably than prior models.
  - Official guide: 'Use subagents frequently... prefer asynchronous communication between orchestrator and subagents over blocking until each subagent returns. Long-lived subagents that keep their context across subtasks save time and cost through cache reads and avoid bottlenecking on the slowest subagent.' Because single requests can run for many minutes (and autonomous runs for hours), Anthropic explicitly tells teams to adjust client timeouts/streaming and 'check on runs asynchronously, for example through scheduled jobs, rather than blocking.'
  - sources: https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5
- **[high]** For verification, use separate fresh-context verifier subagents rather than asking the model to self-critique.
  - Official scaffolding advice: 'Make self-verification explicit in long-run prompts. Separate, fresh-context verifier subagents tend to outperform self-critique.' Suggested instruction: 'Establish a method for checking your own work at an interval of [X] as you build. Run this every [X interval], verifying your work with subagents against the specification.' Practitioner blogs repeat 'use independent sub-agents for verification rather than relying solely on self-critique.'
  - sources: https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5 ; https://knightli.com/en/2026/06/10/claude-fable-5-prompting-guide/
- **[high]** Strip old, over-prescriptive prompts and skills — instructions tuned for prior models can DEGRADE Fable 5's output.
  - Official guide: 'Skills developed for prior models are often too prescriptive for Claude Fable 5 and can degrade output quality. Review and consider removing older instructions if default performance is better.' Because instruction-following is stronger, you can steer 'most behaviors with a brief instruction rather than enumerating each behavior by name' — a short brevity or 'don't over-engineer' paragraph replaces long rule lists. (This matches the user's own global note that over-prescriptive mega-prompts degrade frontier models.)
  - sources: https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5
- **[high]** Do NOT tell Fable 5 to 'show/echo its reasoning' in the response — it can trigger the reasoning_extraction refusal classifier and force a costly fallback to Opus 4.8.
  - Official guide: 'Prompts, skills, or harness instructions that tell the model to echo, transcribe, or explain its internal reasoning as response text can trigger the reasoning_extraction refusal category on Claude Fable 5, causing elevated fallbacks to Claude Opus 4.8. Audit existing skills and system prompts for reflection or show-your-thinking instructions when migrating.' Raw chain-of-thought is never returned; if you need reasoning visibility, read the structured summarized `thinking` blocks (adaptive thinking) instead. This is both a correctness and a cost workflow concern.
  - sources: https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5 ; https://platform.claude.com/docs/en/about-claude/models/introducing-claude-fable-5-and-claude-mythos-5
- **[high]** For long async agents, add a client-side send_to_user tool so the model can surface verbatim deliverables/progress without ending its turn.
  - Official guide provides the exact JSON tool schema plus elicitation language ('Between tool calls, when you have content the user must read verbatim... call the send_to_user tool'). Rationale: 'Tool inputs are never summarized, so the content arrives intact.' Anthropic warns the tool alone isn't enough — without a system-prompt instruction Fable 5 rarely calls it — and cautions not to route narration/reasoning through it.
  - sources: https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5
- **[medium]** A named-developer habit: run a 'blindspot pass' to surface your own unknown-unknowns before implementation.
  - Anthropic developer Thariq Shihipar frames collaboration around four unknowns (known-knowns, known-unknowns, unknown-knowns, unknown-unknowns) and recommends asking Claude to find YOUR blind spots first. Example prompt he shares: 'I'm working on adding a new auth provider but I know nothing about the auth modules in this codebase. Can you do a blindspot pass to help me figure out my relevant unknown unknowns and help me prompt you better.' He pairs it with structured clarifying-question interviews, HTML-artifact prototyping of design directions, an implementation-notes.md file, and 'quizzes' to test your own understanding. Posted on X with a visual guide (thariqs.github.io/html-effectiveness). Reported by The Decoder; it's one developer's personal framework, not official docs.
  - sources: https://the-decoder.com/anthropic-developer-shares-prompting-tips-for-fable-5-that-focus-on-finding-your-own-blind-spots-first/
- **[high]** The cost/refusal profile itself is a workflow-design factor: 2x Opus's price and variable due to safety-classifier fallbacks, so architect for refusals up front.
  - Fable 5 is $10/$50 per M tokens vs Opus 4.8 at $5/$25 and Sonnet 5 at $2/$10 (Sonnet intro rate). Fable 5 runs classifiers targeting offensive cybersecurity, biology/life-sciences, and reasoning-extraction; a refused request returns stop_reason:'refusal' as a 200 (you're not billed if refused before output), and you're expected to configure server-side/client-side fallback to Opus 4.8, with 'fallback credit' refunding the prompt-cache switching cost. Caylent flags that benign security/coding work can hit 'high false-positive rates,' making effective cost workload-dependent. Also note: Fable/Mythos traffic carries mandatory 30-day data retention (no zero-data-retention option). Net practitioner takeaway: reserve the spend for tasks 'where its capability advantage justifies the 2x cost premium.'
  - sources: https://platform.claude.com/docs/en/about-claude/models/introducing-claude-fable-5-and-claude-mythos-5 ; https://caylent.com/blog/claude-fable-5-anthropics-first-public-mythos-class-model ; https://www.codingfleet.com/blog/claude-fable-5-vs-claude-sonnet-5/
- **[high]** Rein in Fable 5's proactivity and over-elaboration with short boundary/brevity instructions — a commonly cited tuning need.
  - At higher effort Fable 5 can over-plan, over-engineer, do 'unrequested tidying,' or take unrequested actions (drafting an email, making defensive git-branch backups). Official guide supplies paste-in blocks: 'When you have enough information to act, act...'; 'Don't add features, refactor, or introduce abstractions beyond what the task requires...'; and a checkpoint rule to 'pause for the user only when the work genuinely requires them: a destructive or irreversible action, a real scope change, or input that only they can provide.' Also 'give the reason, not only the request' (state intent/who-it's-for) improves results.
  - sources: https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5 ; https://knightli.com/en/2026/06/10/claude-fable-5-prompting-guide/
- **[thin-evidence]** A widely-shared 'Claude Code /effort ultracode' mode (xhigh reasoning + automatic dynamic workflow orchestration) is circulating but I could not confirm it in a primary source.
  - This appears in aggregated search summaries of youmind/knightli-style guides ('set /effort ultracode in Claude Code, which combines xhigh reasoning with automatic dynamic workflow orchestration'). Anthropic's official effort docs I fetched describe only low/medium/high/xhigh. Treat 'ultracode' as unverified — possibly a third-party harness feature or blog embellishment, not confirmed Anthropic terminology.
  - sources: https://youmind.com/landing/x-viral-articles/fable-5-official-prompting-guide (aggregated search summary); not found in https://platform.claude.com/docs/en/build-with-claude/effort

  - MindStudio's '6 rules' article attributes them vaguely to 'Anthropic's documented best practices' and 'does not credit individual Anthropic engineers.' They're reasonable general Claude tips, but not Fable-5-specific, and at least one ('Avoid Triggering Extended Thinking When You Don't Need It') sits awkwardly with Fable 5, where adaptive thinking is always on and controlled via the effort parameter rather than toggled. Useful as a beginner checklist, weak as evidence of how expert Fable 5 users actually work; its one concrete model-selection tip ('use Fable 5 for planning/architecture, route implementation to Sonnet') matches the better-sourced model-stack pattern.

- **[thin-evidence]** Reported capability numbers that justify reaching for Fable 5 (e.g., SWE-Bench Pro 80.3% vs Sonnet 5's 63.2%) come from single comparison blogs and should be treated as indicative, not verified.
