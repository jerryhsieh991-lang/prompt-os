# Multi-Agent

`multi-agent` — 5 loop prompts.

### 1. Supervisor-Verified Task Delegation (Multi-Agent Worker + Supervisor) — corrected

- **When:** A fixed <TASK_LIST> of draft-artifact production tasks (code, docs, datasets, diffs) needs completion where the producer's self-assessment can't be trusted, and a second independent role must re-derive pass/fail from the artifact itself against frozen, checkable criteria. Scope is restricted to tasks whose deliverable is a reviewable artifact — it explicitly excludes any task whose "deliverable" is itself a live/irreversible/public/paid action (send the email, post the update, execute the payment, deploy to prod, delete data). Tasks with inherently subjective acceptance criteria (e.g. "compelling copy") must be reduced to an explicit checklist/rubric at freeze time or are out of scope for this loop family.
- **Loop:** ASSESS the worklist and pick one pending/rejected task with budget left → ACT by dispatching one fresh WORKER call that produces exactly one artifact for that one task only (worker never performs any live/public/paid effect) → VERIFY via an independent SUPERVISOR call — a different role/context than the worker, never fed the worker's self-report, and run on a model tier at least as capable as the worker's (never cheaper/weaker, stated as a hard rule inside the VERIFY step itself, not just metadata) — that re-derives ACCEPT/REJECT from the artifact against the frozen rubric and, on REJECT, must classify the failure into exactly one reason-code from a taxonomy fixed before turn 1 (plus free-text detail) → DECIDE: ACCEPT moves to the next task; REJECT logs the reason-code + detail, increments an attempt counter, and forces a materially different delegation strategy on the next dispatch for that task (verbatim retries banned).
- **Stop:** SUCCESS: Every task in <TASK_LIST> is ACCEPTED by the supervisor's independent, rubric-based re-check of the artifact itself — never by the worker's self-report. · BUDGET: <MAX_TURNS> dispatch turns reached (e.g., 2x len(TASK_LIST)); report which tasks are accepted vs still open. · NO-PROGRESS: Attempts 1, 2, and 3 for the same task are all REJECTED with the identical reason-code (free-text wording may vary — only the reason-code match counts, which is why the taxonomy is frozen up front). Attempt 4 for that task MUST use a materially different delegation strategy (mandatory, not optional). If attempt 4 is also REJECTed, for any reason-code, stop immediately — do not make a 5th attempt — mark the task BLOCKED and continue the rest of the worklist. This makes the attempt-numbering unambiguous: 3 identical-reason-code rejects trigger the strategy-change requirement at attempt 4; a failed attempt 4 is the hard ceiling. · BLOCKED: A task needs a resource, credential, or decision the supervisor cannot obtain (human sign-off, missing access, spec ambiguity only a human can resolve); OR it fails the scope gate — its deliverable is a live/public/irreversible/paid action, or its criteria cannot be reduced to a checkable rubric at freeze time; OR it hit the attempt-4 NO-PROGRESS ceiling. Log exactly what's missing/why, mark that task blocked, continue other tasks — halt the entire loop only if 3+ tasks are blocked on the same systemic cause.
- **Model:** Worker: any model sized to the task, including cheaper tiers. Supervisor: must run on a model tier >= the worker's — never cheaper or weaker — because verification quality is the anti-Goodhart safeguard the whole loop depends on. This is not left in a metadata header: the corrected PROMPT text embeds the constraint directly inside the VERIFY step ("run on a model AT LEAST as capable as the worker's model — never a cheaper/weaker tier — this is a hard rule of THIS step"), so it survives being copy-pasted as a standalone executable prompt.

```text
GOAL (frozen): complete every task in <TASK_LIST> by having a WORKER agent produce a deliverable, and a SUPERVISOR agent independently confirm each deliverable meets <SPEC / ACCEPTANCE_CRITERIA>. Success = every task in <TASK_LIST> is marked ACCEPTED, where ACCEPTED is set only by the supervisor's own re-check against <SPEC / ACCEPTANCE_CRITERIA> applied to the artifact itself — never by the worker's self-reported "done" or "looks good." Freeze <TASK_LIST> and <SPEC / ACCEPTANCE_CRITERIA> now; do not loosen the criteria mid-run to force a pass, and do not add new tasks to the list mid-run (park them for a follow-up loop).

SCOPE GATE (apply once per task, before freezing):
- Deliverable check: every task must produce a reviewable artifact (file, diff, doc, dataset, config) that exists BEFORE any live/public/irreversible/paid effect occurs. If a task's "deliverable" IS the live effect itself (send the email, post the update, execute the payment, deploy to prod, delete data, spend money), it is OUT OF SCOPE for this loop as written. Split it into (a) an in-scope task that produces the to-be-sent artifact as a draft, ACCEPTED by the supervisor, and (b) a separate step outside this loop, gated on explicit human approval, that performs the live action. A WORKER's ACT must never itself be the live action.
- Checkability check: for each task, decide whether <SPEC / ACCEPTANCE_CRITERIA> is mechanically checkable (tests pass, schema validates, exact-match/lint rule, measurable threshold). If yes, freeze as-is. If the criteria are inherently subjective (e.g., "compelling copy," "good tone"), do NOT freeze free-text criteria — convert them now into an explicit binary checklist/rubric (a fixed set of yes/no sub-criteria, e.g., "mentions X," "under N words," "includes a CTA," "reading level <= grade N," "no unsupported claims") that the supervisor will check item by item. If a task cannot be reduced to such a checklist even after this step, mark it BLOCKED-OUT-OF-SCOPE now — it needs a human judge, not this loop — rather than let the supervisor's "independent check" collapse into a second free-floating opinion.
- Reason-code taxonomy: before turn 1, fix a closed list of REJECT reason-codes for this run (task-specific lists are fine), e.g. CORRECTNESS, COMPLETENESS, SPEC-MISMATCH, FORMAT, PERFORMANCE, SECURITY, OTHER-WITH-DETAIL. Every REJECT must carry exactly one reason-code from this list plus free-text detail. "Same reason," wherever this prompt uses it, means same reason-code — never a free-text string match — so a supervisor cannot defeat the anti-thrashing guard by rephrasing an identical rejection.

Carry-forward state (compact, updated every turn):
- worklist: {task_id -> status: pending | dispatched | rejected(reason_code, detail, attempt_count) | accepted | blocked}
- rubric per task (mechanical check or explicit checklist, frozen at scope-gate time)
- last supervisor verdict (task_id, verdict, reason_code, concrete detail)
- turns used / budget remaining
- reason-code history per task (for NO-PROGRESS matching)

Each turn:
1. ASSESS — scan the worklist. Pick exactly one task that is `pending` or `rejected`, has remaining budget, and attempt_count <= 4. Note its reason-code history.
2. ACT (one reversible action, artifact-only) — reconfirm the scope gate for this task: its deliverable is a reviewable artifact, not a live/public/irreversible/paid effect. Dispatch ONE fresh WORKER-agent call for that ONE task only, on any model competent enough for the task. Give the worker <SPEC / ACCEPTANCE_CRITERIA> (or its frozen checklist) and any prior reason-code + detail, so it doesn't repeat the same mistake, but not the supervisor's internal checklist verbatim — the worker should solve the task, not pattern-match the grader. The worker returns a concrete deliverable (file, diff, doc, dataset) — never a claim, and never a live/public/paid action.
3. VERIFY (independent) — the SUPERVISOR, a role/context distinct from the worker and never fed the worker's self-assessment, must run on a model AT LEAST as capable as the worker's model — never a cheaper or weaker tier; this is a hard rule of THIS step, not just a planning note — and re-derives pass/fail directly from the deliverable against the frozen <SPEC / ACCEPTANCE_CRITERIA> or checklist (re-run tests, re-read the doc against the brief, re-check the data against the schema, or walk the explicit checklist item by item — never an unstructured "does this feel done" judgment). Output ACCEPT, or REJECT with exactly one reason-code from the frozen taxonomy plus a specific, falsifiable detail (not "not good enough").
4. DECIDE — on ACCEPT: mark task accepted, move to the next pending task. On REJECT: log the reason-code + detail, increment that task's attempt_count, append the reason-code to its history; the next dispatch for this task MUST change something concrete (narrower scope, added constraint, different approach) — a verbatim retry of the same worker prompt is banned. Do not let the supervisor "average out" a hard fail into a pass because most of the deliverable is fine.

STOP on whichever trips first:
SUCCESS — every task in <TASK_LIST> is ACCEPTED by the supervisor's independent, rubric-based check of the artifact itself.
BUDGET — <MAX_TURNS> dispatch turns reached (e.g., 2x len(TASK_LIST)); report which tasks are accepted vs still open.
NO-PROGRESS — attempts 1, 2, and 3 for the same task are all REJECTED with the identical reason-code (wording may vary; only the reason-code matters). Attempt 4 MUST use a materially different delegation strategy — mandatory, not optional. If attempt 4 is also REJECTed, for any reason-code, do not make a 5th attempt: mark the task BLOCKED immediately and continue the rest of the worklist.
BLOCKED — a task needs a resource, credential, or decision the supervisor cannot obtain (human sign-off, missing access, spec ambiguity only a human can resolve); OR it failed the scope gate (its deliverable is a live/public/irreversible/paid action, or its criteria can't be reduced to a checkable rubric); OR it hit the attempt-4 NO-PROGRESS ceiling above. Log exactly what's missing/why, mark that task blocked, and continue other tasks — halt the entire loop only if 3+ tasks are blocked on the same systemic cause.

Never let the supervisor accept a task on the worker's word alone. Never let the worker also perform the verification step, even under a different persona name in the same context. Never let a WORKER's ACT step fire a live/public/irreversible/paid effect — such effects require a human-approved step outside this loop, taken only after the supervisor has ACCEPTED the corresponding draft artifact.
```

### 2. Two-Agent Debate Resolved by a Judge on Evidence

- **When:** Resolve a question [QUESTION] between Side A: [POSITION_A] and Side B: [POSITION_B] via structured, evidence-cited debate scored by a non-debating judge on a frozen rubric (evidence quality, source reliability, logical validity — rhetorical style excluded).
- **Loop:** Assess -> Act -> Verify -> Decide
- **Stop:** SUCCESS: A side's cumulative judge score clears [THRESHOLD] AND, checked only at the end of a fully completed round, zero rebuttals are owed to the other side. If the trailing side still has an active unresolved rebuttal when the score crosses threshold, the round completes and that side gets its turn first — SUCCESS cannot fire mid-round, denying the loser its move. BUDGET always wins over SUCCESS if both would technically be true in the same (final) round. · BUDGET: [N] total rounds reached (state N, e.g. 8) without SUCCESS having cleanly fired. This is the sole catch-all for "no clean threshold clear," including near-ties and unresolved rebuttals — it no longer overlaps with SUCCESS's wording. On firing, the judge must immediately issue the final split-decision verdict with written rationale; that verdict IS budget's output, not a second success path. · NO-PROGRESS: Fires only when, for 3 consecutive rounds, BOTH (a) the A-vs-B score gap changes by less than [EPSILON] AND (b) each side's own cumulative score also changes by less than [EPSILON]. Requiring (b) prevents falsely tripping this arm when both sides are surfacing strong new evidence at similar pace (scores rising in lockstep, gap flat) — that is real progress, not rhetoric recycling. · BLOCKED: A claim central to either side's case depends on a source neither debater nor the judge can access — halt and name the exact missing evidence. Non-central unverifiable citations do NOT trigger this: step 3 now defines a third evidence-status tier, "unverified-noncentral" (zero credit on evidence quality/source reliability, logic still scored, debate continues), closing the gap between "held up," "failed," and "can't check."
- **Model:** Three distinct agents: Debater A on [MODEL_A]/[TEMP_A] (argues [POSITION_A] only), Debater B on [MODEL_B]/[TEMP_B] (argues [POSITION_B] only), Judge on [MODEL_JUDGE] (never debates, independently re-verifies citations). Both debater models/temperatures are frozen and disclosed before round 1; if they differ in capability tier, the judge's final rationale (under either SUCCESS or BUDGET) must explicitly flag the asymmetry and state whether it plausibly drove the evidence-quality gap — required disclosure, not a score adjustment — and recommend a swapped-model confirmation run if the margin is narrow.

```text
GOAL (frozen): resolve the question [QUESTION] between two positions — Side A: [POSITION_A], Side B: [POSITION_B] — via structured debate.

Success = the judge scores one side's cumulative case >= [THRESHOLD] (on a fixed rubric: evidence quality, source reliability, logical validity — rhetorical style excluded), checked only at the end of a fully completed round, with zero rebuttals owed to the other side at that point.

If no side reaches threshold before the round budget is exhausted, the run still produces a usable output: BUDGET forces the judge to issue a final split-decision verdict with written rationale. That is a defined, intentional exit — not a failure — but it is triggered and owned by BUDGET, not by SUCCESS. SUCCESS and BUDGET are mutually exclusive triggers over the same event space (round-cap-hit-without-clean-clear); if both conditions are technically true in the same (final) round, BUDGET wins.

Freeze the rubric, the round budget [N], [THRESHOLD], [EPSILON], and both debaters' models/temperatures now; do not loosen or reassign any of them mid-run.

Roles (three distinct agents, no role ever plays two):
- Debater A — argues [POSITION_A] only, on [MODEL_A] at temperature [TEMP_A]. May concede a specific point if evidence contradicts it, but may not switch sides.
- Debater B — argues [POSITION_B] only, on [MODEL_B] at temperature [TEMP_B]. Same constraint.
- Judge — on [MODEL_JUDGE], never debates, never advocates either side. Sees only the debate transcript and whatever sources it independently verifies (may re-check a cited source itself, not trust the debater's paraphrase). Not shown either debater's private notes or confidence language.

Model-asymmetry disclosure (structural fairness): state [MODEL_A]/[TEMP_A] and [MODEL_B]/[TEMP_B] before round 1. If they differ in known capability tier, the judge's final rationale — whether issued under SUCCESS or BUDGET — must explicitly flag this and state in one sentence whether the capability gap plausibly contributed to the evidence-quality gap between sides. This is a disclosure requirement, not a rubric adjustment: the score itself is not altered. No verdict may be used downstream (see human approval gate) without this disclosure attached. If the asymmetry is material and the margin between sides is narrow, the judge should recommend a swapped-model confirmation run before the verdict is treated as final for any high-stakes decision.

Carry a compact scorecard forward each round: {round #, speaker, claim made, evidence cited, evidence status [held-up / failed / unverified-noncentral], judge score this round (A/B), running cumulative score A vs B, unresolved rebuttals owed, rounds used / budget}.

Each turn:
1. Assess — read the scorecard: whose rebuttal is owed, current score gap, cumulative scores, rounds remaining.
2. Act — exactly ONE move: either an opening statement (round 1 only, one per side, simultaneous) or ONE rebuttal responding to the single most recent unaddressed point from the other side. The move must cite specific evidence (source + quote/data point), not just assert. Never let one side take two moves in a row, and never restate a prior round's argument in new words (no verbatim or near-verbatim retries).
3. Verify — the judge scores this round alone against the frozen rubric: does the new evidence hold up (re-check the citation if feasible), does it actually rebut the specific point raised, is the logic valid. Three possible evidence-status outcomes:
   - Held up — citation verified, supports the claim as stated. Scored normally on all three rubric dimensions.
   - Failed — citation verified but does not support the claim, or is verifiably false/misrepresented. Scored zero on evidence quality and source reliability for that point.
   - Unverifiable — the judge cannot check the citation (paywall, inaccessible source, etc.):
     - Central to either side's case → do NOT score the round; trigger BLOCKED.
     - Not central → score logical validity normally, but cap evidence quality and source reliability at zero for that point (same treatment as "failed"), and label it "unverified-noncentral" on the scorecard. The debate continues; this does not trigger BLOCKED.
   Output is a numeric/rubric score plus one sentence of rationale — never "I feel Side A was more convincing."
4. Decide — append to the scorecard; update cumulative score; mark the rebuttal as resolved or still owed; pick which side moves next:
   - If exactly one side has an unresolved rebuttal owed to it, that side moves next.
   - If both sides are clear, alternate strictly by whichever side did NOT move most recently.
   - Tie-break for the first "both clear" state (start of round 2, right after simultaneous round-1 openings): Side B moves first, as documented compensation for Side A's framing advantage of a fixed opening-position order. This rule is fixed for the whole run — do not re-litigate it round to round.

STOP on whichever trips first — arms are checked in this order, only one fires per event:

SUCCESS — a side's cumulative judge score clears [THRESHOLD] AND, checked only at the end of a fully completed round (never mid-turn), zero rebuttals are owed to the other side. If the trailing side still has an active unresolved rebuttal when the leading side's score crosses threshold, the round completes and the trailing side gets that one rebuttal turn first; if it is then resolved or forfeited, SUCCESS fires immediately without waiting for a further round.

BUDGET — [N] total rounds reached (state N now, e.g. 8) without SUCCESS having cleanly fired. This is the sole catch-all for "threshold never cleanly cleared" — including close scores, ties, or a lead with rebuttals still owed. On firing, the judge must immediately issue a final split-decision verdict with written rationale as BUDGET's required output. BUDGET always wins over SUCCESS if both are technically true in the same final round.

NO-PROGRESS — for 3 consecutive rounds, BOTH: (a) the score gap between A and B changes by less than [EPSILON], AND (b) each side's own cumulative score also changes by less than [EPSILON] round over round. Requiring (b) means two sides both surfacing strong new evidence at similar pace (scores rising in lockstep, gap flat) reads correctly as progress and does not trip this arm — only true stagnation (flat gap AND flat totals) does. When it fires, force the judge to issue a verdict on the evidence as it stands rather than continuing.

BLOCKED — a claim central to either side's case depends on a source neither debater nor the judge can access (paywall, missing internal document, needs a subject-matter expert) — the "central + unverifiable" case from step 3. Halt and name the exact missing evidence rather than let the round proceed unverified. Non-central unverifiable citations never trigger this — they're handled inline in step 3 as "unverified-noncentral," and the debate continues.

A human approval gate is required before any verdict from this loop (issued under SUCCESS or BUDGET) is used to make any irreversible or costly decision (e.g. committing to [POSITION_A] or [POSITION_B] in production/budget/policy). The loop produces a recommendation with rationale plus the model-asymmetry disclosure — not an autonomous action.
```

### 3. Planner → Executor with Per-Step Verification (Multi-Agent)

- **When:** A multi-step task (infra change, schema migration, deployment, agentic pipeline) needs adaptive step-by-step planning based on verified outcomes rather than a fixed upfront plan — and some steps may be irreversible or costly (deploy, delete, external send, spend, credential use, production data mutation), requiring explicit human confirmation before they execute.
- **Loop:** ASSESS (planner reads verified-step log + state summary only, never the raw executor transcript) -> PLAN (planner emits exactly ONE next step, tagged REVERSIBLE or IRREVERSIBLE/COSTLY, with a concrete verifier-runnable acceptance test) -> [HUMAN GATE if IRREVERSIBLE/COSTLY, before ACT] -> ACT (a distinct executor performs only that one step) -> VERIFY (a verifier in a separate role AND separate session/context from planner and executor, which independently re-runs the acceptance test itself — re-executes the check/query/probe — and never accepts the executor's narrated report as evidence) -> DECIDE (PASS: append to log, plan next step · FAIL on a REVERSIBLE step: revert to last verified state, require a materially different next approach, never resubmit identical or oscillate A→B→A · FAIL on an IRREVERSIBLE/gated step: no auto-revert — halt, snapshot exact state, escalate to human for remediation; this resolves as BLOCKED, not a routine retry)
- **Stop:** SUCCESS: The independent verifier — running its own probe against the system, not reading anyone's account of it — confirms <SUCCESS_CRITERIA> on the final state after the last step. · BUDGET: <STEP_BUDGET> steps or <TIME_BUDGET> wall-clock reached. · NO-PROGRESS: The same step (or step type) fails verification twice running; OR — only if GOAL defined a <GOAL_METRIC> — <K> consecutive verified steps produce zero movement on it. If the objective is boolean pass/fail and <GOAL_METRIC> is declared N/A in GOAL, this second trigger is inactive and NO-PROGRESS relies on the first trigger (repeated same-step failure) plus BUDGET as the backstop, so it is never vacuous for scenarios with no natural graded metric. · BLOCKED: A step needs a resource/access/human decision the agents don't have; OR an irreversible/gated step executed (post human confirmation) but then FAILed its own acceptance test and needs human-directed remediation rather than an automatic revert that isn't defined for that case. State exactly what's missing or broken — never substitute a workaround.
- **Model:** Opus 4.8 at xhigh is the sensible default for the planner — picking a genuinely different next approach after a FAIL, and correctly tagging REVERSIBLE vs IRREVERSIBLE, needs real judgment. The verifier's job is narrower (run the probe, diff against the acceptance test, emit PASS/FAIL) so Sonnet 5 is acceptable there, but it MUST run in its own session/context regardless of model choice — never the same conversation as the planner. Escalate the planner to Fable 5 only for high-ambiguity domains (novel distributed-systems migrations, unfamiliar production topology) where the reversible/irreversible boundary itself is hard to call correctly.

```text
GOAL (frozen): achieve <OBJECTIVE> such that <SUCCESS_CRITERIA> holds, as confirmed by <VERIFICATION_METHOD> (e.g. test suite, schema/state check, deployment health probe, domain-specific validator). If progress toward <SUCCESS_CRITERIA> is naturally continuous or gradable, also define <GOAL_METRIC> here — one number the planner can read off the current state each turn, and exactly how to compute it. If the objective is inherently boolean (migration landed / deployment healthy / not), set <GOAL_METRIC> = N/A explicitly — do not leave it undefined. Freeze this bar now — do not loosen it mid-run.

Do NOT draft a full multi-step plan upfront beyond a rough one-line sketch of the end state. The point of this loop is that each step is planned only after the prior one is independently verified, so the plan adapts to what actually happened rather than what was assumed.

Carry a compact state forward each turn:
- verified-step log: [{step description, action taken, REVERSIBLE|IRREVERSIBLE/COSTLY tag, verifier verdict, evidence}, ...]
- current system/task state summary (not the full execution transcript)
- remaining objective (what's still unmet)
- <GOAL_METRIC> current value, or N/A
- steps used so far / budget remaining

Each turn:
1. ASSESS — planner reviews the verified-step log + current state summary only (never the raw executor transcript, to avoid being anchored on execution narrative instead of verified fact).
2. PLAN — planner emits exactly ONE next step: a specific action, explicitly tagged REVERSIBLE or IRREVERSIBLE/COSTLY (deploy, delete, external send, spend, credential use, production data mutation — anything on this list is IRREVERSIBLE/COSTLY by default). A step is allowed to be irreversible; what's required is the tag plus a falsifiable acceptance test written as a concrete probe — a command, query, health check, or diff the verifier can execute itself — never a vague "looks right." No step may be planned without this probe defined.
3. ACT — a distinct executor performs that one step only. No bundling multiple steps into a single action, no skipping ahead. If the step is tagged IRREVERSIBLE/COSTLY, the executor does not act yet — proceed to HUMAN GATE first.
4. VERIFY — a verifier that is both a separate role and a separate session/context from the planner and the executor: it is spun up fresh each turn with only (a) the step's acceptance test and (b) its own independent means of inspecting the system, and nothing else — not the planner's stated rationale, not the executor's narrated account of what it did. The verifier re-runs the probe itself (re-executes the test, re-queries the state, re-checks the resource directly) rather than reading anyone's report of the outcome; the executor's own account of what happened is never treated as evidence. Returns PASS or FAIL-with-reason only — never a confidence score, never self-graded by the executor.
5. DECIDE:
   - PASS → append to the verified-step log, let the planner plan the next step.
   - FAIL on a REVERSIBLE step → revert the action (return to last verified state), log the failure reason, and require the next planned step to use a materially different approach — never resubmit the identical step, and never oscillate between two failing choices (A→B→A).
   - FAIL on an IRREVERSIBLE/COSTLY step (executed after human confirmation, but its own acceptance test then failed — e.g. a confirmed deploy that doesn't come up healthy) → there is no automatic revert defined for this case. Halt the loop immediately, capture an exact snapshot of the current state, report to the human precisely what was done and how it failed, and wait for explicit human-directed remediation (rollback, forward-fix, or a revised acceptance bar) before planning resumes. Treat this as a BLOCKED stop, not a routine FAIL-and-retry.

HUMAN GATE: before any step tagged IRREVERSIBLE/COSTLY executes, stop and get explicit human confirmation — even if the plan calls for it and even mid-run. Being planned as irreversible is allowed; executing without this checkpoint is not. If it later FAILs its acceptance test post-execution, its recovery path is the escalation path in DECIDE above, not the revert path — there is no promise that an irreversible action can be undone, only that a human is in the loop before it happens and is looped back in immediately if it goes wrong.

STOP on whichever trips first:
SUCCESS — the independent verifier confirms <SUCCESS_CRITERIA> on the final state after the last step.
BUDGET — <STEP_BUDGET> steps or <TIME_BUDGET> wall-clock reached.
NO-PROGRESS — the same step (or step type) fails verification twice running; OR, only if <GOAL_METRIC> was defined (not N/A) in GOAL, <K> consecutive verified steps produce zero movement on it. When <GOAL_METRIC> = N/A, this second trigger is inactive by design — NO-PROGRESS still fires via the first trigger, backstopped by BUDGET, so boolean pass/fail scenarios (schema migrations, deployment health) are never left without a working stop condition.
BLOCKED — a step needs a resource/access/human decision the agents don't have; OR an irreversible/gated step executed and then FAILed its acceptance test, requiring human-directed remediation. Halt and state exactly what's missing or broken rather than substituting a workaround.

Never let the planner treat the executor's or verifier's internal reasoning as evidence — only the logged action, its REVERSIBLE/IRREVERSIBLE tag, and the verifier's independently self-observed PASS/FAIL verdict count toward the state.
```

### 4. Best-of-N Candidates vs. Independent Rubric Judge

- **When:** You need one final artifact (a piece of copy, a code fix, a design brief, an agent plan) and want to hedge against any single generation attempt being weak — generate N candidates independently (no candidate sees another), then have a judge that took no part in generation rank them against a rubric fixed before any candidate exists, and select the top-ranked one.
- **Loop:** assess leaderboard (candidates so far, best score, floor, N remaining, position in the K-window) -> generate ONE new candidate from a fresh, isolated context (no visibility into other candidates or the leaderboard) -> independent judge scores that candidate alone against the frozen rubric, blind to which generator produced it -> decide: discard for free (no N slot consumed) if malformed/off-brief OR a near-duplicate of a candidate already on the leaderboard (checked via a lightweight similarity comparison appropriate to the artifact type); otherwise add the scored candidate to the leaderboard and update best-so-far -> repeat until a stop arm trips -> on SUCCESS, or on a BLOCKED near-tie, surface the top candidate(s) to the human for approval/tie-break — this is the one moment multiple candidates are ever shown together, presented order-randomized so the human's read isn't anchored by position or provenance.
- **Stop:** SUCCESS: N candidates have been scored AND the current best clears the frozen floor score. This is the only arm that can fire once N is reached with the floor cleared (see precedence note below — it is checked first, before BUDGET, every turn). Report the winning candidate, its score breakdown, and how many candidates were generated (including free duplicate/malformed discards, reported separately from scored attempts). · BUDGET: Fires only on conditions disjoint from SUCCESS: (a) the wall-clock/cost cap is hit *before* N attempts are exhausted — if the current best already clears the floor even though fewer than N candidates were sampled, ship it with a caveat noting the shortfall; otherwise escalate to BLOCKED — or (b) N generation attempts are exhausted and the best does *not* clear the floor — always escalate to BLOCKED in this case; never ship a sub-floor candidate just because the budget ran out. Because SUCCESS requires "N reached AND floor cleared" and BUDGET's (b) requires "N reached AND floor NOT cleared," the two conditions are mutually exclusive by construction — precedence order (SUCCESS > BUDGET > NO-PROGRESS > BLOCKED, checked in that fixed sequence every turn) is a backstop, not a load-bearing tie-break. · NO-PROGRESS: The best-of-so-far score has not improved across the last K = [K] scored candidates (K is a frozen parameter set in the GOAL block alongside N, FLOOR, and BUDGET_LIMIT before the loop starts — same status as those, not an ad hoc threshold). Diminishing returns: stop early and take the current best rather than spending the rest of the budget chasing marginal gains. Only evaluable once at least K candidates have been scored; before that, this arm cannot trip. · BLOCKED: Any of: the judge is unavailable; two or more top candidates tie within [TIE_TOLERANCE] points on the frozen rubric's scale (TIE_TOLERANCE is a frozen parameter defined in the GOAL block, same status as FLOOR — it is what makes "near-tie" mechanically checkable) and need a human tie-break; or a single generator configuration (same model/temperature/isolated-context recipe) produces malformed, off-brief, OR near-duplicate output twice in a row — duplicates are folded into this same 2-retry rule as malformed output, since both represent a generator config failing to add independent value. Halt and name exactly what's missing or who needs to decide.
- **Model:** Generators can be cheap/fast and even heterogeneous (mix models/temperatures for diversity — that's the point of N independent attempts). The judge must be a distinct role from every generator, ideally the strongest available model, and must be blind to provenance (which generator/model/temperature produced the candidate) on every scoring pass. Because the judge scores exactly ONE candidate in isolation per turn — never shown other candidates or the leaderboard — "order-randomization" does not apply to judge scoring; it has nothing to randomize the order of. Order-randomization instead applies at the single later point where candidates ARE shown together: the human tie-break/final-approval step, where the top candidate(s) are presented in randomized order so position doesn't bias the human's pick either.

```text
GOAL (frozen, defined BEFORE any candidate exists): produce the best [ARTIFACT_TYPE] for [TASK/BRIEF]. Freeze the rubric now, before generation starts: [DIMENSION_1] (0-10), [DIMENSION_2] (0-10), [DIMENSION_3] (0-10) -- weighted or summed into one score -- plus a floor score of [FLOOR] below which no candidate is acceptable regardless of rank. Budget: N = [N] candidates, or [BUDGET_LIMIT] wall-clock/cost, whichever comes first. No-progress window: K = [K] scored candidates -- if the best-of-so-far score hasn't improved across the last K scored candidates, stop early (see NO-PROGRESS). Near-tie tolerance: [TIE_TOLERANCE] points on the rubric's scale -- two or more top candidates scoring within this tolerance of each other count as tied and require a human tie-break rather than an automatic pick (see BLOCKED). Candidates are generated independently: each generator gets only the brief and rubric dimensions (not the rubric weights, not other candidates, not the leaderboard) -- no candidate may be shown another candidate before scoring, and no generator may see a prior generator's output. This prevents anchoring and collapse-to-consensus.

Maintain a compact carry-forward state: candidate count so far (scored, and separately, free discards), best score + which candidate holds it, last K scores (for the no-progress check, K as frozen above), floor score, tie tolerance, N remaining, and a per-generator-config discard count (malformed + near-duplicate combined) for the 2-in-a-row BLOCKED check.

Each turn: (1) Assess -- current leaderboard state, candidates remaining against budget, and position within the K-window. (2) Act -- ONE reversible action: dispatch a fresh, isolated generation for exactly one new candidate. Never regenerate a candidate already scored; never let a generator revise based on judge feedback (that would make it a refinement loop, not best-of-N -- if refinement is what you actually want, use the prompt-optimization pattern instead). (3) Verify -- an independent judge (never a generator, never self-grading) scores that ONE candidate alone against the frozen rubric, blind to which generator produced it (it never sees other candidates, so there is nothing to order-randomize at this step); before scoring counts, run a lightweight similarity/duplicate check against candidates already on the leaderboard, appropriate to the artifact type (text-diff, embedding similarity, structural diff, etc.). (4) Decide -- reject and discard for free (no N slot consumed) if the candidate is malformed, off-brief, or a near-duplicate of an already-leaderboarded candidate; otherwise add the scored candidate to the leaderboard, update best-so-far, and pick the next single action: generate another candidate, or stop. If the same generator configuration produces a malformed/off-brief/duplicate discard twice in a row, do not retry it a third time for free -- that trips BLOCKED.

STOP arms are checked in this fixed order every turn, so exactly one arm ever reports out -- SUCCESS, then BUDGET, then NO-PROGRESS, then BLOCKED:

SUCCESS -- N candidates have been scored AND the best score clears the floor; report the winning candidate, its score breakdown, and how many candidates were generated (scored attempts and free discards reported separately).

BUDGET -- fires only when SUCCESS's condition does not hold: either (a) the wall-clock/cost cap is reached before N scored attempts are exhausted -- if the current best still clears the floor despite the shortfall, ship it with that caveat noted, otherwise escalate to BLOCKED -- or (b) N scored attempts are exhausted and the best does NOT clear the floor -- always escalate to BLOCKED in this case, never ship a sub-floor candidate merely because the budget ran out.

NO-PROGRESS -- the best-of-so-far score has not improved across the last K = [K] scored candidates (diminishing returns from more sampling); stop early and take the current best rather than spending the rest of the budget chasing marginal gains. Not evaluable until at least K candidates have been scored.

BLOCKED -- the judge is unavailable; two or more top candidates tie within [TIE_TOLERANCE] points on the rubric and need a human tie-break; or a single generator configuration produces malformed, off-brief, or near-duplicate output twice in a row (the 2-retry cap applies jointly across all three failure types, not malformed alone) -- halt and name exactly what's missing or who needs to decide.

Before the selected candidate is deployed, published, or otherwise used irreversibly, surface it to the user/owner for explicit approval -- selection by the judge is not itself authorization to ship. When a BLOCKED near-tie or a SUCCESS final-approval step requires showing more than one candidate to a human, present them order-randomized (not by generation order or score order) so position and provenance don't bias the human's read -- this is the only point in the loop where multiple candidates are ever shown together; the judge, in every prior turn, scored exactly one candidate in isolation. Never let the judge or a generator self-report confidence in place of the rubric score; the rubric score is the only currency that moves a candidate up the leaderboard.
```

### 5. Router-to-Specialist Dispatch with Independent Confidence Gate and Mandatory Fallback

- **When:** A queue of requests each needs exactly one of a roster of specialists, but the router's specialist pick can't be trusted blindly — use when you need every request to land on a capability-matched specialist, a non-fitting or bounced request to fall through to a generalist catch-all, and anything the specialist/gate can't resolve (or that is itself high-stakes) to reach a human, with zero silent drops and zero force-fits past an independent rejection.
- **Loop:** ASSESS (pick next: pending first, else a once-rejected request due for fallback — never a second specialist, no exceptions) -> ACT (independent gate pre-screens for high-stakes/irreversible content first; if clear, router proposes exactly ONE assignment — first-ever specialist attempt, OR fallback if the specialist attempt already failed, OR direct human escalation — and the assigned agent makes ONE attempt) -> VERIFY (a separate confidence-gate component, isolated from the router's rationale, checks capability-match + genuine completion for specialist attempts, or completion-only for fallback attempts; side-effecting actions always need human approval regardless of score; a sample of ACCEPTs is logged for calibration audit) -> DECIDE (ACCEPT -> terminal; REJECT on the specialist attempt -> fallback only, never a second specialist; REJECT on the fallback attempt -> escalate to human with full history; any attempted second-specialist assignment is a structural-invariant violation, not a judgment call)
- **Stop:** SUCCESS: Queue is empty and every request's logged terminal state is done-by-specialist (capability-matched by the gate at >= CONFIDENCE_THRESHOLD and confirmed in-scope complete), done-by-fallback (generalist confirmed complete), or escalated-to-human with a logged reason. Zero requests silently dropped; zero requests force-fit to a specialist the gate rejected. · BUDGET: Total routing attempts reach MAX_ROUTING_ATTEMPTS (default = 2x queue length), or the wall-clock cap is hit. This ceiling is now exact, not approximate: the loop enforces at most one specialist touch + one fallback touch per request (2 touches max), so the 2x formula can never be spuriously tripped by legitimate progress — if it fires, it means requests genuinely used their full 2-touch budget. Halt and report remaining pending/in-flight requests. · NO-PROGRESS: (a) pending-queue count is unchanged for K_STALL_TURNS consecutive turns (default K_STALL_TURNS = 3 unless the operator overrides it before the run starts — do not launch with this value unset). (b) Structural-invariant violation: an attempted second specialist assignment for any request — same specialist or a different "better-fit" one — is detected before it executes; force that request straight to HUMAN_ESCALATION_CHANNEL with a "loop-invariant violated" flag the instant it's detected. This should never fire given the DECIDE rules (there is no reroute-to-a-different-specialist path in this loop at all), so its firing is itself a defense-in-depth signal that something upstream broke the contract, not a routine path. · BLOCKED: A request needs a capability outside both the frozen SPECIALIST_ROSTER and GENERALIST_FALLBACK, so it must go to HUMAN_ESCALATION_CHANNEL — but if that channel itself is unavailable or unresponsive, halt the entire loop (don't keep processing other requests indefinitely while one sits stuck and unresolvable) and surface exactly which request(s) and why.
- **Model:** The router (proposer) is explicitly the cheap/fast, non-integrity-critical component and can run on a small model — it is never trusted to self-certify correctness or self-flag high-stakes requests; that duty belongs to the independent gate. The confidence gate is the integrity-critical component: it must run as a structurally separate pass with information isolation (it sees only the raw request and the assigned agent's declared profile/output — never the router's stated rationale or classification label, so it can't anchor toward agreeing with the router) on every run, unconditionally — this isolation is mandatory, not a nice-to-have. On top of that baseline, upgrade the gate's model specifically when the domain has ambiguous or adversarial requests; isolation and model strength are two separate levers, and isolation is never the one you skip to save cost.

```text
GOAL (frozen): every request in <REQUEST_QUEUE> ends up in exactly one terminal state: (a) done-by-specialist — handled by a specialist from <SPECIALIST_ROSTER> whose declared capability profile the independent confidence gate scored >= <CONFIDENCE_THRESHOLD>, AND that specialist confirmed in-scope completion (not a bounce/error/"not my domain"); or (b) done-by-fallback — handled by <GENERALIST_FALLBACK> and confirmed complete; or (c) escalated-to-human via <HUMAN_ESCALATION_CHANNEL> with a logged reason. Success = queue empty, zero requests silently dropped, zero requests force-fit to a specialist the gate rejected. Freeze <SPECIALIST_ROSTER> and <CONFIDENCE_THRESHOLD> now — do not let the router invent new specialist categories or lower the threshold mid-run to clear the queue faster.

HARD STRUCTURAL INVARIANT (binding on every step below, no exceptions): a request gets AT MOST ONE specialist attempt, ever. If that attempt is rejected, the only next step is <GENERALIST_FALLBACK> — there is no reroute to a second, "better-fit" specialist under any circumstance, no matter how clearly the rejection reason points to one. Every request's touch sequence is exactly one of: [specialist:ACCEPT] -> done; [specialist:REJECT -> fallback:ACCEPT] -> done; [specialist:REJECT -> fallback:REJECT] -> escalate; or [flagged high-stakes] -> escalate immediately, no specialist attempt at all.

Carry a compact scratchpad forward each turn: a table of request -> {status: pending / routed-specialist / routed-fallback / escalated / done, specialist tried (max 1, or none), confidence score, attempt count (0/1/2)}; a per-specialist load counter; the last 3 gate confidence scores (trend); routing attempts used vs <MAX_ROUTING_ATTEMPTS> (default = 2x queue length — exact under the invariant above, since each request contributes at most 2 touches: 1 specialist + 1 fallback); pending-queue count for the last <K_STALL_TURNS> turns (default K_STALL_TURNS = 3 unless the operator overrides it before the run starts — do not launch with this unset).

Each turn:
1. ASSESS — pick the next request: prefer a pending (never-touched) request; if none, pick a bounced request that has had exactly one specialist attempt and now must go to fallback. Never a second specialist guess for that request — this is the same absolute rule stated in the invariant above, not a case-by-case call left to DECIDE.
2. ACT (one reversible action) — first, independent of the router: the confidence-gate component screens the raw request text/features for high-stakes/irreversible character. This determination is never the router's self-report — the router is the cheap/fast component and is not trusted to correctly self-flag its own request as sensitive. If the gate flags it, route directly to <HUMAN_ESCALATION_CHANNEL> now (this does not consume a routing attempt) and skip to DECIDE. Otherwise, the router proposes exactly ONE assignment for exactly ONE request: a single named specialist from the frozen roster (only if this request has zero prior specialist attempts), OR <GENERALIST_FALLBACK> (only if this request already has one rejected specialist attempt), OR direct escalation to <HUMAN_ESCALATION_CHANNEL>. The assigned agent then makes ONE attempt at the request. This step produces a proposal + one attempt — it does not itself commit anything with real-world side effects.
3. VERIFY (independent of the router and of the specialist/fallback) — a separate confidence-gate component does the checks below, with information isolation: it sees only the raw request text/features and the assigned agent's declared capability profile plus its output — never the router's stated rationale or classification label — so it cannot anchor toward agreeing with the router's own reasoning.
   - Specialist attempt: (i) does the request's classified intent/features actually match the specialist's declared capability profile, scored against <CONFIDENCE_THRESHOLD>; (ii) did the specialist's own output signal genuine in-scope completion, or a bounce/error/out-of-domain refusal. Output: ACCEPT / REJECT-low-confidence / REJECT-specialist-bounced.
   - Fallback attempt: only check (ii) applies (genuine completion vs. bounce/error) — capability-match scoring doesn't apply to the generalist catch-all. Output: ACCEPT / REJECT-fallback-failed.
   Calibration/audit (mandatory): every 10th ACCEPT (or fewer on a short queue) is logged with its confidence score and outcome to an audit trail; if audited accuracy diverges materially from what <CONFIDENCE_THRESHOLD> implies, flag "gate recalibration needed" in the scratchpad and surface it in the final report — do not keep running silently on a gate you have reason to believe is miscalibrated.
   Hard gate: if the specialist's or fallback's proposed action has any real-world side effect (send a message, execute a transaction, modify a record, anything irreversible or costly), that action requires explicit human approval before execution regardless of confidence score — the confidence gate alone never authorizes it.
4. DECIDE:
   - ACCEPT -> mark the request done-by-specialist (if this was a specialist attempt) or done-by-fallback (if this was a fallback attempt); log the confidence score and attempt count; move to the next request.
   - REJECT (any reason), and this was the request's specialist attempt (attempt count = 1) -> route straight to <GENERALIST_FALLBACK>. There is no "reroute to a different, better-fit specialist" branch — that path does not exist in this loop, full stop.
   - REJECT, and this was the request's fallback attempt (attempt count = 2) -> escalate to <HUMAN_ESCALATION_CHANNEL> with the full rejection history; do not attempt any further routing.
   - Structural-invariant check: if you are ever about to assign a request to a specialist it has already been assigned to, or to a second distinct specialist, stop — that violates the invariant above. Trip NO-PROGRESS immediately for that request instead of executing the assignment.

STOP on whichever trips first:
SUCCESS — queue is empty; every request's logged terminal state is done-by-specialist, done-by-fallback, or escalated-to-human with a reason.
BUDGET — total routing attempts reach <MAX_ROUTING_ATTEMPTS> (default 2x queue length — an exact ceiling under the one-specialist-attempt invariant, never spuriously hit by legitimate progress since no request ever needs a 3rd touch), or the wall-clock cap is hit; halt and report remaining pending/in-flight requests.
NO-PROGRESS — (a) pending-queue count is unchanged for <K_STALL_TURNS> consecutive turns (default 3, operator-settable before the run); (b) invariant violation — an attempted second specialist assignment for any request is detected before executing: force that request straight to <HUMAN_ESCALATION_CHANNEL> with a "loop-invariant violated" flag the moment it's detected. (This should never fire given the DECIDE rules above — there is no reroute-to-a-different-specialist path anywhere in this loop — so its firing is a defense-in-depth signal of an upstream contract break, not a routine path.)
BLOCKED — a request needs a capability outside both the frozen roster and the generalist fallback, so it must go to <HUMAN_ESCALATION_CHANNEL> — but if that channel is itself unavailable or unresponsive, halt the entire loop (don't keep processing other requests indefinitely while one is stuck unresolved and unresolvable) and surface exactly which request(s) and why.

MODEL: the router is the cheap/fast, non-integrity-critical component — it proposes but never self-certifies, and it is never trusted to self-flag high-stakes requests; that duty belongs to the independent gate (step 2). The confidence gate is the integrity-critical component and must run as a structurally separate pass with information isolation on every run, unconditionally (mandatory, not conditional on domain difficulty). Additionally, if the domain has ambiguous or adversarial requests, upgrade the gate's model on top of that baseline isolation — isolation and model strength are separate levers; isolation is never the one you trade away for cost.

Return the full corrected prompt object.
```
