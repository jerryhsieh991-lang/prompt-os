# Structured Extraction

`structured-extraction` — 3 loop prompts.

### 1. Structured Extraction loop prompt — Table Extraction from a Document

- **When:** Use when a source document contains one or more tables that must be extracted into structured rows/columns, with every emitted cell independently checked against the source region.
- **Loop:** assess -> one action (extract/fix one table region) -> independent verify (re-render/OCR the source region and diff against emitted cells) -> decide
- **Stop:** SUCCESS: every table region in the document has 100% cell-level match against the independent verifier on its most recent check · BUDGET: <max_iterations> region-extraction attempts reached · NO-PROGRESS: mismatch count across all regions unchanged for 3 consecutive turns despite trying a different extraction strategy · BLOCKED: a region is unreadable (scan quality, merged/rotated cells, no ground-truth-derivable text) and needs a human to transcribe or approve a fallback
- **Model:** Use a mid-to-high reasoning model for layout-heavy extraction; keep the verifier as a separate OCR/text-layer + geometry mechanism so the extractor cannot grade itself.

```text
Goal (frozen): every table in <source_document>, region by region as identified by <region_list> (page + bounding box per table), is represented in <output_format> (CSV/JSON) with every cell exactly matching what an INDEPENDENT verifier — a separate OCR/text-layer extraction plus geometric cell-boundary reconstruction over the SAME source region, never the extracting model re-reading its own output — reports as ground truth for that region. A region counts as done only when the verifier reports 0 mismatched cells (value, row position, and column position all correct) on its most recent check. Freeze <region_list> and the verifier's OCR/geometry config before the loop starts; do not redefine table boundaries or relax cell-matching tolerance mid-run to force a pass.

Per turn: (1) assess — query the verifier's last run (or run it fresh if none exists) to find which region has mismatches, and pick the SINGLE region with the most outstanding mismatched cells; (2) act — extract or re-extract that ONE region only into <output_format>, choosing exactly one strategy this turn (e.g. re-run OCR with a different segmentation, manually reconcile a merged-cell span, fix a header/data-row misalignment) — do not touch any other region; (3) verify — run the independent verifier against that region's source bounding box, computing exact cell-level diff (value + row index + column index) between emitted output and re-derived ground truth; (4) decide — commit the region's output if mismatches dropped to 0 or decreased, revert to the prior attempt if mismatches increased (a regression means the last strategy was wrong), then move to the next-worst region.

Carry forward each turn: per-region mismatch counts over the last 3 turns, which extraction strategy was already tried per region (never retry a failed one verbatim — e.g. don't re-run the same OCR config that just failed), regions committed as done, iterations used, budget remaining.

Stop on the FIRST that trips: SUCCESS — every region shows 0 mismatched cells on its most recent independent verifier run; BUDGET — <max_iterations> extraction attempts reached; NO-PROGRESS — total mismatch count across all regions unchanged for 3 consecutive turns despite a genuinely different strategy each time; BLOCKED — a region's source quality (rotation, low-res scan, ambiguous merged cells) makes ground truth underivable even by the verifier, and needs a human to transcribe the region or approve a documented fallback. Never widen the verifier's match tolerance or hand-edit its ground truth to manufacture a pass — fix the extraction or escalate to BLOCKED.
```

### 2. Structured Extraction: Invoice/Form Field Extraction with Confidence Gating

- **When:** A batch of invoices/forms (PDF/image/scan) must be converted into structured records against a fixed field schema, and every emitted record must clear both a type/range validator and an independently-corroborated confidence check before it's trusted downstream (e.g. auto-posted to an accounting system).
- **Loop:** assess unprocessed-document count -> extract fields for ONE document (with a bounded, logged retry on the same document if needed) -> run independent checker (schema/type/range validation + confidence gate corroborated by a signal the extractor does not control) -> route to accepted.jsonl or review_queue.jsonl (terminal) with a reason code -> decide
- **Stop:** SUCCESS: 0 unprocessed documents AND a final full-corpus audit re-run of the checker script against every record in accepted.jsonl reports 0 schema/range violations and 0 confidence-corroboration mismatches (this is a distinct end-of-run re-audit, not merely a restatement of the per-turn gate — it catches drift, e.g. a mid-run checker/schema edit or file corruption, that per-turn checks by construction cannot). · BUDGET: max_documents documents processed (each document counts once toward this budget regardless of whether it required its one allowed internal retry). · NO-PROGRESS: the identical reason code at the identical field (schema_violation:<field> or low_confidence:<field>) fires on N consecutive distinct documents (default N=5, configurable). This tracks the real stuck-state variable — repeated identical failure — rather than a rolling acceptance-rate average, which mathematically can't move by >1% per 3 documents on any nontrivial batch and would spuriously halt healthy runs. (Do NOT use "acceptance rate unchanged within 1% for 3 consecutive documents" — that measures batch size, not stuck-ness.) · BLOCKED: a document is unreadable/corrupt even after its one allowed retry (different re-OCR/crop attempt); OR a recurring field type in the batch has no corresponding entry in schema.json; OR the independent confidence-corroboration signal (OCR-engine confidence, or the secondary extraction pass) is unavailable/broken for the batch, since without it the confidence gate silently degrades to the extractor self-certifying.
- **Model:** Per-document extraction is a good fit for a mid-tier model (Sonnet-tier) once the schema and confidence thresholds are frozen — it's high-volume and the checker catches most errors mechanically. Use a stronger model only to (a) author schema.json and calibrate confidence_threshold up front, and (b) periodically review the human calibration-audit sample (see carry-forward) to detect drift in the confidence-corroboration signal — a badly calibrated threshold or a broken corroboration signal either floods the review queue or lets bad extractions through silently.

```text
Goal (frozen): every document in <document_batch> is converted to a structured record matching <schema.json> (field names, types, e.g. invoice_number: string, invoice_date: date, total_amount: decimal, vendor_name: string, line_items: array) and is either (a) written to accepted.jsonl because it passes BOTH a deterministic type/range check against schema.json AND a confidence gate that requires agreement between the extractor's self-reported per-field confidence AND an independent corroborating signal on every required field, or (b) written to review_queue.jsonl (a TERMINAL bucket for this run — see Retry semantics below) with a reason code (schema_violation:<field> | low_confidence:<field> | corroboration_mismatch:<field> | unreadable), until every document in the batch is accounted for in accepted.jsonl + review_queue.jsonl, or a stop condition below trips.

Independent verifier (must NOT be the extraction model grading itself, on EITHER axis it gates):
1. Deterministic checker script — validates every extracted field's type and range/format against schema.json (dates parse, amounts are numeric and >=0, required fields non-null, enums in allowed set, line-item amounts sum to total within a documented tolerance). This axis was already independent in the original design and is unchanged.
2. Confidence axis — this is the axis that was NOT independent before and is fixed here. The checker never trusts the extractor's self-reported per-field confidence alone (a model that always reports high confidence would otherwise sail through). Instead the checker requires BOTH of the following to agree before a required field counts as "confident enough":
   a. the extractor's self-reported per-field confidence >= <confidence_threshold>, AND
   b. an independent corroborating signal the extraction model does not generate or control — either (i) the OCR/document-parsing engine's own per-region confidence score for that field's source area (a different system than the extraction model), or (ii) agreement within documented tolerance between the primary extraction and a second, independently-run extraction pass (a different prompt, crop, or deterministic regex/layout template where the field type supports one — invoice_number, dates, amounts). If the two signals disagree, or the corroborating signal is unavailable, the field is treated as NOT confident regardless of the extractor's self-reported number, and the record routes to review_queue.jsonl with corroboration_mismatch:<field>.
   Calibration audit (out-of-loop, human-run, not a per-turn step): every <calibration_sample_n> documents (e.g. every 50), a human spot-checks a random sample of accepted.jsonl and review_queue.jsonl records to measure true precision of the combined gate; results feed back into recalibrating <confidence_threshold> or the corroboration method — the loop itself never adjusts these mid-run.
A record only reaches accepted.jsonl if the checker independently passes both axes; the extractor never self-certifies on either.

Per turn: (1) assess — count documents not yet attempted at all in <document_batch> (this is "unprocessed"; a document mid-retry within its own turn is not double-counted, see step 2); (2) act — pull exactly ONE unprocessed document and extract all fields in schema.json, emitting a confidence score per field plus the independent corroborating signal; if the checker's initial pass fails specifically on unreadable or corroboration_mismatch grounds, the SAME document may be retried exactly ONCE within this turn with something concretely changed (higher-resolution re-OCR, different page crop, the second extraction method) — log what changed; (3) verify — run the independent checker: type/range validation, then the two-signal confidence gate; (4) route — write the record to accepted.jsonl if it clears both checks (after the retry if one was used), otherwise to review_queue.jsonl with the specific reason code (never partially write, never round a low-confidence or mismatched field up to force a pass); this document is now processed and permanently out of "unprocessed" regardless of which file it landed in; (5) decide.

Retry semantics (resolves the earlier gap between SUCCESS and "reprocessing"): review_queue.jsonl is TERMINAL for this loop run — SUCCESS is defined purely on the unprocessed count and the final audit, and a document routed to review_queue is "accounted for" and does not re-enter step (1)'s unprocessed count later in this run. The one bounded, logged retry described in step (2) is the ONLY reprocessing this loop performs, and it happens inside the same turn on the same document before routing, never across turns and never by re-adding a document to the unprocessed pool. Any further reprocessing of a review_queue document (e.g. after a human fixes the schema gap or supplies a better scan) happens as a NEW, separate batch run with its own fresh unprocessed count — not as a resumption of this run's loop.

Human/approval gate: accepted.jsonl records are extraction output only, not a posting authorization — before any accepted record is used to trigger a paid or irreversible downstream action (auto-posting to <accounting_system>, issuing a payment, updating a vendor ledger), a human must review and approve that specific batch of postings. The loop never crosses that line on its own.

Carry forward each turn: documents processed, cumulative acceptance rate (informational only — NOT a stop-condition input), review_queue count broken down by reason code, current consecutive-identical-reason-code streak (the NO-PROGRESS counter, reset whenever the reason code or field changes), top recurring low-confidence/corroboration_mismatch field, iterations used, budget remaining, retry log (which documents used their one retry and what changed).

Stop on the FIRST that trips: SUCCESS — 0 unprocessed documents AND a final full-corpus audit re-run of the checker script against every record in accepted.jsonl reports 0 schema/range violations and 0 confidence-corroboration mismatches (a distinct end-of-run pass, not just a restatement of the per-turn gate); BUDGET — <max_documents> documents processed; NO-PROGRESS — the identical reason code at the identical field fires on N consecutive distinct documents (default N=5) — this tracks real stuck-ness, not a rolling acceptance-rate average; BLOCKED — a document is unreadable/corrupted even after its one allowed retry, or a recurring field type in the batch has no corresponding entry in schema.json, or the independent confidence-corroboration signal is unavailable/broken for the batch. Do not lower <confidence_threshold>, relax schema.json, or disable the corroboration signal mid-run to manufacture a pass — escalate schema gaps or a broken corroboration signal to a human instead.
```

### 3. Structured Extraction with Human-Review Fallback (fixed) — independent field-validator gate

- **When:** Use when every document in a batch must end up either committed to a structured-extraction output file or routed to a human review queue, and you need a genuinely independent correctness check (not the extractor grading its own homework) gating every commit, with zero silent partial-commits and a mechanically checkable stall/miscalibration detector.
- **Loop:** assess -> act -> verify -> decide
- **Stop:** SUCCESS: Every document in the batch is either fully committed (independently validated) or fully queued for review, reconfirmed by a fresh full-batch re-scan that re-runs &lt;field_validator_ruleset&gt; against every field currently in extracted.json with zero failures. · BUDGET: &lt;max_documents&gt; documents processed or &lt;max_iterations&gt; turns used. · NO-PROGRESS: terminal_count (distinct committed + distinct queued documents) stays flat for 3 consecutive turns while review_queue.json still has at least one document to attempt. This can only arise once all never-yet-attempted documents are exhausted and the loop is cycling solely on corrected-input re-queue candidates (step 4b) that keep failing the gate — e.g., a human correction that doesn't actually fix the flagged field. It is a real, reachable condition (not boilerplate) because assess always prefers never-attempted documents first; the flat-count state is only possible once that supply is gone. · BLOCKED: A document resolves zero fields against schema.json at all (wrong form entirely — see verify step's zero-resolvable-fields rule); the review_queue.json backlog exceeds &lt;max_review_backlog&gt;; or the queue rate — (cumulative fields queued) / (cumulative fields evaluated) — exceeds &lt;miscalibration_rate_threshold&gt; after at least &lt;miscalibration_min_documents&gt; documents have been processed, signalling &lt;field_validator_ruleset&gt; itself is miscalibrated and needs a human recalibration call.
- **Model:** Run the act-step extractor and the verify-step's second independent extraction pass (used only for fields with no formalizable rule) on genuinely different configurations — different model, different prompt framing, or different temperature/seed — never the same call re-asked "are you sure." The formal-rule half of the validator (regex/checksum/cross-field arithmetic) should be plain deterministic code, not an LLM call, wherever the field allows it. Sonnet-class models are adequate for the extraction passes themselves; reserve escalation to a stronger model only for designing/debugging the ruleset up front, not for the per-turn loop execution.

```text
Goal (frozen): every document in &lt;document_batch&gt; ends this run with all fields from &lt;schema.json&gt; in exactly one of two states — committed to extracted.json (only if EVERY field on that document clears its applicable check under &lt;field_validator_ruleset&gt;, a deterministic, extractor-independent validator — never the extractor's own self-reported confidence) or written to review_queue.json with a field-level reason code — with zero fields ever committed that failed their independent check, confirmed by a full re-scan that re-runs &lt;field_validator_ruleset&gt; against every committed field at the end of the run.

Per turn:
(1) assess — count never-yet-attempted documents remaining in &lt;document_batch&gt;, and review_queue.json entries that carry a human-supplied corrected input (see step 4b). Pull exactly ONE document, always preferring a never-yet-attempted document over a corrected re-queue candidate — pull a re-queue candidate only when zero never-yet-attempted documents remain. Never pull a batch of many, so a bad extraction can't silently propagate across documents in one turn.

(2) act — run the extractor once against that document's current input (the original document, or the human-corrected input if this is a re-queue candidate), producing a value for every field in schema.json — this is the ONE reversible action this turn. Do not re-extract the same document twice in a row on a hunch, and do not hand-edit a value to "help it pass." The extractor may still emit a self-reported confidence score; log it for diagnostics only — the gate in step 3 never reads it.

(3) verify — run the independent field validator from &lt;field_validator_ruleset&gt; against every field's extracted VALUE, never against any confidence score:
  - if the field has a formalizable rule (format regex, checksum, or cross-field arithmetic against the source document — e.g., invoice total = sum of line items), apply that rule directly to the extracted value;
  - if the field has no formalizable rule, run a second, differently-configured extraction pass over the same input and require exact (or, for numeric fields, within-tolerance) agreement with the first pass — disagreement is an automatic fail for that field regardless of either pass's self-reported confidence;
  - any field that fails its applicable check, or any required field left missing, fails the gate for that document;
  - zero-resolvable-fields rule: if the extractor cannot resolve ANY field against schema.json at all, that document is document-level BLOCKED (its structure doesn't match the schema's shape at all — likely the wrong form entirely) — do NOT route it into review_queue.json with a reason code. If at least one field resolves, proceed field-by-field as above; an individually-mismatched-but-present field gets `schema_mismatch` as its reason code in review_queue.json, not BLOCKED. This zero-vs-nonzero-resolved-fields line is the sole, mechanical test for which bucket a mismatch falls into.

(4) decide — commit the document's fields to extracted.json ONLY if every field passed its independent check; if even one field failed, commit NOTHING for that document and instead write the full field set plus a reason code per failing field (`failed_rule:<rule_id>`, `no_agreement:<value_a>|<value_b>`, `missing_required`, `schema_mismatch`) to review_queue.json. Never partially commit a document — split-state records are how silent data corruption creeps in. A human review event is the only thing that can move a review_queue.json item to extracted.json, via exactly one of two paths:
  (4a) direct override — a human manually verifies the value and writes it straight to extracted.json themselves, deleting the review_queue.json entry; this happens outside the loop and consumes no turn.
  (4b) corrected-input re-queue — a human attaches a corrected INPUT (not a corrected output value — e.g. a re-scanned source page, a disambiguating note) to a review_queue.json entry. This is the "changed input" that licenses one more pass through steps (2)-(4), executed identically to a first-pass document; the loop itself, not the human, writes the resulting commit-or-requeue outcome. If it fails the gate again, it returns to review_queue.json with refreshed reason codes and waits for another human review event. This is not the loop overriding a commit — the field is being freshly re-verified against the same independent gate, not force-committed.

Carry forward each turn: never-yet-attempted documents remaining; review_queue entries awaiting a corrected input vs. entries with one already attached; terminal_count = distinct documents ever committed + distinct documents ever queued (mutually exclusive at any instant — a document moves between queued and committed only via 4a/4b); per-turn (committed vs. queued) outcome; current review_queue.json size; cumulative fields evaluated and cumulative fields queued (for the miscalibration check); which documents have already round-tripped through extraction once (don't verbatim re-extract without a changed input or a human-supplied correction); iterations used; budget remaining.

Stop on the FIRST that trips:
SUCCESS — every document in the batch is either fully committed (independently validated) or fully queued for review, reconfirmed by a fresh full-batch re-scan that re-runs &lt;field_validator_ruleset&gt; against every field currently in extracted.json with zero failures.
BUDGET — &lt;max_documents&gt; documents processed or &lt;max_iterations&gt; turns used.
NO-PROGRESS — terminal_count stays flat for 3 consecutive turns while review_queue.json still has at least one document to attempt (only reachable once never-yet-attempted documents are exhausted and the loop is cycling solely on corrected-input re-queue candidates that keep failing the gate).
BLOCKED — a document resolves zero fields against schema.json at all (see verify step), the review_queue.json backlog exceeds &lt;max_review_backlog&gt;, or the queue rate — (cumulative fields queued) / (cumulative fields evaluated) — exceeds &lt;miscalibration_rate_threshold&gt; after at least &lt;miscalibration_min_documents&gt; documents processed, signalling &lt;field_validator_ruleset&gt; itself is miscalibrated and needs a human recalibration call.

Never lower any threshold or rule in &lt;field_validator_ruleset&gt;, or relax schema.json, mid-run to manufacture a pass — that is the one thing this loop is not allowed to touch on its own.

Parameters: &lt;document_batch&gt;, &lt;schema.json&gt;, &lt;field_validator_ruleset&gt; (replaces the old &lt;confidence_threshold&gt; — a config of per-field format/checksum/cross-field-arithmetic rules plus the numeric tolerance used for double-extraction agreement on fields with no formal rule), &lt;max_documents&gt;, &lt;max_iterations&gt;, &lt;max_review_backlog&gt;, &lt;miscalibration_rate_threshold&gt;, &lt;miscalibration_min_documents&gt;.
```
