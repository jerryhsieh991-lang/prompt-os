# Faithful Summarization

`summarization-faithful` — 2 loop prompts.

### 1. Claim-Traceable Single-Source Summary (per-sentence entailment + repair-or-remove)

- **When:** When condensing one long source into a fixed-length summary where the failure you most need to prevent is fluent paraphrase-drift — a summary sentence that reads as "obviously implied" but inserts detail, overstates, or asserts something the source never says — so every summary sentence must trace to a specific source span an independent checker confirms entails it.
- **Loop:** assess the claim ledger for the first UNTRACED/HALLUCINATED summary sentence, else the first required-`<SCOPE>` aspect with no claim, else the final whole-summary faithfulness pass -> take ONE reversible action (attach/repair a citation, reword the sentence to match its span, add one span-derived claim, or remove an unsupportable one) -> independently verify with the per-claim entailment checker -> commit if it entails / revert if not / park (remove) after the fail-streak / escalate -> decide
- **Stop:** SUCCESS: every surviving summary sentence is TRACED to a cited source span the independent entailment checker confirms entails it, every unsupportable sentence has been removed (scoping the summary down), every required `<SCOPE / FOCUS>` aspect is covered or explicitly waived, length is within `<LENGTH BUDGET>`, and a final independent faithfulness+coverage pass confirms the prose contains exactly the TRACED claims with no uncited orphan sentence. · BUDGET: total-turns-used reaches N = 3 × C + 3, where C = the live claim-ledger row count (starts at max(3, turn-0 sentence count), grows per added scope-gap claim); recompute off the current ledger, never freeze to the turn-0 draft. · NO-PROGRESS: the tuple (TRACED / UNTRACED / HALLUCINATED / REMOVED sets + draft summary text) is unchanged across 2 consecutive turns; a single sentence retried and auto-removed at fail-streak 2 changes the REMOVED set and is progress, not this arm. · BLOCKED: `<SOURCE DOCUMENT>` is unreadable or the citation system errors, a passage is ambiguous in a way only a human can resolve, or a required `<SCOPE / FOCUS>` aspect has no entailing span anywhere in the source after 2 targeted searches and cannot be waived without a human — the source simply not supporting a drafted sentence is NOT blocked, that resolves via removal under SUCCESS.
- **Model:** The summarizer role (drafting, rewording, proposing/searching citations) may run on a cheap/fast model; the entailment VERIFIER must execute as a structurally separate call — a distinct agent invocation or fresh context fed ONLY the sentence and its cited span (plus the scope slice for coverage), never the summarizer's drafting rationale or confidence. It must be independent because a model grading its own summary shares the blind spot that produced the sentence and reads its own inserted detail as "clearly implied" — it is optimizing for fluency, so letting it also score faithfulness is textbook Goodhart and waves through exactly the paraphrase-drift and invented-detail this loop exists to catch. Prefer a stronger model for VERIFY.

```text
GOAL (frozen — do not redefine mid-loop)
Condense <SOURCE DOCUMENT> into a single summary that covers <SCOPE / FOCUS> and stays within <LENGTH BUDGET> (a fixed sentence or word ceiling). Freeze <SCOPE / FOCUS>, <LENGTH BUDGET>, and the meaning of "faithful" at turn 0 — do not widen the scope, raise the ceiling, or relax the standard once turn 1 begins. At turn 0, produce a draft summary and decompose it into an ordered list of atomic claims, one per summary sentence (split any sentence that asserts two independent facts). "Faithful/supported" has exactly one meaning: a claim is TRACED only when it carries an exact source citation — span-ID / page / paragraph + a literally quoted passage — and an independent entailment check confirms that quoted span STATES the claim (entails it), not merely mentions the same topic. A summary sentence with no such span, or whose span only correlates with it, is not faithful no matter how plausible it reads. The task is done only when every surviving summary sentence is TRACED, every sentence that could not be traced has been removed (the summary is allowed to get shorter — a shorter faithful summary beats a complete unfaithful one), every required <SCOPE / FOCUS> aspect is covered by a TRACED claim or explicitly waived, length is within <LENGTH BUDGET>, and a final independent pass confirms the summary text contains exactly the TRACED claims (no uncited orphan sentence, no TRACED claim silently dropped from the prose).

INDEPENDENT VERIFIER
A per-claim entailment/traceability checker that runs as a separate call from the summarizer (see Model note), receiving ONLY the claim sentence and its cited source span — never the summarizer's drafting reasoning, phrasing rationale, or confidence. For each claim it returns exactly one of: ENTAILS (the span states the claim → TRACED), CONTRADICTS (the span asserts something incompatible → HALLUCINATED, the claim misreads the source), or RELATES-ONLY (topical overlap, adjacent detail, inference from surrounding context, but no actual statement of the claim → stays UNTRACED). A separate coverage check confirms an added claim maps to a real, still-uncovered <SCOPE / FOCUS> aspect; the final faithfulness+coverage pass confirms the prose and the TRACED-claim set agree exactly. A summarizer's own "this looks faithful" is not verification: the model that wrote the sentence shares the blind spot that produced it and will read its own inserted detail as "clearly implied" — entailment must be judged by something that never saw the sentence being written, against the literal span alone.

PER-TURN SHAPE
1. ASSESS — target the first claim in summary order that is UNTRACED or HALLUCINATED. If none, target the first required <SCOPE / FOCUS> aspect with no claim covering it. If none, run the final whole-summary faithfulness+coverage pass.
2. ONE ACTION (exactly one, reversible; keep the prior version so it can be reverted):
   - claim targeted — do ONE of: (a) attach/repair a citation by locating a source span that may entail the claim; (b) reword the claim to match what its cited span actually says (drop the overstatement/inserted detail); or (c) if two prior attempts on this claim already failed, remove it from the summary. Never repeat a search that targeted the same section/phrasing for this claim; vary section/paragraph/synonym on retry.
   - scope-gap targeted — add exactly ONE new claim derived from a specific source span that covers the aspect. Never fabricate a claim to "round out" the summary; it must come from a real span. New claim starts UNTRACED.
   - final-pass targeted — no edit; run the whole-summary faithfulness+coverage check.
3. VERIFY — run the independent verifier on the produced candidate: entailment (ENTAILS / CONTRADICTS / RELATES-ONLY) for a claim or reworded claim; coverage-match for an added claim; the full faithfulness+coverage pass for the final pass. A reworded or re-cited claim is a NEW candidate — re-verify from scratch; never carry a prior ENTAILS verdict onto edited text.
4. DECIDE — commit / revert / park / escalate:
   - ENTAILS → commit: mark TRACED, lock the citation into the draft, reset this claim's fail-streak to 0.
   - RELATES-ONLY or CONTRADICTS → revert the edit to the last known-good text (baseline never regresses) and increment this claim's fail-streak; at fail-streak 2, park the claim — remove it from the summary, log it as unsupported-removed, and scope the summary down (this is the mechanism that yields a faithful summary, not a halt).
   - escalate (BLOCKED) only on the conditions below.
   - always increment total-turns-used by 1.

CARRY-FORWARD STATE (compact)
- claim ledger: claim # | summary sentence | status (UNTRACED / TRACED / HALLUCINATED / REMOVED) | cited span-ID + exact quote | search sections already tried | this claim's consecutive-verify-fail streak (resets to 0 only on TRACED or REMOVED). Rows are never deleted, only re-statused — the row count is C, used by BUDGET.
- coverage map: which required <SCOPE / FOCUS> aspects have a TRACED-or-pending claim; which have none.
- length ledger: current summary length vs <LENGTH BUDGET> (never exceed it).
- total turns used this run; current draft summary text.

ACTION BAN
- Never keep an UNTRACED or HALLUCINATED sentence in the final summary — trace it, repair it to match its span, or remove it.
- Never invent, guess, or paraphrase-into-existence a citation; the quoted span must literally state the claim and be specific enough to re-locate.
- Never let the summarizer grade its own faithfulness — VERIFY is a separate call that never sees the drafting reasoning.
- Never widen <SCOPE / FOCUS>, raise <LENGTH BUDGET>, or relax "entails" to "close enough" to make a sentence pass — freezing these is the point.
- Never coerce/reword a sentence to fit a span and then keep the old ENTAILS verdict; the reworded sentence is a new candidate that must re-verify.
- Never add a claim not grounded in a specific source span, even to fill a scope gap or hit the length target.

STOP — halt on the FIRST of these that trips, in order:
SUCCESS — every summary sentence is TRACED (independent entailment confirmed) or REMOVED, every required <SCOPE / FOCUS> aspect is covered or explicitly waived, length is within <LENGTH BUDGET>, and the final independent faithfulness+coverage pass confirms the prose contains exactly the TRACED claims — no uncited orphan sentence, no TRACED claim dropped from the text.
BUDGET — total-turns-used reaches N = 3 × C + 3, where C = the live claim-ledger row count (starts at max(3, turn-0 sentence count), grows by 1 per added scope-gap claim); recompute N off the current ledger each check, do not freeze it to the turn-0 draft.
NO-PROGRESS — the tuple (TRACED set, UNTRACED set, HALLUCINATED set, REMOVED set, draft summary text) is unchanged across 2 consecutive turns. A single claim retried and auto-removed at fail-streak 2 changes the REMOVED set and is progress, not this arm; this arm is only the whole loop stalling on every tracked dimension at once.
BLOCKED — <SOURCE DOCUMENT> is unreadable or the retrieval/citation system errors; a source passage is ambiguous in a way only a human/domain expert can resolve; or a required <SCOPE / FOCUS> aspect has no entailing span anywhere in the source after 2 differently-targeted searches and cannot be waived without a human. The source simply not supporting a drafted sentence is NOT blocked — that resolves via removal under SUCCESS (a scoped-down faithful summary is the correct output, not a stall).
```

### 2. Multi-Document Synthesis Without Contradiction (per-claim support + cross-source contradiction surfacing)

- **When:** When synthesizing one answer across several documents that may disagree, where the failure you most need to prevent is a clean narrative that silently averages or picks a side — asserting a false consensus the sources don't actually share — or invents a bridging claim no document states; every synthesized statement must trace to a source span, and every genuine cross-source disagreement must be surfaced and attributed, not smoothed away.
- **Loop:** assess the claim ledger for the first UNSUPPORTED claim, else the first claim not yet contradiction-scanned against the other documents, else the first MISREPRESENTED (flat despite a confirmed conflict) claim, else a required-scope gap, else the final reconciliation pass -> take ONE reversible action (attach an entailing span, scan the other docs for a contradicting span, rewrite a flat claim into an attributed both-sides disagreement, add one span-derived claim, or remove an unsupportable one) -> independently verify with two checkers (per-claim support + cross-source contradiction detector) -> commit / revert / park / escalate -> decide
- **Stop:** SUCCESS: every synthesized claim is SUPPORTED by a cited entailing span (doc-ID + quote), every cross-source contradiction the detector confirmed is rendered in the text as an explicit attributed disagreement (both sides cited, neither dropped nor averaged), no invented claim survives, every required scope aspect is covered or waived, length is within budget, and a final independent pass confirms the text contains exactly the SUPPORTED/surfaced claims with every contested point rendered as an attributed both-sides statement. · BUDGET: total-turns-used reaches N = 4 × C + 2 × `<N DOCUMENTS>`, minimum 12, where C = the live ledger row count — contradiction scanning scales with the document count, so N does too; recompute off the current ledger and `<N DOCUMENTS>` each check. · NO-PROGRESS: the tuple (SUPPORTED / UNSUPPORTED / CONTESTED-SURFACED / MISREPRESENTED / REMOVED sets + draft text) is unchanged across 2 consecutive turns; a claim auto-removed at fail-streak 2 or a flat claim rewritten into a surfaced disagreement changes a tracked set and is progress, not this arm. · BLOCKED: `<SYNTHESIS QUESTION / TOPIC>` is ambiguous in a way that changes which claims are relevant (needs human disambiguation); the detector finds an apparent conflict it genuinely cannot classify as a true contradiction vs. a terminology/units/definition mismatch without a domain expert; or a document in `<DOCUMENT SET>` fails to parse/retrieve. Documents merely disagreeing is NOT blocked — surfacing that attributed disagreement is the core success path, not a stall.
- **Model:** The synthesizer (drafting, rewording, querying spans) may run on a cheap/fast model; BOTH verifiers — the per-claim support checker AND the cross-source contradiction detector — must run as structurally separate calls, fed only the claim plus the candidate/other-document spans, never the synthesizer's narrative reasoning or confidence. They must be independent because a model synthesizing toward one coherent narrative is biased to smooth over conflict and manufacture consensus — the exact silent-averaging failure this loop exists to catch — so it cannot be trusted to certify its own output as contradiction-free; and the contradiction detector must scan every other document, not accept the synthesizer's word that "the sources agree." The loop SURFACES disagreements; it never adjudicates which source is right — that stays a human/domain call. Prefer a stronger model for both verifiers.

```text
GOAL (frozen — do not redefine mid-loop)
Synthesize a single answer to <SYNTHESIS QUESTION / TOPIC> across <DOCUMENT SET> (N = <N DOCUMENTS> documents), within <LENGTH BUDGET / SCOPE>. Freeze the question, scope, and length at turn 0. At turn 0, produce a draft synthesis and decompose it into an ordered list of atomic claims (split any sentence asserting two independent facts); each claim is proposed with the source citation(s) it rests on. Two things define a faithful synthesis here, and both are required: (1) SUPPORT — every claim carries an exact citation (doc-ID + span/paragraph + literal quote) and an independent check confirms that span ENTAILS the claim, not merely relates to it; nothing is invented, no bridging claim survives without a span. (2) CONTRADICTION-HONESTY — wherever two or more documents genuinely disagree on a point the synthesis makes, the synthesis renders that as an explicit, attributed disagreement ("<Doc A> reports X; <Doc B> reports not-X"), each side cited; it never silently averages the two, picks one and drops the other, or blends them into a consensus none of the sources actually state. The loop surfaces contradictions; it does NOT decide who is right — adjudicating a genuine conflict is a human/domain decision, not something the loop resolves by fiat. The task is done only when every claim is SUPPORTED, every detector-confirmed cross-source contradiction is surfaced-and-attributed, no invented claim remains, every required scope aspect is covered or explicitly waived, length is within budget, and a final independent pass confirms the text contains exactly the SUPPORTED/surfaced claims with every contested point rendered as an attributed both-sides statement.

INDEPENDENT VERIFIER
Two checkers, both run as calls structurally separate from the synthesizer (see Model note), each fed only the claim and the relevant spans — never the synthesizer's narrative reasoning or confidence:
- Per-claim support checker: does the cited span ENTAIL the claim (states it) → provenance confirmed; or RELATE-ONLY (topical overlap, no statement) → claim stays UNSUPPORTED; or CONTRADICT the claim → the claim misreads its own source.
- Cross-source contradiction detector: given the claim and spans drawn from the OTHER documents, does any other document assert the negation or an incompatible value/finding? It must scan every other document, not sample. If a real conflict exists and the claim is rendered as a flat assertion → mark MISREPRESENTED (must be repaired into an attributed disagreement). If the claim is already rendered as an attributed disagreement → confirm both sides are cited and correctly attributed. If the apparent conflict is really a terminology/units/scope mismatch rather than a true disagreement, it says so (and does not force a false "disagreement").
A synthesizer's own "this reads consistent and faithful" is not verification: a model optimizing for one smooth narrative is precisely the thing that averages away disagreement and reads its own bridging inference as "supported." Contradiction-freedom and support must be judged by something that never wrote the synthesis, against the literal spans.

PER-TURN SHAPE
1. ASSESS — target, in order: the first UNSUPPORTED claim; else the first SUPPORTED claim not yet contradiction-scanned against all other documents; else the first MISREPRESENTED claim (flat despite a confirmed conflict); else the first required scope aspect with no claim; else the final reconciliation pass.
2. ONE ACTION (exactly one, reversible; keep the prior version):
   - support a claim — locate ONE candidate entailing span (doc-ID + quote) for it; vary the document/section/phrasing from prior attempts on this claim.
   - contradiction-scan a claim — pull the spans from the other documents that bear on the claim's point, for the detector to judge. This is a check, not a new claim.
   - repair a MISREPRESENTED claim — rewrite the flat assertion into an explicit attributed disagreement, each side cited to its document.
   - scope-gap — add ONE new claim grounded in a specific source span; never fabricated. Starts UNSUPPORTED.
   - remove — after two failed support attempts, delete an unsupportable claim (park as unsupported-removed).
   - final-pass — no edit; run the whole-synthesis support+contradiction+coverage reconciliation.
3. VERIFY — run the relevant independent checker(s): support checker on a newly cited/reworded claim; contradiction detector on a scanned claim (across ALL other docs) and to confirm a repaired claim's both-sides attribution; the full reconciliation on the final pass. Any reworded/re-cited/repaired claim is a NEW candidate — re-verify from scratch; never carry a prior verdict onto edited text.
4. DECIDE — commit / revert / park / escalate:
   - support ENTAILS and no unsurfaced conflict → commit: mark SUPPORTED, lock citation, reset fail-streak.
   - detector finds a real conflict on a flat claim → mark MISREPRESENTED and route it to the repair action (not a halt — this is how disagreement gets surfaced); a correctly attributed disagreement → mark CONTESTED-SURFACED and commit.
   - RELATE-ONLY / CONTRADICTS-own-source / failed repair → revert to the last known-good text (baseline never regresses) and increment the claim's fail-streak; at fail-streak 2 on support, park (remove) the claim.
   - escalate (BLOCKED) only on the conditions below.
   - always increment total-turns-used by 1.

CARRY-FORWARD STATE (compact)
- claim ledger: claim # | claim text | status (UNSUPPORTED / SUPPORTED / CONTESTED-SURFACED / MISREPRESENTED / REMOVED) | citation(s) doc-ID + quote | which other docs have been contradiction-scanned for this claim | attempts tried | consecutive support-fail streak (resets on SUPPORTED or REMOVED). Rows are never deleted, only re-statused — the row count is C, used by BUDGET.
- contradiction log: point of disagreement | Doc-A span/claim | Doc-B span/claim | surfaced-in-text? Y/N.
- coverage map: required scope aspects with vs. without a claim.
- total turns used; current draft synthesis text; length vs <LENGTH BUDGET / SCOPE>.

ACTION BAN
- Never assert a synthesized claim with no cited entailing span; never invent a bridging fact or a citation to connect two sources.
- Never silently average, blend, or pick-one-and-drop when documents genuinely disagree — a confirmed cross-source contradiction must be surfaced and attributed, never smoothed into a false consensus.
- Never let the loop adjudicate which source is correct; it surfaces the disagreement and leaves the call to a human/domain expert.
- Never let the synthesizer grade its own support or contradiction-freedom — both checks are separate calls that never see the synthesizer's reasoning, and the contradiction detector must scan every other document, not trust "the sources agree."
- Never carry a prior verdict onto a reworded/repaired claim; re-verify the new text.
- Never widen scope, raise the length ceiling, or relax "entails"/"is a real contradiction" to make the narrative cleaner.

STOP — halt on the FIRST of these that trips, in order:
SUCCESS — every claim is SUPPORTED (independent entailment) or REMOVED, every detector-confirmed cross-source contradiction is surfaced as an attributed both-sides statement (none averaged or dropped), no invented claim survives, every required scope aspect is covered or explicitly waived, length is within budget, and the final independent reconciliation confirms the text contains exactly the SUPPORTED/CONTESTED-SURFACED claims — no orphan/uncited statement, no surfaced disagreement collapsed back to one side.
BUDGET — total-turns-used reaches N = 4 × C + 2 × <N DOCUMENTS>, minimum 12, where C = the live claim-ledger row count (starts at max(3, turn-0 claim count), grows by 1 per added scope-gap claim); recompute N off the current ledger and the document count each check — contradiction scanning scales with <N DOCUMENTS>, so the budget does too.
NO-PROGRESS — the tuple (SUPPORTED set, UNSUPPORTED set, CONTESTED-SURFACED set, MISREPRESENTED set, REMOVED set, draft text) is unchanged across 2 consecutive turns. A claim auto-removed at fail-streak 2, or a MISREPRESENTED claim rewritten into a surfaced disagreement, changes a tracked set and is progress; this arm is only the whole loop stalling on every dimension at once.
BLOCKED — <SYNTHESIS QUESTION / TOPIC> is ambiguous in a way that changes which claims are relevant (needs human disambiguation before claims can be judged); the detector finds an apparent conflict it genuinely cannot classify as a true contradiction vs. a terminology/units/definition mismatch without a domain expert; or a document in <DOCUMENT SET> fails to parse or retrieve. Documents genuinely disagreeing is NOT blocked — surfacing the attributed disagreement is the core success path, not a stall.
```
