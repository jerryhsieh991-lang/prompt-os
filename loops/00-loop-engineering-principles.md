# Loop-engineering principles

A reliable agent-loop prompt turns an open-ended task into a bounded control loop: a frozen, mechanically-checkable goal; small reversible steps; an independent verifier that decides "done"; and an explicit, multi-armed stop condition so the loop halts on success, budget-exhaustion, non-progress, OR a block — whichever trips first. The prompt should prescribe a light per-turn shape (assess → one action → verify → decide) and carry compact state forward, while resisting the two classic failure modes: never terminating (no budget / self-grading / undetected oscillation) and never staying on-target (scope creep, gold-plating, moving the goalposts). Keep the prompt minimal-with-strong-heuristics — the structure enforces the loop, the model supplies the judgment; over-prescriptive mega-prompts degrade frontier models.

## Principles

- **Anchor to a measurable, frozen goal** — State the exit target as something checkable every turn — tests pass, metric >= threshold, no findings survive verification, judge rubric satisfied. If success can't be evaluated mechanically, define the judge/rubric BEFORE the loop starts. Freeze the goal at loop start so 'done' means the same thing on turn 1 and turn 20.
- **One reversible action per turn, then observe** — Each iteration = the smallest coherent change plus a single verification, so any regression is attributable to exactly one action and can be rolled back cleanly. Bundling several edits before checking destroys the observe->attribute signal and makes rollback all-or-nothing. This is what makes build->verify and refactor-in-small-steps loops debuggable.
- **Make the stop condition explicit and multi-armed** — Enumerate every exit up front and halt on the FIRST that trips, not only on success: (a) SUCCESS — goal met and independently verified; (b) BUDGET — max iterations / tokens / wall-clock reached; (c) NO-PROGRESS — target metric hasn't improved for K turns; (d) BLOCKED — needs human input or an unavailable resource. A loop with only a success exit is an infinite loop waiting to happen.
- **Verify with an independent signal, not self-assessment** — The mechanism that decides 'done' must differ from the thing being optimized: compiler, test suite, linter, a fixed eval harness, or a separate judge. A model narrating its own confidence is not verification. Prefer ground truth (does it run? does the test pass? does the claim survive an adversarial check?) over self-graded 'looks correct'. Beware Goodhart: don't verify with the exact signal you optimized against.
- **Separate actor, verifier, and judge** — In build->verify, red-team/verify, and judge->refine loops, run the critique in a fresh frame or a distinct role so it doesn't inherit the author's blind spots and rationalizations. The refiner should act ONLY on concrete verifier findings, not on its own re-reading of its own work.
- **Carry compact state across turns; don't re-derive or re-dump** — Maintain a running scratchpad — goal, what's been tried, current best, last verifier result, remaining budget — and feed the distilled state forward rather than the whole transcript. This prevents context bloat, lets the agent avoid repeating dead ends, and makes 'remaining budget' visible each turn so it triages toward the highest-value next action.
- **Detect and break non-progress and oscillation** — Track the metric/diff per turn. If it flatlines for K turns, or the agent cycles A->B->A, force a strategy change or halt — never keep looping on a stalled trajectory. Ban verbatim repetition of a failed action: a retry must differ in approach, not just re-attempt the same thing hoping for a different result.
- **Fail loud after repeated failure; escalate, don't grind** — If the same step fails ~3 times, stop the loop, surface the concrete error and everything already tried, and request human guidance. This is cheaper and safer than blindly burning the iteration budget, and it converts a silent runaway loop into an actionable handoff.
- **Freeze scope and ban gold-plating** — The loop closes the defined gap and nothing else. Ideas, edge cases, and 'while I'm here' improvements discovered mid-run go to a backlog, not into this run. The target is 'good enough to pass the exit criterion', not 'as good as possible' — scope creep is the main way loops that don't infinite-loop still fail to terminate usefully.
- **Preserve a loop invariant: known-good workspace each turn** — End every iteration in a state you'd be willing to keep — compiles, tests green or explicitly-marked-red. Make it literal where possible: commit on improvement, revert (git reset) on regression, so a bad turn can never corrupt the baseline. This guarantees the loop's 'current best' only ever moves forward or holds.
- **For research/exploration loops, define saturation ('dry')** — Stop when new sources or searches stop changing the answer — no new claims, corroboration converges — not after a fixed hit count and not when the agent is merely tired. Verify each claim adversarially (independent corroboration) before it counts toward the answer, so 'dry' means evidence-saturated, not effort-exhausted.
- **Prescribe a light, consistent per-turn shape** — Give the loop a small fixed turn structure — assess current state vs goal -> choose ONE action -> act -> verify -> decide continue/stop — and keep everything else minimal-with-strong-heuristics. The rigid part is the loop control (one action, verify, check all stop arms); the flexible part is the model's judgment about which action. Mega-prompts that over-script the reasoning degrade frontier models.

## Antipatterns (what makes a loop fail)

- No stop condition — 'keep iterating until it's perfect', which never terminates because 'perfect' is never mechanically true.
- Only a success exit and no budget/no-progress/blocked arms, so a loop that can't reach success runs forever.
- Self-grading: the agent that produced the output also declares it correct, with no independent verifier or ground truth.
- Vague goal ('make it better', 'clean this up') with no per-turn checkable criterion, so 'done' drifts turn to turn.
- Batching many edits/actions per turn, so when verification fails you can't attribute the regression or roll back cleanly.
- Retrying the exact failed action verbatim, hoping for a different result, instead of changing approach.
- Undetected oscillation — the agent flips A->B->A between two states while reporting 'progress'.
- Verifying with the same signal that was optimized (Goodhart), e.g. tuning to a metric then citing that metric as proof of done.
- Scope creep / gold-plating: adding new features or edge cases discovered mid-loop instead of parking them and closing the defined gap.
- Moving the goalposts to declare victory — redefining success downward when the real criterion won't be met.
- Passing the full raw transcript every turn (context bloat) instead of a compacted running state, causing drift and lost budget.
- Treating 'no errors this turn' or 'this step looks fine' as completion without re-checking the actual goal criterion.
- Grinding blindly through the budget on a repeatedly-failing step instead of halting and escalating to a human.
- Leaving the workspace in a broken/uncommitted state mid-loop, so one bad turn corrupts the baseline the loop is building on.
- Ending a research loop on a fixed count or fatigue rather than evidence saturation, and counting unverified claims as findings.
