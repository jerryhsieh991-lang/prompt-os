# RAG Answer

`rag-answer` — 4 loop prompts.

### 1. Multi-Hop RAG Answer (with cross-hop synthesis check)

- **When:** When answering a multi-hop question that requires chaining facts across separate documents in a corpus, where the failure mode you most need to guard against is a plausible-looking but unsupported link between hops (e.g. assuming "the company" in hop 2 is the same entity resolved in hop 1) — not just a bad retrieval.
- **Loop:** assess hop ledger + link table for the first UNRESOLVED/CONTESTED hop, else the first unjustified adjacent-hop link, else the stability check -> issue ONE retrieval / link-evidence / re-derivation action -> verify independently (entailment for hops, joint-referent evidence for links, deductive-closure for synthesis, exact-match for stability) -> update ledger, link table, per-hop fail-streaks, and draft -> decide
- **Stop:** SUCCESS: Every hop is RESOLVED, every adjacent hop-to-hop link is explicitly justified by evidence (not assumed), synthesis verification confirms the answer follows deductively from the linked chain alone, AND the re-derived answer is stable across 2 consecutive stability passes. · BUDGET: Total turns used this run — retrieval turns AND synthesis/stability turns both count — reaches N, where N = 3 x (number of hops) + 3, minimum 10. Size N to the actual hop count when instantiating this prompt; do not leave it as an unsized placeholder. · NO-PROGRESS: Trips on EITHER: (a) any single hop's consecutive-verification-fail streak reaches 2 — tracked per hop in the ledger, persisting even if ASSESS targets other hops in between, so it can't be dodged by ASSESS bouncing between hops; or (b) the stability check oscillates — the re-derived draft answer changes on 2 consecutive stability passes without ever landing on 2 consecutive identical passes. On this arm: halt and report the stalled hop or the oscillating answer. Do NOT reformulate, merge, or reword the frozen hop decomposition to route around it — the hop chain stays frozen for the life of this run; redefining it is a human decision made between runs, not a mid-loop escape hatch. · BLOCKED: A hop appears to need evidence outside the corpus (the corpus genuinely doesn't contain the answer), a retrieved passage is ambiguous in a way only a human/domain expert can resolve, or synthesis verification repeatedly (2x) finds the same cross-hop link unjustifiable with the available corpus. Halt and surface exactly which hop or link stalled and why.
- **Model:** Use a strong reasoning model (Opus 4.8 xhigh or Fable 5) for every VERIFY sub-step — entailment-vs-correlation judgment on retrieved passages, and especially the synthesis-verification pass that checks the cross-hop deduction itself. That synthesis check is exactly where weaker models rationalize a plausible-looking but unsupported link between hops, which is the specific failure mode this loop design exists to prevent. A cheaper/faster model is fine for formulating retrieval queries in the ACT step.

```text
GOAL (frozen): Answer <QUESTION> using only evidence retrieved from <DOCUMENT CORPUS/INDEX>. Decompose the question into an ordered chain of atomic reasoning hops up front, at turn 0, and freeze that decomposition for the entire run — no hop may be added, removed, merged, or reworded once turn 1 begins. Each hop counts as RESOLVED only when it has a direct quote/passage-ID from the corpus that entails it, confirmed by a verification pass independent of the retrieval that found it. The task counts as done only when every hop is RESOLVED, every adjacent pair of hops has an explicitly justified link (proof that the entity/fact carried from hop N into hop N+1 is actually the same entity/fact, not an assumed coreference), and the final answer follows deductively from the linked hop chain alone, confirmed by a synthesis-verification pass distinct from both the per-hop retrieval-verifier and the stability check.

Carry forward this state across turns:
- hop ledger: hop # | hop question | status (UNRESOLVED/RESOLVED/CONTESTED) | supporting passage-ID(s)/quote | retrieval queries already tried for this hop | this hop's consecutive-verification-fail streak (persists across turns even while ASSESS is targeting a different hop in between; resets to 0 only when this specific hop resolves)
- cross-hop link table: (hop i -> hop i+1) | justified? Y/N | justifying quote/passage-ID — one row per adjacent hop pair, populated only once both hops in the pair are RESOLVED
- total turns used this run (every turn counts toward this — retrieval turns AND synthesis/stability turns alike; this is what BUDGET's N measures)
- current draft answer, plus a count of consecutive stable stability-check passes (meaningful only once every hop is RESOLVED and every link is justified)

Each turn:
1. ASSESS — if any hop is UNRESOLVED or CONTESTED, target the first such hop in chain order. Otherwise, if any adjacent-hop link is unjustified, target that link. Otherwise, target the final-answer stability check.
2. ACT (one reversible action):
   - hop targeted: issue exactly ONE retrieval query against the corpus for that hop. Never repeat a prior query string verbatim for the same hop; a retry must target a different phrasing/section/document than the last attempt on this hop.
   - link targeted: pull the specific passage(s) needed to check whether hop i's resolved fact and hop i+1's resolved fact refer to the same entity/event. This does not create a new hop — it only checks the join between two already-frozen, already-resolved hops.
   - stability targeted: re-derive the final answer from the ledger and link table alone, with no new retrieval.
3. VERIFY (independent of whichever step produced the candidate):
   - hop verification: does the retrieved passage ENTAIL the hop (state the needed fact) or merely correlate with it (topical overlap, related terms, no actual statement of the fact)? Confirm provenance (document ID/section) is loggable. Entailment resolves the hop; correlation-only leaves it UNRESOLVED. If two retrieved passages for the same hop conflict, mark CONTESTED rather than picking one.
   - link verification: does the pulled evidence show hop i and hop i+1 genuinely share the same referent, or is the link an unstated assumption? "Plausible" is not "justified" — an unproven link stays open.
   - synthesis verification (runs once all hops are RESOLVED and all links are justified, before the first stability pass, and again any time the draft changes): does the final answer follow deductively from the ledger + link table alone, with no leap the evidence doesn't support? If not, identify the weakest hop or link, reopen it to CONTESTED, and return the loop to step 1 — this is what actually guards against a plausible-but-unsupported cross-hop deduction, which no per-hop check catches on its own.
   - stability verification: does this pass's re-derived answer match the prior draft exactly?
4. DECIDE:
   - hop targeted: if newly RESOLVED (or a CONTESTED hop is now settled by the new evidence), update the ledger and reset THAT hop's fail-streak to 0. Otherwise increment only that hop's fail-streak.
   - link targeted: if justified, record it in the link table with its justifying quote; otherwise leave it open and log what's still missing.
   - stability targeted: if this pass matches the prior draft, increment the stable-pass counter; if it changed, update the draft, reset the stable-pass counter to 0, and log the change as an oscillation event.
   - always increment total-turns-used by 1, regardless of which branch ran.

STOP on the first of these that trips, in order:
- SUCCESS: every hop RESOLVED, every adjacent-hop link justified, synthesis verification passed, and the re-derived answer is stable across 2 consecutive stability passes.
- BUDGET: total-turns-used reaches N, where N = 3 x (number of hops) + 3, minimum 10 (size N to the actual hop count when you instantiate this prompt — do not leave it as an unsized placeholder).
- NO-PROGRESS: either (a) any single hop's consecutive-fail streak reaches 2 (evidence keeps being retrieved for that hop, with different queries, but keeps failing entailment), or (b) the stability check oscillates — the draft answer changes on 2 consecutive stability passes without ever reaching 2 consecutive identical passes. Halt and report the specific stalled hop or the oscillating answer. Do not redefine, merge, or reword the frozen hop decomposition to work around this — that decision belongs to a human between runs, not to this loop mid-run.
- BLOCKED: a hop appears to require evidence outside <DOCUMENT CORPUS/INDEX> (the corpus genuinely doesn't contain the answer), a retrieved passage is ambiguous in a way only a human/domain expert can resolve, or synthesis verification repeatedly (2x) finds the same cross-hop link unjustifiable with the available corpus. Halt and surface the specific hop or link and why.

Never mark a hop RESOLVED on the retrieving pass's own say-so — verification must be a separate check against the entailment criterion, not the retrieval step's confidence score. Never let a CONTESTED hop silently resolve by ignoring the conflicting passage — log both and let VERIFY adjudicate with a fresh, targeted query. Never let the final answer be accepted on the stability check's self-consistency alone — synthesis verification (that the answer actually follows from the linked chain, and that each cross-hop link is evidenced rather than assumed) must pass independently before any stability pass starts counting toward SUCCESS.

Report: final answer, full hop ledger with passage citations, cross-hop link table with justifying quotes, which arm fired, and (if BLOCKED or NO-PROGRESS) exactly which hop or link stalled and why.
```

### 2. Refuse-When-Unsupported RAG Answer (per-claim grounding + calibrated refusal)

- **When:** When answering a question against a fixed corpus where the dominant failure mode is a fluent answer that quietly asserts more than the corpus actually supports — not missing hops, but ordinary claim-level overreach (extrapolation, filled-in "obvious" detail, stale training-data leakage). Correctly refusing, in full or in part, is a valid and desired terminal state here, not a failure — this is what distinguishes the loop from the multi-hop-synthesis loop above, where every hop must eventually resolve.
- **Loop:** assess the draft's claim ledger for the first UNSUPPORTED/CONTRADICTED claim, else a coverage gap (part of the question no claim addresses yet), else the final refusal-scope decision -> issue ONE retrieval query or ONE new candidate-claim addition -> verify independently per claim (entailment-vs-extrapolation, contradiction check) -> update ledger, per-claim fail-streaks, and draft/refusal state -> decide
- **Stop:** SUCCESS: every claim remaining in the draft is SUPPORTED by a cited passage, every claim that stayed UNSUPPORTED after exhausting retries has been explicitly dropped (partial refusal) or, if that empties the answerable part of the question, the whole question is refused (full refusal) — either way a final independent coverage-match pass confirms the answer text contains no claim absent from the ledger and the ledger contains no SUPPORTED claim silently omitted from the text. · BUDGET: total turns used (retrieval turns AND claim-addition/coverage/refusal-decision turns all count) reaches N = 3 x (number of claims in the turn-0 draft, minimum 3) + 3, minimum 10 — size N to the actual draft when instantiating. · NO-PROGRESS: the ledger's overall state (set of SUPPORTED claims, set of UNSUPPORTED claims, set of dropped claims) is byte-identical across 2 consecutive turns — no claim resolved, none dropped, none added — meaning ASSESS/ACT are cycling without changing anything checkable. (A single claim retried with reformulated queries and still failing after 2 consecutive attempts is not NO-PROGRESS by itself — DECIDE auto-drops that claim and the loop keeps moving; NO-PROGRESS is reserved for the whole loop stalling.) · BLOCKED: the question is ambiguous in a way that changes which claims would even be relevant (needs human disambiguation before claims can be judged), two retrieved passages contradict each other on a load-bearing claim in a way no further retrieval can resolve, or the retrieval system itself errors/times out. Halt and surface exactly which claim or ambiguity stalled and why. Note: the corpus simply not containing the answer is NOT a BLOCKED condition — that resolves through SUCCESS as a refusal, since correctly saying "the corpus doesn't support this" is the correct output, not a stall.
- **Model:** Use a strong reasoning model (Opus 4.8 xhigh or Fable 5) for every claim VERIFY sub-step — this is exactly where a weaker model rationalizes "close enough" and lets a claim through that the passage doesn't actually state, which is the specific failure mode this loop exists to prevent. Drafting candidate claims and formulating retrieval queries in ACT can run on a cheaper/faster model.

```text
GOAL (frozen): Answer <QUESTION> using only evidence retrieved from <DOCUMENT CORPUS/INDEX>. At turn 0, produce an initial draft answer and decompose it into an ordered list of atomic claims — each claim a single checkable assertion. Freeze the question's scope at turn 0; claims may be added, resolved, or dropped as the loop runs, but the question being answered does not change mid-run. A claim counts as SUPPORTED only when a directly quoted passage from the corpus entails it (states the fact, not merely correlates with it), confirmed by a verification pass independent of whichever step drafted or retrieved for the claim. The task counts as done only when the final answer text contains exactly the SUPPORTED claims (no more, no less) — every UNSUPPORTED claim has been either resolved, dropped with the answer scoped down around it (partial refusal), or has caused a full refusal if dropping it leaves no answerable claims for the question asked — and an independent coverage-match pass confirms the answer text and the claim ledger agree exactly.

Carry forward this state across turns:
- claim ledger: claim # | claim text | status (UNSUPPORTED/SUPPORTED/CONTRADICTED/DROPPED) | supporting passage-ID(s)/quote (if SUPPORTED) | retrieval queries already tried for this claim | this claim's consecutive-verification-fail streak (resets to 0 only when the claim resolves to SUPPORTED or is DROPPED)
- coverage map: which parts of <QUESTION> are addressed by at least one SUPPORTED-or-still-pending claim, and which parts have no claim yet
- refusal state: NONE (full answer expected) / PARTIAL (list of dropped claims + the scoped-down answer that remains) / FULL (question cannot be answered from the corpus at all)
- total turns used this run (every turn counts — retrieval, claim-addition, coverage, and refusal-decision turns alike; this is what BUDGET's N measures)
- current draft answer text

Each turn:
1. ASSESS — if any claim is UNSUPPORTED or CONTRADICTED, target the first such claim in ledger order. Otherwise, if the coverage map shows a part of <QUESTION> with no claim addressing it, target that gap. Otherwise, target the final refusal-scope decision.
2. ACT (one reversible action):
   - claim targeted: issue exactly ONE retrieval query against the corpus for that claim. Never repeat a prior query string verbatim for the same claim; a retry must target a different phrasing/section/document than the last attempt on this claim.
   - coverage gap targeted: add exactly ONE new candidate claim to the ledger that addresses the gap, derived only from evidence already surfaced or a fresh targeted retrieval — never fabricated to "fill in" the answer. New claim starts UNSUPPORTED.
   - refusal-scope targeted: no retrieval — recompute the draft answer from the ledger's SUPPORTED claims only, and classify refusal state as NONE/PARTIAL/FULL based on whether every part of the coverage map is satisfied, partly satisfied, or unsatisfied.
3. VERIFY (independent of whichever step produced the candidate):
   - claim verification: does the retrieved passage ENTAIL the claim (states the fact as written) or merely correlate with it (topical overlap, adjacent detail, no actual statement)? Confirm citeable provenance (document ID/section). Entailment resolves to SUPPORTED; correlation-only leaves it UNSUPPORTED. If a passage states something that conflicts with the claim, mark CONTRADICTED rather than SUPPORTED or UNSUPPORTED.
   - coverage verification: does the newly added claim actually correspond to a real, unaddressed part of <QUESTION> — not a restatement of an existing claim or an out-of-scope addition?
   - refusal-scope verification (runs once no claim is UNSUPPORTED/CONTRADICTED and no coverage gaps remain, and again any time the draft or refusal state changes): does the draft answer text contain every SUPPORTED claim and no others? Does the stated refusal scope (NONE/PARTIAL/FULL) actually match what the ledger supports — e.g. is it claiming NONE (full answer) when a claim was in fact dropped? This is the check that prevents silent overreach from surviving into the final answer.
4. DECIDE:
   - claim targeted: if newly SUPPORTED or a CONTRADICTED claim is now settled by new evidence, update the ledger and reset that claim's fail-streak to 0. Otherwise increment only that claim's fail-streak; if the streak reaches 2, mark the claim DROPPED (not a halt — this is the mechanism that produces partial refusal) and log why.
   - coverage gap targeted: add the new claim to the ledger as UNSUPPORTED if it passed coverage verification; otherwise discard it and log why (no ledger change).
   - refusal-scope targeted: update refusal state (NONE/PARTIAL/FULL) and the draft answer text to match the ledger exactly.
   - always increment total-turns-used by 1, regardless of which branch ran.

STOP on the first of these that trips, in order:
- SUCCESS: no claim remains UNSUPPORTED or CONTRADICTED (each has resolved to SUPPORTED or DROPPED), no coverage gap remains unaddressed, refusal state is finalized (NONE/PARTIAL/FULL), and the independent coverage-match pass confirms the answer text and ledger agree exactly.
- BUDGET: total-turns-used reaches N, where N = 3 x (number of claims in the turn-0 draft, minimum 3) + 3, minimum 10 (size N to the actual draft when you instantiate this prompt).
- NO-PROGRESS: the ledger's overall state (SUPPORTED set, UNSUPPORTED set, DROPPED set) is unchanged across 2 consecutive turns — nothing resolved, nothing dropped, nothing added. Halt and report the frozen ledger state. (Per-claim retries that individually fail twice are not this arm — DECIDE auto-drops that one claim and the loop continues; this arm is only for the whole loop stalling.)
- BLOCKED: the question is ambiguous in a way that changes which claims are even relevant and needs human disambiguation before claims can be judged; two retrieved passages contradict each other on a load-bearing claim in a way no further retrieval can resolve; or the retrieval system itself errors or times out. Halt and surface the specific claim or ambiguity and why. The corpus simply lacking the answer is not BLOCKED — that resolves via SUCCESS as a refusal.

Never mark a claim SUPPORTED on the drafting or retrieving step's own say-so — verification must be a separate pass checking entailment, not the retrieval step's confidence. Never let a CONTRADICTED claim silently resolve by picking whichever passage came first — log both and let VERIFY adjudicate with a fresh, targeted query. Never let the final answer be accepted on the drafting step's self-assessment of "supported enough" — the coverage-match pass (that the answer text is exactly the SUPPORTED-claim set, and that the stated refusal scope isn't understating what got dropped) must pass independently before SUCCESS fires. A full or partial refusal is a correct, first-class outcome of this loop, not a fallback to be avoided — never let refusal-scope verification get relaxed under pressure to "just answer something."

Report: final answer (or refusal statement, full or partial), full claim ledger with citations for every SUPPORTED claim and reasons for every DROPPED claim, refusal state, which arm fired, and (if BLOCKED or NO-PROGRESS) exactly which claim or ambiguity stalled and why.
```

### 3. Structured Extraction (RAG Answer family) — schema-validated, cell-cited

- **When:** When extracting a defined set of fields (e.g. invoice/contract/form fields) from a document or document set into a structured record, where the failure mode you most need to guard against is a value that looks right but either violates the target schema's type/format or has no exact, checkable source location — not just a missed field.
- **Loop:** assess field ledger for the first UNFILLED/CONTESTED field, else the first field never search-attempted twice for NOT-PRESENT determination -> issue ONE locate-and-propose action for exactly that field -> verify independently via (1) schema validator on the proposed value and (2) a citation-match check that the quoted source span literally supports the value -> update ledger, draft record, and per-field fail-streak -> decide
- **Stop:** SUCCESS: Every required field in <SCHEMA> is FILLED with a value that passes schema validation AND has a citation (doc-ID/page/section/cell + exact quoted span) confirmed by an independent match check to actually contain/entail that value, OR is explicitly marked NOT-PRESENT-IN-DOCUMENT after 2 independently-targeted searches; every optional field is similarly resolved or explicitly skipped; and the assembled record as a whole passes full schema validation in one final pass. · BUDGET: Total turns used this run reaches N, where N = 2 x (number of schema fields) + 3, minimum 8. Size N to the actual field count when instantiating this prompt — do not leave it unsized. Every turn counts (locate turns and re-verification turns alike). · NO-PROGRESS: Trips when any single field's consecutive-verification-fail streak reaches 2 — tracked per field in the ledger, persisting even if ASSESS visits other fields in between, so it can't be dodged by bouncing across fields. A "fail" is either the schema validator rejecting the value or the citation-match check finding the cited span does not support it. On this arm: halt and report the exact field, its last proposed value, its last citation, and which check failed. Do not loosen the schema or accept an uncited/self-asserted value to route around it — that is a human decision made between runs, not a mid-loop escape hatch. · BLOCKED: Either: (a) a required field is confirmed genuinely absent from the document (2 independent searches targeting different sections/synonyms both fail to locate it) and the schema gives no default — needs a human to waive the requirement or supply the value externally; or (b) the schema itself is ambiguous or self-contradictory for a field (e.g. conflicting type/format constraints, an enum that doesn't cover an observed real value) — needs a human to fix the schema, not the loop to guess an interpretation. Halt and surface exactly which field and why.
- **Model:** Use a strong reasoning model (Opus 4.8 xhigh or Fable 5) for the citation-match/entailment check specifically — that is where a plausible-looking but unsupported value slips through (the model paraphrases or infers rather than quoting an actual span). A cheaper/faster model is fine for the locate-and-propose ACT step and for running the mechanical schema validator (which should be an actual validator — JSON Schema / pydantic / equivalent — not an LLM self-check).

```text
GOAL (frozen): Extract every field defined in <SCHEMA> (field name, type/format/enum constraint, required-or-optional) from <DOCUMENT / DOCUMENT SET> into one structured record. Freeze the field list from <SCHEMA> at turn 0 — no field may be added, removed, renamed, or have its type loosened once turn 1 begins; schema problems are a BLOCKED condition for a human to fix between runs, not something this loop edits mid-run. A field counts as FILLED only when both hold, confirmed by checks independent of the extraction step that proposed the value: (1) the value passes schema validation for that field's type/format/enum/required constraint, run through an actual validator, not model self-assessment; and (2) the value has an exact source citation — document-ID/page/section/table-cell + a literal quoted span — and a separate match check confirms that quoted span actually contains/entails the value (not merely nearby or topically related text). The task is complete only when every required field is FILLED-and-verified or explicitly NOT-PRESENT-IN-DOCUMENT after two independently-targeted searches, every optional field is similarly resolved or explicitly skipped, and the assembled record as a whole re-passes full schema validation in one final pass.

Carry forward this state across turns:
- field ledger: field name | type/format/enum constraint | status (UNFILLED / FILLED / CONTESTED / NOT-PRESENT) | proposed value | citation (doc-ID/page/section/cell + exact quoted span) | search locations already tried for this field | this field's consecutive-verification-fail streak (persists across turns even while ASSESS targets other fields in between; resets to 0 only when this specific field is FILLED or confirmed NOT-PRESENT)
- total turns used this run (every turn counts — locate turns and re-verification turns alike; this is what BUDGET's N measures)
- draft structured record (the JSON/object being assembled from the ledger's FILLED fields)
- schema-validator results log: field | pass/fail | validator error message (if any)

Each turn:
1. ASSESS — target the first field in schema order that is UNFILLED or CONTESTED. If none, target the first field that has exactly one failed search attempt (to run its second, differently-targeted search before it can be marked NOT-PRESENT). Otherwise, if all fields are resolved, run the final whole-record schema validation pass.
2. ACT (one reversible action):
   - field targeted: locate exactly ONE candidate value for that field in <DOCUMENT / DOCUMENT SET> and propose it with its citation (doc-ID/page/section/cell + exact quoted span containing it). Never repeat a prior search targeting the same section/synonym for the same field on retry — vary the section, table, page, or phrasing searched.
   - final-record targeted: re-run schema validation over the entire assembled record as one object, not field-by-field.
3. VERIFY (independent of the ACT step that produced the candidate):
   - schema validation: run the proposed value through an actual schema validator against <SCHEMA>'s constraint for that field (type, format, enum membership, numeric range, required-ness). This is a mechanical check, not an LLM opinion. Fail = validator rejects it.
   - citation-match: does the exact quoted span at the cited location literally contain or directly entail the proposed value — not a paraphrase, not an inference from surrounding context, not a different cell/row that merely looks related? Confirm the citation is specific enough to be independently re-located (doc-ID/page/section/cell). Fail = span doesn't support the value, or the citation is missing/too vague to check.
   - both checks must pass independently for the field to become FILLED. If either fails, the field stays UNFILLED/CONTESTED and the specific failure (validator error vs. citation mismatch) is logged.
   - final-record verification: does the whole assembled record validate against <SCHEMA> as a single object (e.g. cross-field constraints, required-field completeness)? If not, identify which field(s) broke it, reopen them to CONTESTED, and return the loop to step 1.
4. DECIDE:
   - field targeted: if both schema validation and citation-match pass, mark FILLED, record the value+citation in the draft record, and reset this field's fail-streak to 0. If either fails, increment this field's fail-streak by 1 and log which check failed and why.
   - field on its second failed search: if the second, differently-targeted search also fails to locate any candidate, mark NOT-PRESENT and log both searched locations. If it succeeds, treat the new candidate through normal schema+citation verification.
   - final-record targeted: if it passes, this is SUCCESS. If not, reopen the specific failing field(s) as above.
   - always increment total-turns-used by 1, regardless of which branch ran.

STOP on the first of these that trips, in order:
- SUCCESS: every required field FILLED-and-verified (schema pass + citation-match pass) or explicitly NOT-PRESENT after two independently-targeted searches; every optional field resolved or explicitly skipped; the assembled record passes whole-record schema validation.
- BUDGET: total-turns-used reaches N, where N = 2 x (number of schema fields) + 3, minimum 8 (size N to the actual field count when instantiating this prompt — do not leave it unsized).
- NO-PROGRESS: any single field's consecutive-verification-fail streak reaches 2 (a new candidate keeps being proposed, from different search locations, but keeps failing schema validation or citation-match). Halt and report the exact field, its last proposed value, its last citation, and which check failed. Do not loosen the schema or accept an uncited value to work around this — that is a human decision made between runs, not a mid-loop escape hatch.
- BLOCKED: (a) a required field is confirmed genuinely absent from the document after two independently-targeted searches and the schema supplies no default — needs a human to waive the requirement or supply the value externally; or (b) <SCHEMA> itself is ambiguous or self-contradictory for a field (conflicting constraints, an enum that doesn't cover an observed real-world value) — needs a human to fix the schema. Halt and surface the exact field and why.

Never mark a field FILLED on the extraction step's own confidence — schema validation and citation-match must both be separate checks, run after the value is proposed, not folded into the same step that proposed it. Never accept a citation that is a paraphrase, a nearby cell, or "close enough" — the quoted span must literally contain the value. Never silently reformat/coerce a value to satisfy the schema (e.g. rewriting a date's format, rounding a number) without re-running the citation-match check against the coerced form — a coercion that no longer matches its own citation is a new candidate, not a confirmed one. Never let a CONTESTED field resolve by picking between two conflicting candidate values without a fresh, targeted re-check — log both candidates and let VERIFY adjudicate.

Report: final structured record (JSON) with every field's value, per-field final status (FILLED / NOT-PRESENT / left CONTESTED if halted), and per-field citation (doc-ID/page/section/cell + quoted span); full field ledger; schema-validator log; which arm fired; and (if BLOCKED or NO-PROGRESS) the exact field, its last attempted value/citation, and why it stalled.
```

### 4. RAG Answer — Fully-cited answer over a document set

- **When:** Use when a question must be answered strictly from a fixed corpus/index, every surviving claim needs an independently-verified supporting quote, and partial/full refusal is an acceptable (not a failure) outcome when the corpus doesn't fully support an answer.
- **Loop:** ASSESS -> ACT -> VERIFY -> DECIDE
- **Stop:** SUCCESS: No claim remains UNSUPPORTED or CONTRADICTED (each resolved to SUPPORTED or DROPPED), no coverage gap remains unaddressed, refusal state is finalized (NONE/PARTIAL/FULL), and an independent coverage-match pass (run as a separate call from whichever step last touched the draft) confirms the answer text and the claim ledger agree exactly. · BUDGET: N = 3×C + 3, where C = the current count of rows in the claim ledger (claims are never removed from the ledger, only marked DROPPED, so C is monotonic non-decreasing). C starts at max(3, number of claims in the turn-0 draft) and increments by 1 each time DECIDE adds a new claim via the coverage-gap branch — so N is recomputed live off the ledger, not frozen to the turn-0 draft. No separate numeric floor is stated on N itself: C's own floor of 3 already guarantees N ≥ 12, so a second "minimum 10" would be dead code and is deliberately omitted. · NO-PROGRESS: Halts when the tuple (SUPPORTED set, UNSUPPORTED set, DROPPED set, refusal state, draft answer text) is identical across 2 consecutive turns — i.e. nothing resolved, nothing dropped, nothing added, AND refusal state didn't change, AND draft text didn't change. Tracking refusal state and draft text (not just the three ledger sets) is what lets the final reconciliation phase — which can legitimately spend several turns adjusting refusal-scope and draft text to match the ledger without touching any claim's status — register as real progress instead of tripping a premature halt. · BLOCKED: Halts when: the question is ambiguous in a way that changes which claims are relevant and needs human disambiguation before claims can be judged; two retrieved passages contradict each other on a load-bearing claim in a way no further retrieval can resolve; or the retrieval system itself errors or times out. The corpus simply lacking the answer is not BLOCKED — that resolves via SUCCESS as a refusal.
- **Model:** ACT (issuing a retrieval query, drafting a candidate claim, or adding a coverage-gap claim) may run on a cheaper/faster model — these are reversible, low-stakes actions. VERIFY (entailment check on a claim, coverage-gap check, or the final coverage-match/refusal-scope check) MUST run as a structurally separate call from the ACT step it's checking: a distinct agent invocation or a fresh context window that receives only the claim text, the retrieved passage(s), and (for coverage verification) the relevant slice of <QUESTION> — never ACT's retrieval reasoning, confidence language, or drafting rationale. Prefer a stronger model for VERIFY than for ACT, but the isolation is the binding requirement, not the model tier: a single continuous reasoning stream that drafts a claim and then "double-checks" itself in the same turn does not count as independent verification, no matter which model is running it or how the prompt phrases the self-check instruction.

```text
GOAL (frozen): Answer <QUESTION> using only evidence retrieved from <DOCUMENT CORPUS/INDEX>. At turn 0, produce an initial draft answer and decompose it into an ordered list of atomic claims — each claim a single checkable assertion. Freeze the question's scope at turn 0; claims may be added, resolved, or dropped as the loop runs, but the question being answered does not change mid-run. A claim counts as SUPPORTED only when a directly quoted passage from the corpus entails it (states the fact, not merely correlates with it), confirmed by a verification pass that is structurally independent of whichever step drafted or retrieved for the claim (see MODEL section — this independence is a binding execution requirement, not a stylistic preference). The task counts as done only when the final answer text contains exactly the SUPPORTED claims (no more, no less) — every UNSUPPORTED claim has been either resolved, dropped with the answer scoped down around it (partial refusal), or has caused a full refusal if dropping it leaves no answerable claims for the question asked — and an independent coverage-match pass confirms the answer text and the claim ledger agree exactly.

Carry forward this state across turns:
- claim ledger: claim # | claim text | status (UNSUPPORTED/SUPPORTED/CONTRADICTED/DROPPED) | supporting passage-ID(s)/quote (if SUPPORTED) | retrieval queries already tried for this claim | this claim's consecutive-verification-fail streak (resets to 0 only when the claim resolves to SUPPORTED or is DROPPED). Rows are never deleted, only re-statused — the row count is C, used by BUDGET below.
- coverage map: which parts of <QUESTION> are addressed by at least one SUPPORTED-or-still-pending claim, and which parts have no claim yet
- refusal state: NONE (full answer expected) / PARTIAL (list of dropped claims + the scoped-down answer that remains) / FULL (question cannot be answered from the corpus at all)
- total turns used this run (every turn counts — retrieval, claim-addition, coverage, and refusal-decision turns alike; this is what BUDGET's N measures)
- current draft answer text

Each turn:
1. ASSESS — if any claim is UNSUPPORTED or CONTRADICTED, target the first such claim in ledger order. Otherwise, if the coverage map shows a part of <QUESTION> with no claim addressing it, target that gap. Otherwise, target the final refusal-scope decision.
2. ACT (one reversible action; may run on the cheaper ACT model — see MODEL section):
   - claim targeted: issue exactly ONE retrieval query against the corpus for that claim. Never repeat a prior query string verbatim for the same claim; a retry must target a different phrasing/section/document than the last attempt on this claim.
   - coverage gap targeted: add exactly ONE new candidate claim to the ledger that addresses the gap, derived only from evidence already surfaced or a fresh targeted retrieval — never fabricated to "fill in" the answer. New claim starts UNSUPPORTED. (This increments C, and therefore BUDGET's N, by 1.)
   - refusal-scope targeted: no retrieval — recompute the draft answer from the ledger's SUPPORTED claims only, and classify refusal state as NONE/PARTIAL/FULL based on whether every part of the coverage map is satisfied, partly satisfied, or unsatisfied.
3. VERIFY (mandatory structural independence — a separate agent call/fresh context from step 2, receiving only the claim/passage/question-slice, never ACT's reasoning trace; see MODEL section):
   - claim verification: does the retrieved passage ENTAIL the claim (states the fact as written) or merely correlate with it (topical overlap, adjacent detail, no actual statement)? Confirm citeable provenance (document ID/section). Entailment resolves to SUPPORTED; correlation-only leaves it UNSUPPORTED. If a passage states something that conflicts with the claim, mark CONTRADICTED rather than SUPPORTED or UNSUPPORTED.
   - coverage verification: does the newly added claim actually correspond to a real, unaddressed part of <QUESTION> — not a restatement of an existing claim or an out-of-scope addition?
   - refusal-scope verification (runs once no claim is UNSUPPORTED/CONTRADICTED and no coverage gaps remain, and again any time the draft or refusal state changes): does the draft answer text contain every SUPPORTED claim and no others? Does the stated refusal scope (NONE/PARTIAL/FULL) actually match what the ledger supports — e.g. is it claiming NONE (full answer) when a claim was in fact dropped? This is the check that prevents silent overreach from surviving into the final answer.
4. DECIDE:
   - claim targeted: if newly SUPPORTED or a CONTRADICTED claim is now settled by new evidence, update the ledger and reset that claim's fail-streak to 0. Otherwise increment only that claim's fail-streak; if the streak reaches 2, mark the claim DROPPED (not a halt — this is the mechanism that produces partial refusal) and log why.
   - coverage gap targeted: add the new claim to the ledger as UNSUPPORTED if it passed coverage verification; otherwise discard it and log why (no ledger change, no C increment).
   - refusal-scope targeted: update refusal state (NONE/PARTIAL/FULL) and the draft answer text to match the ledger exactly.
   - always increment total-turns-used by 1, regardless of which branch ran.

STOP on the first of these that trips, in order:
- SUCCESS: no claim remains UNSUPPORTED or CONTRADICTED (each has resolved to SUPPORTED or DROPPED), no coverage gap remains unaddressed, refusal state is finalized (NONE/PARTIAL/FULL), and an independent coverage-match pass (a separate call from whatever step last touched the draft) confirms the answer text and ledger agree exactly.
- BUDGET: total-turns-used reaches N, where N = 3×C + 3 and C = the live count of rows in the claim ledger (never decreases; grows by 1 each time DECIDE adds a coverage-gap claim). C starts at max(3, number of claims in the turn-0 draft), so N starts at ≥12 and grows automatically as coverage-gap claims are discovered — recompute N against the current ledger each time you check this arm, don't freeze it to the turn-0 draft.
- NO-PROGRESS: the tuple (SUPPORTED set, UNSUPPORTED set, DROPPED set, refusal state, draft answer text) is unchanged across 2 consecutive turns — nothing resolved, nothing dropped, nothing added, refusal state unchanged, draft text unchanged. Halt and report the frozen ledger state. (Per-claim retries that individually fail twice are not this arm — DECIDE auto-drops that one claim, which changes the DROPPED set and counts as progress, and the loop continues; this arm is only for the whole loop stalling on every tracked dimension at once, including during final reconciliation where refusal-state/draft-text movement alone is enough to reset the counter.)
- BLOCKED: the question is ambiguous in a way that changes which claims are even relevant and needs human disambiguation before claims can be judged; two retrieved passages contradict each other on a load-bearing claim in a way no further retrieval can resolve; or the retrieval system itself errors or times out. Halt and surface the specific claim or ambiguity and why. The corpus simply lacking the answer is not BLOCKED — that resolves via SUCCESS as a refusal.

MODEL: ACT (retrieval-query issuing, claim drafting, coverage-gap claim addition) may run on a cheap/fast model — these are reversible, low-stakes actions. VERIFY (entailment check, coverage-gap check, refusal-scope/coverage-match check) MUST execute as a structurally separate call from the ACT step it checks — a distinct agent invocation or fresh context window, receiving only the claim text, the retrieved passage(s), and (for coverage checks) the relevant slice of <QUESTION>, never ACT's retrieval reasoning, confidence phrasing, or drafting rationale. Prefer a stronger model for VERIFY, but the isolation itself is the binding mechanism, not the model tier — a single continuous reasoning stream that drafts and then "double-checks" itself in the same turn does not satisfy this requirement regardless of instruction wording.

Never mark a claim SUPPORTED on the drafting or retrieving step's own say-so — verification must be a separate call checking entailment, not the retrieval step's confidence, and must not see ACT's reasoning trace (see MODEL section). Never run VERIFY in the same context/continuation as the ACT step it's checking, even under instructions to "be independent" — a shared reasoning stream lets "close enough" entailment slip through undetected. Never let a CONTRADICTED claim silently resolve by picking whichever passage came first — log both and let VERIFY adjudicate with a fresh, targeted query. Never let the final answer be accepted on the drafting step's self-assessment of "supported enough" — the coverage-match pass (that the answer text is exactly the SUPPORTED-claim set, and that the stated refusal scope isn't understating what got dropped) must pass independently before SUCCESS fires. Never let refusal-scope verification get relaxed under pressure to "just answer something" — a full or partial refusal is a correct, first-class outcome of this loop, not a fallback to be avoided.

Report: final answer (or refusal statement, full or partial), full claim ledger with citations for every SUPPORTED claim and reasons for every DROPPED claim, refusal state, which arm fired, and (if BLOCKED or NO-PROGRESS) exactly which claim or ambiguity stalled and why.

Placeholders to fill when instantiating: <QUESTION> = the user's question; <DOCUMENT CORPUS/INDEX> = the retrieval source (vector index, search API, or fixed document set) this loop is scoped to query.
```
