# Accessibility Audit

`accessibility-audit` — 2 loop prompts.

### 1. Single-Violation WCAG Remediation Verified by an Automated Scanner

- **When:** A rendered page or component must be driven toward zero machine-detectable WCAG violations, fixing exactly one flagged violation per turn and keeping the fix only if an automated accessibility scanner — re-run as an independent check — confirms the count dropped with no new violation introduced.
- **Loop:** assess the scanner's current violation list and pick the single highest-impact violation -> make exactly ONE reversible source/markup/style edit targeting only that violation -> re-run `<SCANNER>` over the freshly-rendered `<TARGET>` as the independent verifier -> commit if total violations strictly dropped AND no new violation appeared, else git-revert -> decide continue/stop
- **Stop:** SUCCESS: `<SCANNER>` reports zero violations under the frozen `<RULESET>` on a clean re-scan of `<TARGET>`, every fix committed and no rule suppressed · BUDGET: `<MAX_ITERATIONS>` fix+rescan turns or `<MAX_HOURS>` elapsed · NO-PROGRESS: violation count unchanged or oscillating for `<K>` consecutive turns despite materially different fix approaches · BLOCKED: the only remaining reductions require judgment automated scanning cannot make — a "needs-review"/incomplete result, a meaningful-alt-text or reading-order call, or a design-token/copy decision outside the fixer's authority — hand these to a human/AT user
- **Model:** A mid-tier/fast model runs this well — the scanner is deterministic ground truth, so the work is mechanical: read the violation list, apply one documented fix, re-scan, commit or revert. Escalate only for ambiguous ARIA-pattern fixes on custom widgets or deciding hide-vs-relabel. The verifier MUST be the independent scanner re-run, not the fixer's inspection, because the party who wrote the fix already believes it works — only a fresh parse of the rendered DOM under a frozen, unsuppressed ruleset can show the count actually dropped with no new violation sneaking in; a scanner whose rules the fixer is free to disable is no verifier at all.

```text
GOAL (frozen — do not redefine mid-loop)
Drive <TARGET> (the rendered page/component/route under audit, at its actual post-JavaScript DOM) to zero violations reported by <SCANNER> (an automated accessibility engine such as axe-core, Pa11y, or IBM Equal Access) under the frozen <RULESET> (the exact rule tags/standard, e.g. wcag2a + wcag2aa + wcag21aa + wcag22aa mapping to <WCAG_TARGET> = WCAG 2.2 Level AA), fixing exactly ONE violation per turn. Freeze before turn 1 and never mutate mid-loop: <TARGET> and the states it must be scanned in (default view PLUS each distinct interactive state — menu open, error shown, modal up — since a scanner only sees the DOM currently rendered), <SCANNER> and its version, <RULESET>, and the baseline violation count <BASELINE_COUNT> from the turn-0 scan. "Done" means the same scanner, same ruleset, same rendered states report zero on turn 1 and turn 20 — never narrow the ruleset, ignore a rule, or drop a state to cross the line. Honest scope: passing this scanner is a FLOOR, not proof of accessibility (see INDEPENDENT VERIFIER) — report <TARGET> only as "no machine-detectable violations remain," never as "accessible."

INDEPENDENT VERIFIER
<SCANNER> is the verifier: it parses the ACTUAL rendered DOM of <TARGET> and applies deterministic, published WCAG rule checks (missing/empty accessible names, insufficient text contrast on solid backgrounds, invalid or conflicting ARIA, unlabeled form controls, duplicate IDs, missing lang, etc.), emitting {rule, impact, node, WCAG-criterion}. It is independent of the fix because the fixer edits source/markup/CSS while the scanner re-parses a freshly rendered DOM and never reads the diff, the commit message, or the author's rationale — the mechanism that judges "resolved" differs from the one that changed the code. It cannot rubber-stamp itself ONLY as long as its ruleset stays frozen and unsuppressed: adding an ignore selector, disabling a rule, or narrowing <RULESET> makes the count drop without the defect being fixed — that is Goodharting the verifier and is banned. Critically, an automated scanner detects only the programmatically-detectable subset of WCAG — commonly a third to a half of the success criteria. It CANNOT judge whether alt text is meaningful vs. "IMG_042.png", whether focus/reading order is logical, whether headings form a sensible outline, whether an accessible name actually describes the control's purpose, or contrast over gradients/images/text-in-images. Zero scanner violations therefore means "no known machine-detectable defect," never "usable by disabled people" — the operability/AT-experience checks and real AT-user testing are the ceiling this loop does not reach.

PER-TURN SHAPE
1. ASSESS — read the current scan's violation list; pick the single highest-impact unresolved violation (critical/serious before moderate/minor; a violation on a flow-blocking control before a cosmetic one). Do not target a "needs-review"/incomplete result — the scanner did not decide it, so neither can this loop.
2. ONE ACTION — make exactly one reversible edit addressing only that violation (add a real label/alt, fix a contrast value, correct one ARIA attribute/role, remove a duplicate id). Never batch fixes; never resolve a violation by hiding a genuine interactive control from the scanner (aria-hidden/display:none on real content) unless that is genuinely correct and introduces no new violation.
3. VERIFY — re-run <SCANNER> over <TARGET> in ALL frozen states, not just the node you touched. Record total count and the set of violation instances, and compare against the previous scan.
4. DECIDE — commit the edit only if total violations strictly DROPPED and NO new violation (a rule/node absent from the prior scan) appeared; otherwise git-revert immediately and mark that approach tried-and-failed for this violation. A fix that clears one rule but trips another (darkened text now failing contrast elsewhere, an added role invalidating ARIA) is net-zero-or-worse and must be reverted. End every turn on a committed, cleanly-scanning-or-baseline workspace.

CARRY-FORWARD STATE (compact)
Goal + <WCAG_TARGET>/<RULESET>/<SCANNER> version (all frozen); baseline vs current violation count and the per-turn trend over the last 3 turns; violations resolved (committed) vs remaining by impact; fix approaches already tried-and-reverted per violation (don't repeat one, no A->B->A); items handed to human/AT review; budget remaining (turns, hours).

ACTION BAN
Never suppress, ignore-list, or disable a scanner rule, add an axe-ignore/data-* bypass, or narrow <RULESET> to lower the count — the frozen ruleset is the verifier; lowering the bar is Goodhart, not a fix. Never mark a violation resolved from your own reading of the markup — only the scanner's re-run count decides. Never batch two fixes in one turn (you lose attribution of which one regressed). Never resolve a violation by hiding a real interactive control from assistive tech or the scanner. Never edit or re-take the baseline scan, or re-run in fewer states, to manufacture a drop. Never report <TARGET> as "accessible" from a clean scan — only "no machine-detectable violations remain"; meaningful-alt, reading-order, and name-quality judgments are out of scope here.

STOP — halt on the FIRST of:
SUCCESS (<SCANNER> reports zero violations under the frozen <RULESET> across every frozen state of <TARGET> on a clean re-scan, all fixes committed, no rule suppressed) | BUDGET (<MAX_ITERATIONS> fix+rescan turns or <MAX_HOURS> elapsed) | NO-PROGRESS (violation count unchanged or oscillating for <K> consecutive turns despite materially different fix approaches — stop and escalate rather than keep tweaking) | BLOCKED (the remaining items are ones automated scanning fundamentally cannot confirm — a scanner "incomplete"/needs-review result, a meaningful-alt-text or logical-reading-order judgment, or a fix needing a design-system color-token/copy decision outside your authority; or <TARGET> can't be rendered for the scanner to parse — surface to a human/AT user; never suppress a rule as a stand-in for a fix)
```

### 2. Keyboard + AT Operability of a Flow Verified by a Scripted Focus/Accessible-Name Harness

- **When:** A user flow must be proven fully operable by keyboard (logical focus order, visible focus, no traps) and to expose the correct accessible name/role/state to assistive tech, fixing exactly one issue per turn and confirming each fix with a scripted keyboard/AT harness rather than the author's own tabbing-through.
- **Loop:** assess the harness's failing assertions and pick the single highest-severity one -> make exactly ONE reversible fix targeting only that issue -> re-run the WHOLE flow through `<AT_HARNESS>` (drives real keystrokes + reads the accessibility tree) as the independent verifier -> commit if the targeted assertion passes AND no previously-passing assertion regressed, else git-revert -> decide continue/stop
- **Stop:** SUCCESS: `<AT_HARNESS>` reports every assertion in `<EXPECTED>` passing across the full `<FLOW>` — focus order, visible focus, trap-free, and correct name/role/state — on a clean end-to-end run · BUDGET: `<MAX_ITERATIONS>` fix+replay turns or `<MAX_HOURS>` elapsed · NO-PROGRESS: failing-assertion count unchanged or oscillating for `<K>` consecutive turns despite materially different fixes · BLOCKED: a step has no keyboard path or the harness can't drive it (native dialog, CAPTCHA, drag-only widget), or the sole open question is whether the announced SR experience actually makes sense — hand to a human/AT user
- **Model:** A mid-tier model runs the mechanical fix+replay loop well once `<AT_HARNESS>` and `<EXPECTED>` exist; escalate to a stronger reasoning tier for ambiguous ARIA-pattern decisions on custom composite widgets (comboboxes, tree grids, tablists) where the correct name/role/state contract is non-obvious. The verifier MUST be the scripted harness driving real keystrokes and reading the accessibility tree, not the author's tab-through, because whoever wrote the markup already "knows" it works and will tab through confirming their own intent — only an external run that records where focus actually lands and what name/role/state the a11y tree actually computes can catch a trap, a stale aria-expanded, or an empty accessible name that inspection glosses over; and even that harness only certifies the programmatic contract, so genuinely experiential SR questions are escalated to a human, never self-graded.

```text
GOAL (frozen — do not redefine mid-loop)
Prove that <FLOW> (the ordered user flow under test — e.g. open menu -> pick item -> fill+submit form -> reach confirmation, each step with its DOM) is (a) fully operable by keyboard alone and (b) exposes the correct programmatic name/role/state to assistive tech, verified end-to-end by <AT_HARNESS>. "Operable + correctly exposed" means, per <EXPECTED> (the frozen spec): every interactive element is reachable and actuatable by keyboard in a logical focus order matching visual/reading order; every focused element shows a visible focus indicator; no keyboard trap (focus can enter AND leave every composite/menu/modal, Escape closes dismissible layers and returns focus to the trigger); and every interactive element has a non-empty accessible name, the correct role, and correct state that updates on interaction (aria-expanded, aria-selected, aria-checked, aria-current, disabled, invalid). Freeze before turn 1 and never mutate: <FLOW> and its steps, <EXPECTED> (intended focus order + per-element name/role/state and state transitions), and <AT_HARNESS> with its browser/driver versions. "Done" means the same harness asserts the same expected contract on turn 1 and turn 20 — never loosen <EXPECTED> or skip a step to pass. Honest scope: a green harness proves the machine-readable contract, NOT that a real screen-reader user has a sensible experience (see INDEPENDENT VERIFIER) — report the flow as "keyboard-operable with correct programmatic name/role/state," not "screen-reader accessible."

INDEPENDENT VERIFIER
<AT_HARNESS> is the verifier: a scripted harness (e.g. Playwright/WebDriver) that drives REAL keyboard events through a real browser — Tab, Shift+Tab, Enter, Space, Escape, arrow keys — records the actual focus sequence (which element receives focus after each keystroke), and queries the platform accessibility tree for each element's computed accessible name, role, and state (the same programmatic surface a screen reader consumes), then asserts the recorded sequence and computed name/role/state against <EXPECTED>, emitting {step, element, assertion, expected, observed}. It is independent of the fix because it observes runtime behavior — where focus actually lands, what the a11y tree actually computes — not the author's markup or intent: the fixer edits code, the harness reads the live accessibility tree and focus log. It cannot rubber-stamp itself because every pass/fail is a mechanical comparison of keystroke-driven focus + computed AOM values against a frozen expected spec: "I added aria-expanded" never counts — only the harness reading state=expanded off the tree after actuating the control counts, and a positive tabindex that fakes order or an aria-label that is merely non-empty will still fail its order/role assertion. Honest limit: the harness proves reachability, trap-freedom, and correct PROGRAMMATIC name/role/state — it does NOT prove real-AT usability: whether NVDA/JAWS/VoiceOver announce in a sensible order and verbosity, whether a live region actually fires on a given SR+browser pair, or whether a non-empty accessible name is meaningful to a human rather than a lie. Passing is necessary, not sufficient; a real AT user on real SR/browser combos is the ceiling this loop does not reach, and any such question is a BLOCKED escalation, not a loop pass.

PER-TURN SHAPE
1. ASSESS — read the harness's failing assertions; pick the single highest-severity one in this order: keyboard trap or unreachable/unactuatable control > missing/wrong accessible name or role > wrong/stale state > focus-order or visible-focus defect. Do not chase a cosmetic focus-ring issue while a trap blocks the flow.
2. ONE ACTION — make exactly one reversible fix targeting only that assertion: repair one label/name, correct one role, wire one state attribute to the actual UI state, add Escape-to-close + focus-return for one layer, fix one element's focus order (via DOM order, not a positive tabindex), or add a visible :focus-visible style. Never batch fixes.
3. VERIFY — re-run the ENTIRE <FLOW> through <AT_HARNESS> (not just the step you touched), since focus order and traps are sequence-dependent. Record which assertions pass/fail.
4. DECIDE — commit only if the targeted assertion now passes AND no previously-passing assertion regressed (a common trap: a tabindex "fix" that reorders focus elsewhere, or an added role that drops the accessible name). Otherwise git-revert immediately and mark that approach tried-and-failed for this issue; on a 2nd distinct failed repair of the same issue, park it and move to the next. End every turn on a committed state whose harness result is no worse than the last commit.

CARRY-FORWARD STATE (compact)
Goal + frozen <EXPECTED>/<AT_HARNESS> versions; passing vs failing assertion counts and the trend over the last 3 turns; issues fixed (committed) vs remaining by severity; fix approaches already tried-and-reverted per issue (no repeats, no A->B->A); parked issues and items handed to human/AT review; budget remaining (turns, hours).

ACTION BAN
Never assert operability from your own inspection or a one-off manual tab-through — only the harness's recorded focus sequence and computed name/role/state count. Never use a positive tabindex to force focus order (it desynchronizes from DOM/reading order and breaks on the next change) — fix order via DOM structure. Never satisfy a name assertion with a misleading or duplicative aria-label just to make it non-empty; a lying name is worse than a missing one, and the harness checks presence/role, not meaning — that meaning is a human's call. Never suppress or skip a failing assertion, or loosen <EXPECTED>, to pass. Never batch fixes (focus/trap regressions become unattributable). Never "fix" a missing visible focus by disabling focus management such that focus is lost or sent to <body>. Never report the flow as "screen-reader accessible" from a green harness — only "keyboard-operable with correct programmatic name/role/state."

STOP — halt on the FIRST of:
SUCCESS (<AT_HARNESS> reports every assertion in <EXPECTED> passing across the full <FLOW> — logical focus order, visible focus, trap-free, correct name/role/state and state transitions — on a clean end-to-end run, all fixes committed) | BUDGET (<MAX_ITERATIONS> fix+replay turns or <MAX_HOURS> elapsed) | NO-PROGRESS (failing-assertion count unchanged or oscillating for <K> consecutive turns despite materially different fixes — halt and escalate rather than keep tweaking) | BLOCKED (a flow step has no keyboard-operable path or the harness cannot drive it — native OS dialog, CAPTCHA, third-party iframe, drag-only/canvas widget needing a design decision outside your authority — or the sole remaining question is whether the announced screen-reader experience actually makes sense to a user, i.e. announcement order/verbosity/live-region timing on real SR+browser, which the harness cannot adjudicate — surface to a human/AT user; never loosen an assertion as a stand-in for a fix)
```
