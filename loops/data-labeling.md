# Data Labeling

`data-labeling` — 3 loop prompts.

### 1. Low-Agreement Label Resolution via Frozen Adjudication Ruleset

- **When:** Annotator agreement on a batch of labeled items falls below threshold (e.g. Krippendorff's alpha, pairwise %, or a fixed vote-margin rule) and the disagreements need resolving — but resolution must come from a deterministic rule/decision tree plus a gold-labeled reference set, not from having the same disagreeing labelers re-vote or "talk it out."
- **Loop:** assess the largest same-root-cause cluster in the unresolved disagreement queue -> take ONE reversible action (edit exactly one clause in the adjudication ruleset, OR route exactly one cluster to a designated third-party senior adjudicator — never back to the original disagreeing labelers) -> verify by mechanically re-running the updated ruleset against the frozen gold set (script/rule engine, not model or labeler judgment) -> commit ruleset version + newly-resolved items if gold accuracy holds and unresolved count dropped, else revert -> decide
- **Stop:** SUCCESS: unresolved-item count in the disagreement queue = 0 AND a fresh full run of the current adjudication ruleset against the frozen gold set scores >= the gold-accuracy target (no cherry-picked partial run counts) · BUDGET: <MAX_TURNS> turns or <MAX_HOURS> wall-clock elapsed · NO-PROGRESS: unresolved-item count is unchanged for <K> consecutive turns despite trying different root-cause levers (ruleset edit vs. senior-adjudicator escalation) — force a lever-category switch or escalate rather than keep tweaking the same clause · BLOCKED: the frozen gold set itself contains two cases with the same root cause adjudicated inconsistently (policy contradiction), or the designated senior adjudicator resource is unavailable — surface to a human; never fall back to majority vote of the same disagreeing labeler pool as a substitute verifier
- **Model:** The verifier (ruleset-vs-gold-set check) should run as a deterministic script/rule engine wherever the ruleset is expressible as rules — trust doesn't scale with model tier there. Where the senior adjudicator's ruling itself must be elicited via a model call (e.g. simulating/drafting a proposed ruling for a human to approve), use a distinct, ideally stronger model or a fresh context with no memory of which labelers disagreed, so it isn't anchored on either original label. Never let the model that drafted a ruleset edit also grade its own edit against gold — the gold check must be a separate, fixed pass.

```text
Goal: on the frozen low-agreement queue [<DISAGREEMENT_QUEUE> — items where annotator agreement < <AGREEMENT_THRESHOLD> per <AGREEMENT_METRIC>, e.g. Krippendorff's alpha or vote-margin], drive the unresolved-item count to 0, where EVERY resolution is produced by applying the current version of <ADJUDICATION_RULESET> (a versioned, deterministic tie-break/decision-tree document) — never by asking the original disagreeing labelers to re-vote or reach informal consensus. The ruleset must score >= <GOLD_ACCURACY_TARGET> (e.g. 100%) against the frozen <GOLD_SET> (<N> already-adjudicated cases with documented rationale, covering the known hard root-causes) before any queue resolution it produces counts as final. Freeze the queue, the gold set, and the agreement metric before turn 1 — no new items enter either set mid-loop.

Per turn:
1. Assess: pull the current unresolved count and identify the single largest cluster of disagreement items sharing one root cause (e.g. same category pair confused, same rubric clause read two ways, same edge-case type).
2. Act — ONE reversible change only:
   (a) add or edit exactly one clause/branch in <ADJUDICATION_RULESET> to cover that root cause, or
   (b) route that one cluster to <SENIOR_ADJUDICATOR> (a designated third party distinct from the original disagreeing labelers) for a binding ruling, then encode that ruling back into the ruleset as a new rule or worked example.
   Never bundle a ruleset edit and an escalation in the same turn.
3. Verify independently: re-run the updated ruleset mechanically (script/rule engine) against the frozen <GOLD_SET> — this is the verifier, and it is separate from whoever proposed the edit. Accept the change only if gold accuracy stays >= <GOLD_ACCURACY_TARGET>. If it holds, re-apply the ruleset across the full disagreement queue and record how many items now resolve unambiguously (hit a defined rule branch) vs. still fall through.
4. Decide: commit the new ruleset version + the newly-resolved queue items if unresolved count dropped AND gold accuracy held; otherwise git-revert/roll back the ruleset edit and mark that lever as tried-and-failed for this root cause.

Carry forward each turn (compact, not the full transcript): goal + both thresholds, ruleset version number, unresolved-count trend over the last 3 turns, last gold-accuracy check result, root-cause clusters already addressed (don't re-target the same one), which lever type was last tried, budget remaining.

Ban: re-editing a rubric clause you already reverted for the same root cause (no A -> B -> A oscillation — if a clause change already failed the gold check once, the next attempt on that root cause must be the senior-adjudicator escalation, not a re-wording of the same clause); treating "labelers re-voted and now agree" as resolution under any circumstance — only ruleset-driven or senior-adjudicator-driven resolutions that pass the gold check count; applying a ruleset change to the queue before it has passed the gold-set check.

Stop on whichever trips first:
- SUCCESS — unresolved count = 0 AND a fresh full run of the ruleset against the gold set scores >= <GOLD_ACCURACY_TARGET>.
- BUDGET — <MAX_TURNS> turns or <MAX_HOURS> elapsed.
- NO-PROGRESS — unresolved count unchanged for <K> consecutive turns despite alternating lever types (ruleset edit vs. escalation) — then halt ruleset tweaking and escalate the whole stalled cluster to a human policy owner rather than keep iterating.
- BLOCKED — the gold set contains contradictory adjudications for the same root cause, or <SENIOR_ADJUDICATOR> is unavailable — surface to a human; do not substitute a vote among the same disagreeing labelers as a stand-in verifier.
```

### 2. Annotation-Guideline Refinement to Fresh-Labeler Agreement Target

- **When:** A labeling guideline produces high agreement only among the labelers who helped write it, and you need a brand-new labeler — reading the document and nothing else — to reach a target inter-annotator agreement on a frozen gold set; you refine the *guideline*, never coach the labelers, until a held-out fresh labeler passes cold.
- **Loop:** assess the dominant residual disagreement between the last fresh labeler's labels and gold -> make ONE reversible edit to the guideline (one clause / decision rule / worked example) -> hand the revised guideline to the NEXT uncontaminated fresh labeler and mechanically score their gold-set agreement -> commit the edit if agreement rose with no new regressions, else revert -> decide
- **Stop:** SUCCESS: a fresh, never-before-used labeler scores >= `<AGREEMENT_TARGET>` on the full frozen gold set on a clean run · BUDGET: `<MAX_TURNS>` turns / `<MAX_FRESH_LABELERS>` labelers / `<MAX_HOURS>` elapsed · NO-PROGRESS: agreement flat within `<NOISE_BAND>` for `<K>` turns despite retargeting which confusion/clause is addressed · BLOCKED: fresh-labeler pool exhausted, gold set is internally inconsistent, or a required fix can't be written as a general rule without leaking gold answers
- **Model:** The actor that edits the guideline can run on a cheap/fast tier — it does not grade itself. The fresh labeler IS the verifier and must be independent: a real held-out human, or if simulated, a distinct model/family in a fresh context with zero memory of the edits and no sight of the intended gold answers. The guideline author labeling the gold set themselves is self-grading and forbidden — they know the answers, so agreement would be trivially high and measure nothing. Escalate to a stronger labeler model only to confirm a borderline SUCCESS. What is being measured is the *transmissibility of the written document to someone who did not write it*, not whether the author can apply their own intent.

```text
GOAL (frozen — do not redefine mid-loop)
Refine <GUIDELINE_DOC> (the written labeling instructions) until a FRESH labeler — one who took no part in writing or revising it, sees only the document, and has never seen the gold labels — reaches inter-annotator agreement >= <AGREEMENT_TARGET> against <GOLD_SET> under the frozen metric <AGREEMENT_METRIC> (e.g. Cohen's/Fleiss' kappa, Krippendorff's alpha, or pairwise-% vs gold — pick ONE and freeze it). Freeze before turn 1 and never mutate mid-loop: the gold set (<N> items with a single adjudicated correct label each, spanning the known hard confusions), the agreement metric and its pass threshold, the label taxonomy, and the roster/source of fresh labelers. "Done" means the same cold-read pass on turn 1 and turn 20 — do not lower the target or swap the metric to declare victory.

INDEPENDENT VERIFIER
The verifier is a fresh, uncontaminated labeler applying the current guideline to the frozen gold set, with their agreement computed MECHANICALLY by a fixed scoring script (metric + threshold fixed at turn 0). It is independent because: (a) the labeler is NOT the guideline author and never sees the gold answers or prior versions — so the score reflects what the document alone transmits, not shared context; (b) the score is arithmetic over labels vs gold, not anyone's judgment that the guideline "reads clearly." A guideline author labeling the gold set, or reusing a labeler who already saw an earlier version, cannot rubber-stamp the doc — both are banned because a contaminated labeler's agreement measures memory/intent, not the writing. Each guideline revision that will be scored for a pass gets a labeler who has never touched any prior version.

PER-TURN SHAPE
1. ASSESS — from the last fresh labeler's confusion matrix vs gold, pick the single largest same-root-cause error cluster (one category pair confused, one boundary case, one under-specified rule). Do not chase scattered one-off misses.
2. ONE ACTION — make exactly one reversible edit to <GUIDELINE_DOC>: add/reword one decision rule, add one worked positive+negative example for that confusion, or tighten one definition. Never bundle two edits; never edit the gold set or the metric.
3. VERIFY — run the independent verifier: give the revised guideline to the NEXT fresh labeler (never one who saw a prior version), collect their gold-set labels blind, and compute <AGREEMENT_METRIC> mechanically. Record per-category deltas, not just the aggregate, to catch a fix that helped one confusion while regressing another.
4. DECIDE — commit the edit (version the doc) if aggregate agreement rose AND no category regressed past its prior level; otherwise revert the edit to the last committed version and mark that clause+root-cause as tried-and-failed. End every turn on a committed, self-consistent guideline version.

CARRY-FORWARD STATE (compact)
Goal + frozen metric/threshold; current guideline version number; agreement trend over last 3 turns; last fresh-labeler's per-category confusion vs gold; root-cause clusters already addressed (don't re-target the same one two ways in a row); which fresh labelers are already spent (contaminated) vs still available; budget remaining (turns, labelers, hours).

ACTION BAN
Never let the guideline author (or any labeler who shaped/revised it, or one who has seen the gold answers) act as the verifier — that is self-grading. Never reuse a labeler across versions being scored — a second read is contaminated. Never paste gold items or their answers verbatim into the guideline (teaching to the test / Goodharting the gold set) — fixes must be general rules or fresh illustrative examples, not the graded items. Never coach the labeler outside the document, answer their questions, or "clarify verbally" — only the written doc may change. Never re-apply an edit already reverted for the same root cause (no A->B->A oscillation); the next attempt on that confusion must be a different lever. Never lower <AGREEMENT_TARGET>, swap <AGREEMENT_METRIC>, or drop a hard gold item to cross the line.

STOP — halt on the FIRST of:
SUCCESS (a fresh, never-used labeler scores >= <AGREEMENT_TARGET> on the full frozen gold set on a clean cold read) | BUDGET (<MAX_TURNS> turns, <MAX_FRESH_LABELERS> fresh labelers, or <MAX_HOURS> elapsed) | NO-PROGRESS (agreement flat within <NOISE_BAND> for <K> consecutive turns despite retargeting the confusion/clause — escalate the guideline to a human editor rather than keep tweaking) | BLOCKED (fresh-labeler pool exhausted so no uncontaminated verifier remains, the gold set contains two same-root-cause items labeled inconsistently, or the residual confusion cannot be expressed as a general rule without leaking gold answers — surface to a human; never substitute a contaminated labeler or the author as a stand-in verifier)
```

### 3. Active-Learning Batch Selection Verified by Held-Out Rare-Class Lift

- **When:** Under heavy class imbalance you must choose the next batch of unlabeled items to send for labeling to most improve a downstream model on the rare class, and you want each batch's value proven by an actual retrain-and-measure on a frozen held-out set — not by trusting the uncertainty/diversity acquisition score that selected it.
- **Loop:** assess where the current model is weakest on the rare class from held-out error analysis -> select ONE batch of `<BATCH_SIZE>` items via ONE sampling strategy and get them labeled -> retrain and measure the rare-class metric on the frozen held-out set -> keep the batch in the train pool if held-out lift clears the noise band, else drop it and mark that strategy tried -> decide
- **Stop:** SUCCESS: frozen held-out rare-class `<METRIC>` >= `<TARGET>` on a clean retrain · BUDGET: `<MAX_BATCHES>` batches / `<LABEL_BUDGET>` labels / `<MAX_HOURS>` elapsed · NO-PROGRESS: held-out metric flat within `<NOISE_BAND>` for `<K>` batches despite switching sampling strategy · BLOCKED: unlabeled pool has no rare-class candidates left, held-out variance exceeds the target lift so a real gain can't be detected, or the labeling oracle is unavailable
- **Model:** Candidate scoring/sampling can run on a cheap model — it only proposes what to label. The verifier is a MECHANICAL retrain + eval on a fixed held-out split at a fixed seed (or averaged over fixed seeds), and it must never be the acquisition score itself. Beware Goodhart: high average uncertainty or diversity of the selected batch is the quantity you optimized, so it cannot also serve as proof of improvement — only measured held-out lift counts. Reserve the production-tier model for the final SUCCESS gate; if the same strategy stalls, switching the sampling model rarely helps — switch the strategy or the slice.

```text
GOAL (frozen — do not redefine mid-loop)
Raise the downstream model's rare-class performance to <METRIC> >= <TARGET> on the frozen held-out set <HELDOUT_SET>, by iteratively selecting and labeling batches from the unlabeled pool <UNLABELED_POOL>. <METRIC> is a rare-class-sensitive measure (e.g. rare-class recall, F1, or PR-AUC — NOT overall accuracy, which imbalance renders misleading) chosen and frozen at turn 0. Freeze before turn 1 and never mutate: the held-out set (never sampled from, never labeled by this loop, never added to train), the metric and target, the model architecture + training recipe + random seed(s), the per-batch label budget <BATCH_SIZE>, and the train/eval split boundary. "Done" means the same held-out number clears the same bar on turn 1 and turn 20.

INDEPENDENT VERIFIER
The verifier is a retrain of the model on (current train set + the newly-labeled batch) followed by scoring <METRIC> on the frozen <HELDOUT_SET>, run by a fixed harness at fixed seed(s). It is independent of the selection step because the held-out set is disjoint from both the unlabeled pool and the labeled train pool, is never selected into, and is never touched by the sampling logic — so it cannot reflect whatever the acquisition function preferred. It cannot rubber-stamp itself: the acquisition score (uncertainty, margin, diversity, predicted rare-class probability) is the signal being optimized, and using it to declare success would be Goodhart — the batch can look maximally "informative" while the model does not actually improve, or while the batch is full of near-duplicate false positives. Only the held-out retrain lift, exceeding the measured <NOISE_BAND>, counts as improvement. (A meaningful <NOISE_BAND> requires the harness to run multiple fixed seeds — a single-seed setup cannot estimate it.)

PER-TURN SHAPE
1. ASSESS — from the current model's held-out errors, identify the single rare-class weakness driving the gap (a confusable sub-region, false-negative cluster, or under-represented feature slice). Note the current held-out <METRIC> and the retrain noise band.
2. ONE ACTION — select exactly ONE batch of <BATCH_SIZE> unlabeled items by ONE named sampling strategy targeting that weakness (e.g. uncertainty near the decision boundary, diversity/coverage to avoid near-duplicates, or predicted-rare-class density) and send them to the labeling oracle. One strategy, one batch per turn — never blend strategies or oversize the batch to smuggle in more labels.
3. VERIFY — run the independent verifier: retrain on train+batch at the fixed seed(s) and score <METRIC> on the frozen held-out set. Compare the lift against <NOISE_BAND> (estimated from seed-to-seed variance), not against zero — a sub-noise "gain" is not a gain.
4. DECIDE — commit the batch into the permanent train pool if held-out lift > <NOISE_BAND> with no rare-class regression; otherwise remove the batch from the train pool (keep its labels archived, don't re-select them) and mark that strategy+slice as tried-and-flat. End every turn with a committed train pool and a recorded held-out number.

CARRY-FORWARD STATE (compact)
Goal + frozen metric/target/seed(s); current held-out <METRIC> and noise band; held-out lift per batch over last 3 batches; which sampling strategies × slices have been tried and their measured lift; current train-pool size and labels spent vs <LABEL_BUDGET>; rare-class candidates remaining in the pool; budget remaining (batches, labels, hours).

ACTION BAN
Never treat the acquisition/sampling score (average uncertainty, diversity, predicted rare-class rate) as the success signal — that is Goodharting the metric you optimized; only held-out retrain lift decides. Never touch the held-out set: don't sample from it, label it into train, tune the seed/recipe to flatter it, or peek at per-item held-out labels to steer selection. Never count a sub-noise-band bump as improvement, and never change the frozen recipe/seed and add a batch in the same turn (you'd lose attribution). Never re-select a batch already dropped for the same slice (no oscillation) — the next attempt must be a different strategy or a different weakness. Never dump all predicted-rare-class items in one batch without diversity — redundant near-duplicates and false positives burn label budget for no lift. Never lower <TARGET> or swap to overall accuracy to declare done.

STOP — halt on the FIRST of:
SUCCESS (frozen held-out <METRIC> >= <TARGET> on a clean retrain at the fixed seed(s)) | BUDGET (<MAX_BATCHES> batches, <LABEL_BUDGET> labels, or <MAX_HOURS> elapsed) | NO-PROGRESS (held-out <METRIC> flat within <NOISE_BAND> for <K> consecutive batches despite switching sampling strategy AND targeted slice — stop sampling and escalate; the gap may need more capacity, better features, or relabeling, not more data) | BLOCKED (the unlabeled pool has no rare-class candidates left, held-out variance exceeds the target lift so a genuine gain cannot be detected, or the labeling oracle is unavailable — surface to a human; never substitute the acquisition score for the held-out verifier)
```
