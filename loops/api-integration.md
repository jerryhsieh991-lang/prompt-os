# API Integration

`api-integration` — 1 loop prompts.

### 1. API Contract Verification Loop (v2 — fixed)

- **When:** Use when an agent must call an external/internal API endpoint (or a small set of related endpoints) and confirm the RESPONSE CONTRACT holds — not just that a request went through and the happy path looked plausible. Covers status codes, response schema (required fields, types, enums, nullability), and declared error/edge behavior (4xx validation errors, 401/403, 429 rate-limit, 5xx, empty/paginated results) as defined by an OpenAPI spec, JSON Schema, or equivalent contract document.
- **Loop:** assess which case in `<CASE_SET>` to attempt next (skip any case currently PARKED, prefer happy-path first, then listed order) -> issue exactly ONE API call for that case -> run the independent contract validator against the raw response (status + schema + headers) -> commit the case as passed, log the violation and repair, or PARK it after 2 distinct failed repairs so a stuck case can't stall the rest of the set -> decide continue/stop
- **Stop:** SUCCESS: SUCCESS — the independent validator reports zero contract violations across every case in `<CASE_SET>` (happy path AND every declared error/edge case, including any case whose declared expectation IS a 401/403 auth-failure), confirmed on the FULL case set, not just the case last touched. · BUDGET: BUDGET — `<MAX_ITERATIONS>` total call+verify turns used. · NO-PROGRESS: NO-PROGRESS — fires only when EVERY remaining unresolved case is PARKED (2 distinct repairs already tried and failed, nothing unparked left to attempt) and no case has newly reached "passed" for 3 consecutive turns; or the identical {case, fix} pair recurs; or a fix oscillates A->B->A without new rationale. A single hard case being parked never triggers this on its own — the loop must first have exhausted every other unresolved case (passed, blocked, or itself parked). · BLOCKED: BLOCKED — a call NOT targeting a declared auth-failure case unexpectedly returns 401/403 (the agent's own working credentials appear rejected — never retried, immediate halt); a case in `<CASE_SET>` cannot actually be triggered against the live API; the contract spec and observed live behavior conflict with no authority to resolve which is correct; or the next call is paid/quota-consuming/state-mutating with no cleared human-approval gate for THAT specific case. Carve-out: a 401/403 returned by a call that IS deliberately targeting the "401 missing/bad auth" or "403 forbidden" case in `<CASE_SET>` is NEVER a BLOCKED trigger — it is the expected condition and is scored PASS/FAIL by `<CONTRACT_VALIDATOR>` like any other case.
- **Model:** Sonnet 5 handles the mechanical loop (issue call, run validator, classify violation, apply a documented fix, park/unpark cases) fine once `<CONTRACT_SPEC>` and an independently-sourced `<CONTRACT_VALIDATOR>` exist. Escalate only to interpret an ambiguous spec-vs-observed-behavior conflict, to design the initial `<CASE_SET>`, or — if no pre-existing validator tool is available — to author the validator as a blind second pass with no visibility into the call-construction reasoning.

### API Contract Verification Loop

- **When:** Use when an agent must call an external/internal API endpoint (or a small set of related endpoints) and confirm the RESPONSE CONTRACT holds — not just that a request went through and the happy path looked plausible. Covers status codes, response schema (required fields, types, enums, nullability), and declared error/edge behavior (4xx validation errors, 401/403, 429 rate-limit, 5xx, empty/paginated results) as defined by an OpenAPI spec, JSON Schema, or equivalent contract document.

- **Loop:** assess which case in `<CASE_SET>` to attempt next (skip any case currently PARKED, prefer happy-path first, then listed order) -> issue exactly ONE API call for that case -> run the independent contract validator against the raw response (status + schema + headers) -> commit the case as passed, log the violation and repair, or PARK it after 2 distinct failed repairs so a stuck case can't stall the rest of the set -> decide continue/stop

- **Stop:** SUCCESS — the independent validator reports zero contract violations across every case in `<CASE_SET>` (happy path AND every declared error/edge case, including any case whose declared expectation IS a 401/403 auth-failure), confirmed on the FULL case set, not just the case last touched · BUDGET — `<MAX_ITERATIONS>` total call+verify turns used · NO-PROGRESS — every remaining unresolved case is PARKED (2 distinct repairs already tried and failed, nothing unparked left to attempt) and no case has newly reached "passed" for 3 consecutive turns, or the same {case, fix} pair recurs, or a fix oscillates A->B->A · BLOCKED — a call NOT targeting a declared auth-failure case unexpectedly returns 401/403 (broken credentials — never retried, immediate halt); a case in `<CASE_SET>` cannot actually be triggered against the live API (e.g. no way to force a 500 without server access); the contract spec and observed live behavior conflict with no authority to resolve which is correct; or the next call is paid/quota-consuming/state-mutating with no cleared human-approval gate for THAT specific case. **Carve-out:** a 401/403 returned by a call that IS deliberately targeting the "401 missing/bad auth" or "403 forbidden" case in `<CASE_SET>` is never a BLOCKED trigger — it is the expected condition for that case and is scored PASS/FAIL by `<CONTRACT_VALIDATOR>` exactly like any other case.

- **Model:** Sonnet 5 handles the mechanical loop (issue call, run validator, classify violation, apply a documented fix, park/unpark cases) fine once `<CONTRACT_SPEC>` and an independently-sourced `<CONTRACT_VALIDATOR>` exist. Escalate only to interpret an ambiguous spec-vs-observed-behavior conflict, to design the initial `<CASE_SET>` (which edge/error cases actually matter), or — if no pre-existing validator tool is available and one must be authored in-session — to author it as a blind second pass with no visibility into the call-construction reasoning, to preserve genuine independence.

``​`text
GOAL (frozen — do not redefine mid-loop)
Confirm that <ENDPOINT> (method + path, e.g. "POST /v1/orders") satisfies its response contract as defined in <CONTRACT_SPEC> (an OpenAPI/JSON-Schema doc or equivalent, frozen before turn 1 — do not edit it mid-run to make a failing case pass), for EVERY case in <CASE_SET>: the happy path PLUS every declared error/edge case that matters for this integration (e.g. 400 invalid-body, 401 missing/bad auth, 403 forbidden, 404 not-found, 409 conflict, 422 validation, 429 rate-limited, 500/503 server error, empty-result page, last-page pagination boundary, null-vs-omitted optional field). A response with status 200 and a body that "looks reasonable" is NEVER sufficient — every case is only marked done when the INDEPENDENT contract validator confirms it, on a run separate from the client code that issued the request. Note that "401 missing/bad auth" and "403 forbidden" being IN <CASE_SET> means those responses are EXPECTED, PASSING outcomes when deliberately probed for — they are validated like any other case, not treated as failures of the run itself (see BLOCKED carve-out below for the distinct, unrelated failure mode of the agent's own credentials being broken).

Freeze before turn 1:
- <ENDPOINT> and its required request parameters/auth for each case.
- <CONTRACT_SPEC> — the schema/status contract, versioned, not edited mid-run.
- <CASE_SET> — the explicit list of cases to verify (happy path + named error/edge cases, explicitly flagging which cases, if any, are themselves auth-failure probes e.g. "401 missing/bad auth", "403 forbidden"). If a case can't actually be triggered against the live API (no test double for a 500, no way to force rate-limiting), say so now and mark it BLOCKED-eligible rather than discovering that mid-run.
- <CONTRACT_VALIDATOR> — an independent tool/library, sourced so it cannot silently inherit the same misreading of <CONTRACT_SPEC> that shaped the calls. Prefer an externally-supplied, pre-existing validator (a JSON Schema validator run against the raw response body + status + headers, or a contract-testing tool like Dredd/Schemathesis/Pact) that was not authored from scratch in this session. If no such tool exists and one must be built in-session, it must be authored as a distinct sub-task/second pass with NO visibility into the reasoning that designed <CASE_SET> or the calls — reading only <CONTRACT_SPEC> itself — so a shared misreading of the spec can't produce a validator and a call set that agree with each other for the wrong reason. Either way, invocation of the validator is a separate step from the call itself; the calling code's own "it didn't throw" or "status was 2xx" is never the validator.
- <MAX_ITERATIONS>.
- Whether any case(s) in <CASE_SET> are paid/quota-consuming/state-mutating (e.g. hitting a metered endpoint, creating a real resource) — if so, name the human-approval gate for EACH such case now (there may be more than one).

INDEPENDENCE REQUIREMENT
The validator must be a program/library that parses <CONTRACT_SPEC> and checks the ACTUAL response object (status code, headers, full body) against it field-by-field — never the same reasoning pass that decided which call to make declaring the response fine by inspection. Genuine independence means the validator's authorship must not share the same interpretation of <CONTRACT_SPEC> that produced the calls: use a pre-existing external validator tool where possible; if one must be constructed mechanically in-session, build it in a separate sub-task that reads only <CONTRACT_SPEC> and has no access to the call-construction reasoning, before the first turn, rather than eyeballing responses turn by turn.

PER-TURN SHAPE
1. ASSESS — From <CASE_SET>, pick the next case to attempt: happy path first if unresolved, then declared error/edge cases in the order listed, SKIPPING any case currently marked PARKED. Only return to a PARKED case once every other unresolved case has itself reached passed, blocked, or parked (parking defers a stuck case so it can't stall ones that might pass easily — it never removes the case from the run). Confirm the chosen case isn't gated (paid/state-mutating) without a cleared human-approval gate for THIS specific case (human_gate_status is tracked per gated case, not globally).
2. ONE ACTION — Issue exactly ONE call to <ENDPOINT> that is expected to produce this case's condition (e.g. omit a required field to trigger 422, use an expired token to trigger 401, request a page past the last one). Never bundle multiple cases into one call. If this case's declared expectation IS an auth-failure status (401/403), a 401/403 response is the target outcome for this call, not a signal to halt.
3. VERIFY — Run <CONTRACT_VALIDATOR> against the raw response: status code matches what <CONTRACT_SPEC> declares for this case, body schema matches (required fields present, types/enums/nullability correct), and relevant headers (e.g. Content-Type, rate-limit headers) match. Record violations as {case, field/status, error_code, expected, got}.
4. DECIDE —
   - If validator reports zero violations for this case: mark it passed in case_status (un-parking it if it had been parked), move to the next unverified/unparked case.
   - If violations remain: classify (TRANSIENT — 5xx/timeout worth one backoff retry, the loop's only sanctioned identical-call repeat; CONTRACT_MISMATCH — spec and response genuinely disagree on a field/status/type; UNREACHABLE_CASE — this case can't actually be forced against the live API). For CONTRACT_MISMATCH, apply exactly one repair to how the case is invoked or, if the client code itself is the bug, patch that one thing — never edit <CONTRACT_SPEC> to match the wrong observed behavior. Log the fix in repair_log; never repeat an identical {case, fix} pair. If this is the case's 2nd distinct failed repair and it still doesn't validate, mark it PARKED(attempts=2, last_violation) rather than continuing to grind — move on to the next unresolved, unparked case in <CASE_SET>. A parked case remains eligible for a later turn once nothing unparked is left to attempt.
   - If a call NOT targeting a declared auth-failure case unexpectedly returns 401/403 (i.e. the agent's own credentials appear broken, unrelated to the case under test) → STOP: BLOCKED, immediately, never retried.
   - If a call DID target a declared 401/403 case, its response is never a BLOCKED trigger — it is scored via step 3 like any other case.
   - If a case is UNREACHABLE_CASE with no way to trigger it → STOP: BLOCKED, name the case and why.
   - If a case's status/schema conflict can't be resolved because it's unclear which of spec-vs-live is authoritative → STOP: BLOCKED, do not guess which one is "right."
   - If turn_count reaches <MAX_ITERATIONS> → STOP: BUDGET.
   - If every remaining unresolved case is PARKED (none left to attempt) and no case has newly reached "passed" for 3 consecutive turns, or the same {case, fix} pair recurs, or a fix oscillates A->B->A → STOP: NO-PROGRESS.
   - Otherwise continue to the next turn.

CARRY-FORWARD STATE (compact)
- case_status: {case_name: passed | failing(last_violation) | parked(attempts, last_violation) | blocked} for every case in <CASE_SET>
- unresolved_case_count and parked_case_count: integers — NO-PROGRESS only fires when parked_case_count == unresolved_case_count (nothing unparked left) and no case has moved to passed in 3 turns
- repair_log: last 5 {case, error_code, fix_applied} entries — used to ban repeating an identical fix
- turn_count: <int> / budget <MAX_ITERATIONS>
- human_gate_status: {case_name: cleared|not-cleared} — one entry per gated (paid/state-mutating) case in <CASE_SET>, supporting multiple simultaneously-gated cases rather than assuming only one

ACTION BAN
- Never mark a case passed because the response "looked right" — only <CONTRACT_VALIDATOR>'s explicit zero-violation result counts.
- Never treat a 401/403 as automatic BLOCKED when the case under test's declared expectation is 401/403 — only an auth failure surfacing on a call NOT targeting such a case triggers BLOCKED.
- Never repeat a byte-identical call after a non-transient failure — the only sanctioned identical repeat is a single TRANSIENT/5xx backoff retry.
- Never edit <CONTRACT_SPEC> mid-run to make a stubborn failing case pass — fix the call or the client code, park it after 2 distinct failed repairs, or escalate to BLOCKED if the spec itself is wrong (that's a human call, not a loop action).
- Never oscillate a fix between two prior values (A->B->A) without a new rationale in repair_log.
- Never grind indefinitely on one hard case at the expense of never attempting the rest of <CASE_SET> — park it after 2 distinct failed repairs and move on; only return once nothing unparked remains.
- Never fire a paid/quota-consuming/state-mutating case without a fresh human-approval clearance for that specific case (checked per-case in human_gate_status, not assumed cleared globally).
- Never test only the happy-path case and declare the endpoint "verified" — every case in <CASE_SET> must independently pass.

STOP — halt on the FIRST of:
SUCCESS (<CONTRACT_VALIDATOR> reports zero violations across every case in <CASE_SET>, including any auth-failure-probe cases scored normally) | BUDGET (<MAX_ITERATIONS> total turns used) | NO-PROGRESS (every remaining unresolved case is PARKED with nothing left to attempt and no case has newly passed for 3 consecutive turns, or a repeated identical {case, fix}, or an A->B->A oscillation) | BLOCKED (a call not targeting a declared auth-failure case unexpectedly hits 401/403; an UNREACHABLE_CASE with no way to trigger it; an unresolvable spec-vs-live conflict; or a gated paid/state-mutating case with no cleared approval for that specific case). Report case_status (including parked cases), repair_log, and turn_count at the point of halt.
