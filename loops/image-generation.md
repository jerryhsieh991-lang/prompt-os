# Image Generation

`image-generation` — 4 loop prompts.

### 1. Product Photo to Brand Spec (Image-Gen build-verify loop)

- **When:** Use when an agent must produce a product photo that satisfies a fixed, pre-written brand-spec checklist (background/lighting/logo placement/palette ΔE/aspect ratio/resolution/no-artifacts) and needs an independently-verified, human-signed-off final file rather than a self-graded "looks right" output.
- **Loop:** assess last rubric result -> ACTION TURN (ONE new generate-or-edit action targeting the highest-priority failing item(s), consumes 1 of 6 BUDGET) -> independent rubric verify (deterministic tool for quantitative items, vision-LLM call for qualitative items, no access to prompt/generator confidence) -> if all-pass: CONFIRMATION TURN (re-verify the identical file, zero new action, does not consume BUDGET) -> if CONFIRMATION also all-pass: human sign-off gate -> SUCCESS; else (any fail at either turn) -> back to assess/ACTION TURN
- **Stop:** SUCCESS: 2 consecutive independent rubric all-pass results on the SAME file — one ACTION TURN's verify plus the immediately following CONFIRMATION TURN's verify (no edit action in between) — AND explicit human sign-off on that exact file before any publish/spend/delete action. · BUDGET: 6 generation/edit attempts, counted ACTION-TURN-only. CONFIRMATION TURNs (re-verifying an already all-pass file with no new generation/edit call) never consume a budget slot — this must be stated explicitly in the scratchpad accounting or the loop stalls in an unaccounted re-check state. Structurally, a CONFIRMATION TURN can never be followed by another CONFIRMATION TURN (it either ends in SUCCESS+sign-off or resets to an ACTION TURN), so free turns can't accumulate into an unbounded loop. · NO-PROGRESS: 2 consecutive ACTION TURNs (not counting CONFIRMATION TURNs, which by definition start from zero failing items) show the identical set of failing items with no new item resolved. · BLOCKED: A required brand-spec asset (logo file, palette reference, approved sample shot) doesn't exist; the image-gen tool errors/rate-limits and can't be retried within budget; or a QUANTITATIVE rubric item (ΔE, resolution, aspect ratio) has no deterministic script/tool available to score it — such an item must never fall back to LLM eyeballing.
- **Model:** Solo loop, mid-tier model (Sonnet) for the ACTION TURN's generation/edit prompting and CONFIRMATION TURN bookkeeping; the independent verifier's qualitative calls (logo clear-space, garbled-text/artifact detection) also run fine on Sonnet-tier vision — no need for a top-tier/Fable-5 escalation since the hard quantitative work is delegated to deterministic tooling, not model judgment. Escalate to Opus only if a NO-PROGRESS halt fires and a human wants a one-off root-cause diagnosis of why the generator keeps failing the same item.

```text
Freeze the goal before turn 1: generate a product photo of "<PRODUCT>" satisfying brand spec "<BRAND_SPEC_DOC_OR_CHECKLIST>" (e.g.: background = <HEX/description>, lighting = <e.g. soft key, 45° left, no harsh shadow>, logo = <placement/size/clear-space rule>, palette ΔE tolerance = <N>, aspect ratio = <W:H>, min resolution = <PX>, no hallucinated/garbled text, no extra limbs/artifacts).

Write the full checklist out as discrete pass/fail items BEFORE generating anything — this is the rubric, and it does not change mid-loop. Tag every item as one of:
- QUANTITATIVE — anything checkable by a deterministic script (palette ΔE, exact pixel resolution, aspect ratio).
- QUALITATIVE — anything requiring visual judgment (logo clear-space/placement, garbled/hallucinated text, extra-limb/artifact detection, light-direction plausibility).

Maintain a scratchpad: turn-type log (ACTION / CONFIRMATION), attempts_used (ACTION TURNs only, out of 6 BUDGET), last prompt/seed used, rubric result log (turn: turn type, failing items, resolved items), consecutive-all-pass streak count, human sign-off status.

Each turn is exactly one of two types:

ACTION TURN — the default, and always turn 1. Runs whenever the previous turn's verify did NOT return all-pass (or there was no previous turn):
1. Assess — read the last rubric result; identify the highest-priority still-failing item(s).
2. Take ONE reversible action: generate a new candidate, or edit the previous candidate to target the specific failing item(s) (e.g. re-light, swap background, resize logo). Do not regenerate from scratch and re-edit in the same turn — that's two actions. Never resubmit a prompt/seed combination already tried verbatim, and never oscillate between two prior variants (A->B->A) — each turn's change must be a genuinely new adjustment, logged as a diff from the last attempt. This consumes one of the 6 BUDGET attempts.
3. Verify (see "Independent rubric verify" below) against the new file.
4. Log the per-item result. If all-pass, the next turn is a CONFIRMATION TURN on this exact file. If not, the next turn is another ACTION TURN.

CONFIRMATION TURN — runs only immediately after an ACTION TURN's verify returned all-pass. Takes NO generation/edit action and does NOT consume a BUDGET attempt; it exists solely to re-confirm the exact same output file (no new bytes, no re-generation, no edit):
1. Re-run the independent rubric verify against the identical file from the prior ACTION TURN.
2. If all-pass again: this is the 2nd of the 2 required consecutive passes — proceed to the human sign-off gate below. Do not take a further loop turn on this file unless the human requests changes.
3. If any item now fails (rubric non-determinism on a qualitative call, e.g. a borderline clear-space judgment): the consecutive-pass streak resets to 0, and the NEXT turn is an ACTION TURN treating the newly-failing item(s) as its assessment target. A CONFIRMATION TURN is never followed by another CONFIRMATION TURN — it always resolves to either sign-off or a budget-consuming ACTION TURN, which bounds the loop even though CONFIRMATION TURNs are free.

Independent rubric verify (applies identically to ACTION and CONFIRMATION turns): a separate call/reviewer with no access to the generation prompt, the generator's own confidence, or prior verify results, scores every checklist item pass/fail against the actual output file:
- QUANTITATIVE items MUST be scored by a deterministic script/tool call — a real color-space/ΔE calculation, an image-metadata read for exact pixel resolution, a width/height ratio computation for aspect ratio. Never score these by an LLM eyeballing a number against a numeric tolerance; that is not meaningfully more mechanical than aesthetic judgment and reintroduces Goodhart risk. If no such deterministic tool is available for a given item, halt BLOCKED on that item — do not substitute judgment.
- QUALITATIVE items are scored by a vision-capable reviewer call (logo clear-space/placement, garbled/hallucinated text, extra-limb/artifact detection, light-direction plausibility), still with no access to the generation prompt.
- Never accept the generator's own "this matches the brief" as verification, for either item type.

If the independent rubric returns all-pass on a CONFIRMATION TURN (the 2nd consecutive pass), do NOT auto-publish: surface the candidate and the full checklist result to a human for explicit sign-off before any irreversible or paid action (publishing to the live listing, spending generation credits beyond the stated budget, deleting prior candidates). SUCCESS requires both the rubric pass (x2 consecutive, one ACTION TURN verify + the immediately following CONFIRMATION TURN verify on the same file) and that human sign-off — not either alone.

Halt on the first of:
- SUCCESS — 2 consecutive independent rubric passes (one ACTION TURN verify + the following CONFIRMATION TURN verify, same file, no edit action between them) are all-pass AND a human has signed off on that exact file.
- BUDGET — 6 generation/edit attempts used, counted ACTION-TURN-only; CONFIRMATION TURNs never count against this cap.
- NO-PROGRESS — 2 consecutive ACTION TURNs show the identical set of failing items with no new item resolved (stuck, not iterating). CONFIRMATION TURNs are excluded from this comparison since they start from an all-pass state by construction.
- BLOCKED — a required brand-spec asset (logo file, palette reference, approved sample shot) doesn't exist; the image-gen tool errors/rate-limits and can't be retried within budget; or a QUANTITATIVE rubric item has no deterministic tool available to score it.

On any halt, report: the last candidate, the full rubric scorecard (tagged by turn type and item type — QUANTITATIVE/QUALITATIVE), and which stop condition fired — never end silently on an unlabeled last attempt.
```

### 2. Character-Consistency Image Regeneration Loop

- **When:** Use when regenerating a fixed set of character images against frozen scene briefs, one worst-image-at-a-time, until every image passes an independent identity + scene-compliance judge rather than the generator's own confidence.
- **Loop:** ASSESS -> ACT -> VERIFY -> DECIDE
- **Stop:** SUCCESS: A fresh full-set judge run (embedding score E + vision-rubric identity score V_identity + vision-rubric scene-compliance score V_scene, computed for every image with no uncertain-band shortcut) shows every image at PASS per the combination rule: E >= likeness_threshold AND V_identity >= likeness_threshold AND V_scene >= scene_compliance_threshold. · BUDGET: [N] iterations or [T] time/compute elapsed, whichever comes first — checked after BLOCKED and NO-PROGRESS. · NO-PROGRESS: The current worst image's currently-failing sub-score (whichever of E, V_identity, or V_scene is below threshold) is flat within +/-[margin] for [K] consecutive turns despite lever changes on that image — force a lever CATEGORY switch (not another tweak in the same category) on the next turn. · BLOCKED: Checked first, before NO-PROGRESS/BUDGET/SUCCESS, since it implies verification itself may be untrustworthy: the reference set can't establish ground truth for a required angle/feature, OR E and V_identity disagree by more than [disagreement_margin] on any image where both were computed, OR the judge's verdict conflicts with plain human inspection on 2+ images. Also halts for human confirmation before any paid high-res/upscale pass or before the finished set ships anywhere. Escalate to a human rather than redefining the rubric or lowering thresholds yourself.
- **Model:** This is an image-generation + judge-model loop, not a Claude-reasoning-depth problem, so run the orchestrating agent at the default Opus 4.8 xhigh; only escalate to Fable 5 if NO-PROGRESS triggers repeatedly and lever-category selection itself needs deeper one-shot reasoning about the failure taxonomy.

```text
Goal: produce a set of images of [character_name] across the frozen scene list [scenes.json — list of N scene/pose/outfit briefs], where EVERY image PASSES per the COMBINATION RULE below, evaluated against the frozen reference set [reference_images/ — 3-5 canonical shots covering face, outfit, and any distinguishing features]. An image's "score" is never a single ambiguous number — it is a pass/fail derived from three named sub-scores (E, V_identity, V_scene) combined by an explicit rule, so two graders can never silently disagree about whether an image passed.

JUDGE DEFINITION (frozen before turn 1):
- E(image) = deterministic identity-embedding similarity score (0-100) between the image and the closest-matching reference in [reference_images/]. State the exact embedding model/checkpoint. This model MUST be architecturally distinct from, and not derived from the same base weights as, any embedding the generator uses for reference-conditioning (e.g., if the generator conditions via an ArcFace- or CLIP-derived IP-Adapter/FaceID/InstantID embedding, the judge's embedding model must come from a different architecture family and be independently trained). Record the generator's conditioning-embedding family and the judge's embedding family side by side in the frozen judge spec, so "independent of the generator" is an auditable fact, not an assertion.
- V(image) = a separate vision-model judge (different model/family than the generator, stated explicitly) scoring a fixed rubric with two named sub-scores:
  - V_identity (0-100): face structure, hair, distinguishing marks, outfit/palette consistency vs. the reference set.
  - V_scene (0-100): does the image match its assigned scene brief — correct pose, correct setting, correct outfit-for-that-scene where scenes call for outfit changes.

COMBINATION RULE:
- Per-image PASS := E >= [likeness_threshold] (e.g. 85/100) AND V_identity >= [likeness_threshold] AND V_scene >= [scene_compliance_threshold] (e.g. 85/100).
- Per-turn single-image VERIFY (compute-cost optimization): always compute E. Compute V_identity + V_scene whenever E falls inside the uncertain band [likeness_threshold - margin_e, likeness_threshold + margin_e], OR whenever ASSESS's dominant-failure-category for that image is outfit/palette mismatch, art-style inconsistency, pose-induced identity loss, or scene-brief drift (categories embeddings are known to be weak on), OR whenever this VERIFY is part of a full-set run.
- Full-set judge runs (every 5 turns, at first PASS-crossing for an image, and the SUCCESS check) ALWAYS compute E, V_identity, AND V_scene for every image, with no uncertain-band shortcut. This closes the gap where an image could look like it passed on embedding similarity alone — which can share signal with the generator's own conditioning embedding — without ever clearing the independent vision-rubric check before counting toward SUCCESS.
- Disagreement tie-break: if E and V_identity disagree by more than [disagreement_margin] (e.g. 15 points) on any run where both were computed, do not average or take the max — this counts toward the BLOCKED stop condition instead.

Freeze before turn 1: the reference set, the scene list, the judge method (embedding-model family AND vision-judge model/rubric, including the declared generator-vs-judge embedding independence pairing above), all thresholds (likeness_threshold, scene_compliance_threshold, margin_e, disagreement_margin), and the budget. The generator's own output confidence or self-description never counts as verification.

Each turn:
1. ASSESS — pull the latest per-image score table (patched in place after every single-image VERIFY — see Carry forward). Identify the single worst-scoring image (its lowest failing sub-score among E, V_identity, V_scene) and its dominant failure category: face drift, outfit/palette mismatch, art-style inconsistency, pose-induced identity loss, or scene-brief drift (wrong pose/setting/outfit for that scene).
2. ACT — take exactly ONE reversible action on ONLY that worst image: regenerate it with one lever changed (reference-image conditioning strength, seed, character-LoRA/embedding weight, one negative-prompt term, a targeted inpaint pass on the drifted region, or a scene-brief re-conditioning pass if the failure category is scene-brief drift) — never touch other images this turn, never bundle multiple levers.
3. VERIFY — run the judge on the new version of that image per the COMBINATION RULE (E always; V_identity/V_scene per the conditions above). Patch that image's row in the score table in place immediately — do not wait for a full-set run. Every [5] turns, or whenever an image first PASSes on all three sub-scores, additionally re-run the full judge (E + V_identity + V_scene) across the ENTIRE set to catch regressions elsewhere and refresh every row.
4. DECIDE — keep the new version only if: (i) the image's identity signal(s) computed this turn (E, and V_identity if computed) improved or held steady, (ii) its scene-compliance signal (V_scene, if computed) did not drop below scene_compliance_threshold - margin, and (iii) no other image regressed on the most recent full-set run. Otherwise revert to the prior version of that image (git/version the image set) before the next turn.

Carry forward each turn: goal; per-image score table holding all three sub-scores (E, V_identity, V_scene) and derived PASS/FAIL per image — updated in place after every single-image VERIFY, with full-set runs refreshing all rows at once (full-set cadence never gates when a row is allowed to update); current worst-image ID + failure category; lever(s) already tried on that image this streak (blocks verbatim retries and A->B->A oscillation back to a rejected value); dead-end lever categories per image; budget used/remaining.

Stop on whichever trips first, checked in this order:
1. BLOCKED — the reference set can't establish ground truth for a required angle/feature, OR any image shows E vs. V_identity disagreeing by more than [disagreement_margin] on a run where both were computed, OR the judge's verdict conflicts with plain human inspection on 2+ images. Stop and escalate to a human rather than redefining the rubric or lowering thresholds yourself. Also halt for human confirmation before any paid high-res/upscale pass or before the finished set is published/shipped anywhere. (Checked first: it implies verification itself may be untrustworthy, so it preempts a NO-PROGRESS streak measured on the same possibly-unreliable signal.)
2. NO-PROGRESS — the current worst image's currently-failing sub-score is flat within +/-[margin] for [K] consecutive turns despite lever changes — switch lever CATEGORY (not another tweak within the same category) on the next turn.
3. SUCCESS — a fresh full-set judge run (E + V_identity + V_scene, every image, no shortcuts) shows every image at PASS.
4. BUDGET — [N] iterations or [T] time/compute elapsed.
```

### 3. Icon Set to Grid Spec — Geometry+Palette Checker Loop

- **When:** Use when a design-system icon set must be driven to zero violations against a deterministic geometry+palette checker (grid/stroke/padding/palette), with a human sign-off gate before the set is treated as shippable.
- **Loop:** run CHECKER_COMMAND against the full set's vector source files and read the violation report -> pick the one highest-violation icon for a single-icon fix, OR (only if 3+ icons share one identical off-palette color) a global palette swap for that color, never both in the same turn -> apply exactly that one action, touching no icon outside its declared scope -> re-run CHECKER_COMMAND across the FULL set (same command as step 1, full-set total, not just the touched icon(s)) -> if the full-set total dropped, commit; if it rose or held equal, revert that icon(s) to last known-good and plan a materially different fix next turn
- **Stop:** SUCCESS: Checker reports zero violations across all &lt;ICON_COUNT&gt; icons AND a human has reviewed the rendered set — including the running list of flagged ambiguous-glyph design-judgment calls from the MODEL rule below — and explicitly signed off. A zero-violation checker run alone is "checker-clean, pending human sign-off," not done. Present the full rendered set plus the flagged-simplifications list at this point and wait for approval before treating the icon set as final/shippable; this is a shared design-system artifact, not a private draft. · BUDGET: &lt;MAX_ITERATIONS&gt; reached before reaching checker-clean status — stop and report remaining full-set violation count/breakdown, iterations used, and the current per-icon focus state. · NO-PROGRESS: Full-set total violation count unchanged for 3 consecutive turns, or a single icon cycles between two known-bad states (A→B→A) for 2 turns — halt immediately rather than continuing to grind. Report the stuck icon(s), both failed approaches tried on it, current violation breakdown, and ask whether to escalate model tier (see MODEL) or change scope. · BLOCKED: Halt and ask a human, do not silently loosen tolerances or proceed, when: the spec file is missing or internally contradictory; the palette lacks a needed semantic distinction; no vector source of truth distinct from the export is available (see checker-format rule below) — checks must never run against PNG geometry; or a required glyph is UNDRAWABLE — no interpretation fits within grid+stroke+padding while remaining recognizable. Undrawable is distinct from merely ambiguous (multiple recognizable interpretations, at least one fits spec) — ambiguous glyphs are handled per MODEL, not BLOCKED; do not treat "I'm not sure which version is right" as grounds to halt if at least one drawable, in-spec interpretation exists.
- **Model:** Default tier for regeneration/hand-edits. Escalate to Fable 5 only for a glyph that is ambiguous (not undrawable) — Fable 5 may pick and simplify one interpretation, but must log the simplification verbatim in carry-forward state as a flagged design-judgment call that stays pending until the human sign-off gate at SUCCESS approves or rejects it; if no interpretation can be drawn recognizably within grid+stroke+padding at all, that is UNDRAWABLE, not ambiguous — report BLOCKED immediately instead of escalating model tier or letting Fable 5 decide unilaterally.

```text
Goal (frozen): every icon in the set &lt;ICON_SET_NAME&gt; (&lt;ICON_COUNT&gt; icons: &lt;ICON_LIST&gt;) passes an independent geometry+palette checker with zero violations — checker is `&lt;CHECKER_COMMAND&gt;` (a deterministic script separate from whatever generates the icons), not your visual read of the render — AND a human has signed off on the reviewed set before it is treated as final/shippable. Frozen spec, do not edit to match your output if it looks wrong — report BLOCKED instead:
- Canvas: exactly &lt;GRID_SIZE&gt;x&lt;GRID_SIZE&gt;px
- Grid: all path vertices snapped to a &lt;GRID_UNIT&gt;px grid, zero off-grid points
- Stroke: &lt;STROKE_WEIGHT&gt;px ± &lt;STROKE_TOLERANCE&gt;px, consistent across every icon
- Padding: &lt;PADDING&gt;px safe margin on all sides (live/bleed area respected)
- Palette: every fill/stroke color must match one of {&lt;PALETTE_HEX_LIST&gt;} within &lt;COLOR_TOLERANCE&gt; delta-E — no off-palette pixels
- Format: source of truth is &lt;VECTOR_SOURCE_FORMAT&gt; (e.g. SVG/path-data) for every icon; export as &lt;EXPORT_FORMAT&gt; (e.g. SVG/PNG) at &lt;EXPORT_SIZE&gt;

Checker/format rule: grid, stroke, and padding checks always run against each icon's &lt;VECTOR_SOURCE_FORMAT&gt; source file, never against a rendered PNG — "off-grid vertex" and "stroke width" are vector-path properties, not raster ones. If EXPORT_FORMAT includes PNG, that PNG is a derived artifact regenerated from the vector source and is checked only for canvas size and palette (pixel color sampling), never geometry. If the pipeline cannot produce or access a vector source distinct from the export, report BLOCKED before turn 1 rather than running geometry checks against PNG.

Action-scope rule: each turn takes exactly ONE action, which is either (a) a single-icon fix — regenerate or hand-edit exactly one icon chosen in step 2, touching no other icon file — or (b) a global palette swap — a single deterministic find-replace of one off-palette color value with its correct palette hex across every icon currently using it, used ONLY to clear palette violations (never grid/stroke/padding), and only when 3+ icons share that identical off-palette value. (a) is the default; (b) is the sole exception to "don't touch icons you didn't select," and the two are mutually exclusive within a turn — never do both. Whichever you choose, do not touch any icon outside that action's declared scope.

Each turn: (1) run the checker across the full set (against vector sources) and read the violation report — counts by icon and by violation type (grid/stroke/padding/palette); (2) pick the ONE icon with the most violations for a single-icon fix, or qualify for a global palette swap per the action-scope rule above; (3) apply that one action per the action-scope rule — one reversible change; (4) re-run the SAME checker command across the FULL set as verification — never eyeball the render or trust the generator's own "looks good" claim, and always compare against the full-set total from step (1), regardless of whether this turn's action was single-icon or a palette swap. If the full-set total dropped, keep the change (commit/save this version); if it rose or held equal, revert to the last known-good version of whatever this turn touched and try a materially different fix next turn (different regeneration prompt, different manual edit, or the other action type — not a re-run of the same fix).

Ambiguous vs. undrawable glyphs: if a glyph has multiple recognizable interpretations and at least one fits within grid+stroke+padding, that's ambiguous — may escalate to Fable 5 to pick and simplify one interpretation, but log the simplification verbatim in carry-forward state as a flagged design-judgment call pending human sign-off. If NO interpretation can be drawn recognizably within spec, that's undrawable — halt and report BLOCKED immediately; do not let a model unilaterally bake in an irreversible simplification for an undrawable glyph.

Never retry the exact same prompt/edit verbatim on a failed icon, and never oscillate an icon between two known-bad states (A→B→A) — if two different approaches both fail on the same icon, that's a signal to escalate model tier or flag BLOCKED, not to alternate between them.

Carry forward each turn: per-icon violation counts (grid/stroke/padding/palette breakdown), which icon or palette-swap is currently in focus and what was already tried on it, total full-set violation count trend for the last 3 turns, fixes ruled out, flagged ambiguous-glyph design-judgment simplifications awaiting human sign-off, and iterations remaining out of &lt;MAX_ITERATIONS&gt;.

Stop and report on the FIRST of: SUCCESS — checker reports zero violations across all &lt;ICON_COUNT&gt; icons AND a human has reviewed the rendered set plus the flagged-simplifications list and signed off (zero violations alone is "checker-clean, pending sign-off," not done); BUDGET — &lt;MAX_ITERATIONS&gt; reached; NO-PROGRESS — full-set total violation count unchanged for 3 straight turns, or an icon is stuck in an A→B→A cycle for 2 turns (halt, don't keep grinding — report the stuck icon and both failed approaches); BLOCKED — the spec file is missing/contradictory, no vector source of truth is available, the palette lacks a needed semantic distinction, or a required glyph is undrawable (not merely ambiguous) within grid+stroke constraints (surface it and wait for a human decision on scope).
```

### 4. Photoreal Shot Brief vs. Independent Rubric Judge

- **When:** Use when iterating a photoreal image-generation prompt/parameters against a fixed shot brief (subject, composition, lighting) and no single "correct" pixel output exists, so a rubric judge must be defined before iterating.
- **Loop:** assess (best image + per-dimension scores + worst-scoring dimension + changes already tried) -> ONE change to the generation prompt/parameters targeting only the worst dimension -> generate the image and have an independent judge rubric-score it against the frozen brief -> commit as new best or discard -> check 4 stop arms
- **Stop:** SUCCESS: Mean rubric score >= <TARGET_SCORE> (e.g. 4.2/5) across composition, lighting, and subject-fidelity, AND every individual dimension >= <MIN_DIMENSION_SCORE> (e.g. 3/5), on the same image. · BUDGET: <MAX_ITER> generations (e.g. 8) or <MAX_SPEND> in API/render credits, whichever trips first. · NO-PROGRESS: Mean score flat within ±0.1, or the loop oscillates between two prompt variants (A->B->A), for 3 consecutive turns -- force one structurally different lever (not a reword) before halting if still flat. · BLOCKED: Judge unavailable; the shot brief is ambiguous or self-contradictory (e.g. conflicting lighting cues) and needs human clarification; or the shot requires a licensed/trademarked subject or real person's likeness needing human/legal sign-off before generating.
- **Model:** The judge must be a separate model/instance from the one drafting the generation prompt, and must see ONLY the rendered image plus the frozen brief -- never the generation prompt, seed, or parameters that produced it. Use a stronger model (Opus/Fable-5-class) in the judge role for nuanced composition/lighting/subject calls; the drafting role that proposes prompt edits can be a cheaper model since it only needs to propose changes, not evaluate them. Image generation itself is a fixed tool call to whichever photoreal generator is configured (e.g. Higgsfield, Midjourney, a diffusion API) -- not a model-routing decision.

```text
Optimize a photoreal image-generation prompt against a frozen shot brief: <SHOT_BRIEF> (subject, composition/framing, lighting direction & quality, lens/mood, aspect ratio). No single correct pixel output exists, so define the judge FIRST: an independent rubric judge -- a separate model/instance from the one drafting the generation prompt, seeing ONLY the rendered image and the frozen brief, never the generation prompt/seed/parameters -- scores 1-5 on: (1) Composition -- framing, subject placement, lines/balance as specified in the brief; (2) Lighting -- direction, quality, color temperature, time-of-day match; (3) Subject fidelity -- correct subject, pose, count, attributes, no deformities/extra limbs/artifacts; (4) Technical/photoreal fidelity -- resolution, absence of AI-tells (warped hands, melted objects, texture smear), consistent physics. The judge reports per-dimension scores plus one concrete failure note for any dimension below threshold.

GOAL (frozen): mean rubric score >= <TARGET_SCORE> (e.g. 4.2/5) AND every dimension >= <MIN_DIMENSION_SCORE> (e.g. 3/5), on the same image. Budget: <MAX_ITER> generations (e.g. 8) or <MAX_SPEND> credits, whichever first.

Per turn: (1) Assess -- read carry-forward state: current best image id, its per-dimension scores, the single worst-scoring dimension, and the list of prompt changes already tried (never repeat one verbatim). (2) Act -- make ONE change targeting only the worst-scoring dimension (e.g. add an explicit lighting-direction cue, add a lens/focal-length descriptor, add a negative-prompt term for a recurring artifact, adjust framing) -- never touch more than one lever per turn, never resubmit an identical prompt+seed pair already tried. (3) Verify -- generate the image; hand it with the frozen brief to the independent judge for a fresh rubric score; the generating role never grades its own output. (4) Decide -- if the mean score improves AND no dimension regresses below its prior value, commit as new best (store image id, prompt text, full score breakdown) and update the scratchpad; otherwise discard the candidate and keep the previous best.

Carry-forward state (compact, update every turn): {best_image_id, best_prompt_text, best_scores: {composition, lighting, subject, technical}, worst_dimension, changes_tried: [...], iterations_used/budget, consecutive_flat_turns}.

Stop the instant ONE arm trips: SUCCESS (thresholds met on one image) -- BUDGET (<MAX_ITER> generations or <MAX_SPEND> reached) -- NO-PROGRESS (mean score flat +/-0.1, or oscillating between two prompt variants, for 3 consecutive turns -- force one structurally different lever, e.g. swap composition strategy rather than reword, then halt if still flat) -- BLOCKED (judge unavailable; brief is ambiguous/self-contradictory and needs human clarification; or the subject requires licensed/trademarked/real-person likeness approval before generating).

Guardrails: no scope creep -- optimize ONLY this frozen brief's composition/lighting/subject, not new shot ideas or alternate briefs (backlog those). Never treat "looks good to me" as verification -- only the independent judge's score counts. If the accepted final image is destined for commercial/client/licensed use, get explicit human approval before that image is marked deliverable-ready -- the search iterations are cheap and reversible, but shipping the final asset is not, and that step gets a human gate regardless of remaining budget.

Report at stop: final/best image id, full score breakdown, stop reason, and the prompt-change history with scores per iteration.
```
